# 第八章：交易系統與撮合引擎基礎
> 從零開始理解撮合引擎的領域知識，並用 Java 實作核心邏輯

---

## 一、撮合引擎是什麼？

撮合引擎（Matching Engine）是交易所的核心系統，負責將買方與賣方的訂單配對成交。

```
使用者 A 想以 100 元買 1 股 → 買單（Bid）
使用者 B 想以 100 元賣 1 股 → 賣單（Ask）

撮合引擎：「價格一樣，成交！」
```

現實世界的撮合引擎每秒處理數百萬筆訂單，延遲要求在**微秒（μs）甚至奈秒（ns）**等級。

---

## 二、訂單的基本類型

### 限價單（Limit Order）
「我要以**不超過（買）/ 不低於（賣）**某個價格成交」

```java
// 買單：最高願意付 100 元，買 10 股
Order buyOrder = new Order("BUY", 100.0, 10);

// 賣單：最低願意賣 102 元，賣 5 股
Order sellOrder = new Order("SELL", 102.0, 5);
// 這兩筆無法成交（買 100 < 賣 102）
```

### 市價單（Market Order）
「我要**立刻成交**，不管價格」→ 以當前最優價格吃單

### 止損單（Stop Order）
「當價格跌到某個點，自動變成市價單賣出」→ 保護型訂單

### FOK / IOC（特殊執行條件）
- **FOK（Fill or Kill）**：全部成交或全部取消
- **IOC（Immediate or Cancel）**：立刻能成交多少就成交多少，剩下取消

## 三、訂單簿（Order Book）

訂單簿是交易所公開的買賣雙方掛單記錄，格式如下：

```
        賣方（Ask）
價格    數量
─────────────
105     200
104      50
103     100  ← 最佳賣價（Best Ask）
─────────────  ← Spread（買賣價差）
102     150  ← 最佳買價（Best Bid）
101     300
100     500
─────────────
        買方（Bid）
```

**關鍵術語：**
| 術語 | 說明 |
|------|------|
| Best Bid | 目前最高的買單價格 |
| Best Ask | 目前最低的賣單價格 |
| Spread | Best Ask - Best Bid，越小流動性越好 |
| Depth | 訂單簿的厚度，反映市場流動性 |
| Mid Price | (Best Bid + Best Ask) / 2 |

---

## 四、撮合規則：價格優先 + 時間優先（FIFO）

大多數交易所使用「**價格-時間優先**」規則：

1. **價格優先**：出價更好的訂單先成交
   - 買單：出價**越高**優先級越高
   - 賣單：要求**越低**優先級越高

2. **時間優先**：相同價格下，**先掛的訂單先成交**

```
買方訂單簿（依優先級排列）：
優先級  時間    價格   數量
  1   09:00   102    100   ← 同價格中最早的
  2   09:05   102     50
  3   09:10   101    200
  4   09:15   100    300

賣方新進來一筆：賣 102 元，80 股
→ 與優先級 1 的買單成交 80 股（09:00 那筆剩 20 股）
```

## 五、用 Java 實作簡單訂單簿

### 訂單物件

```java
import java.time.Instant;
import java.util.*;

public class Order {
    public enum Side { BUY, SELL }
    
    private final String orderId;
    private final Side side;
    private final double price;
    private int quantity;           // 剩餘數量（成交後會減少）
    private final Instant timestamp;
    
    public Order(String orderId, Side side, double price, int quantity) {
        this.orderId   = orderId;
        this.side      = side;
        this.price     = price;
        this.quantity  = quantity;
        this.timestamp = Instant.now();
    }
    
    // Getters
    public String  getOrderId()  { return orderId; }
    public Side    getSide()     { return side; }
    public double  getPrice()    { return price; }
    public int     getQuantity() { return quantity; }
    public Instant getTimestamp(){ return timestamp; }
    
    public void reduceQuantity(int amount) { quantity -= amount; }
    public boolean isFilled() { return quantity <= 0; }
    
    @Override
    public String toString() {
        return String.format("[%s] %s %.2f x %d @ %s",
            orderId, side, price, quantity, timestamp);
    }
}
```

### 價格層（Price Level）

同一個價格的所有訂單，用 Queue 維持時間順序：

```java
public class PriceLevel {
    private final double price;
    // LinkedList 作為 Queue：先進先出（時間優先）
    private final Queue<Order> orders = new LinkedList<>();
    private int totalQuantity = 0;
    
    public PriceLevel(double price) {
        this.price = price;
    }
    
    public void addOrder(Order order) {
        orders.offer(order);
        totalQuantity += order.getQuantity();
    }
    
    public void removeOrder(Order order) {
        orders.remove(order);
        totalQuantity -= order.getQuantity();
    }
    
    public Order peek() { return orders.peek(); }  // 查看最優先的訂單
    public boolean isEmpty() { return orders.isEmpty(); }
    public int getTotalQuantity() { return totalQuantity; }
    public double getPrice() { return price; }
}
```

### 訂單簿核心

```java
public class OrderBook {
    private final String symbol;  // 交易標的，例如 "BTC/USDT"
    
    // 買單簿：價格「由高到低」排序（最高買價優先）
    // TreeMap<價格, 該價格的所有訂單>
    private final TreeMap<Double, PriceLevel> bids =
        new TreeMap<>(Comparator.reverseOrder());
    
    // 賣單簿：價格「由低到高」排序（最低賣價優先）
    private final TreeMap<Double, PriceLevel> asks =
        new TreeMap<>();
    
    // 成交紀錄
    private final List<Trade> trades = new ArrayList<>();
    
    public OrderBook(String symbol) {
        this.symbol = symbol;
    }
    
    // 新增限價單，並嘗試撮合
    public void addOrder(Order newOrder) {
        if (newOrder.getSide() == Order.Side.BUY) {
            matchBuyOrder(newOrder);
            if (!newOrder.isFilled()) {
                // 沒有完全成交，放入買單簿等待
                addToBids(newOrder);
            }
        } else {
            matchSellOrder(newOrder);
            if (!newOrder.isFilled()) {
                addToAsks(newOrder);
            }
        }
    }
    
    // 嘗試撮合買單
    private void matchBuyOrder(Order buyOrder) {
        // 持續與最低賣價撮合，直到買單成交完畢或沒有合適賣單
        while (!buyOrder.isFilled() && !asks.isEmpty()) {
            double bestAskPrice = asks.firstKey();
            
            // 買價 < 最低賣價 → 無法成交，停止
            if (buyOrder.getPrice() < bestAskPrice) break;
            
            PriceLevel askLevel = asks.get(bestAskPrice);
            execute(buyOrder, askLevel, bestAskPrice);
            
            // 若該價格層的賣單全部成交，移除這個價格層
            if (askLevel.isEmpty()) asks.remove(bestAskPrice);
        }
    }
    
    // 嘗試撮合賣單
    private void matchSellOrder(Order sellOrder) {
        while (!sellOrder.isFilled() && !bids.isEmpty()) {
            double bestBidPrice = bids.firstKey();
            
            // 賣價 > 最高買價 → 無法成交，停止
            if (sellOrder.getPrice() > bestBidPrice) break;
            
            PriceLevel bidLevel = bids.get(bestBidPrice);
            execute(sellOrder, bidLevel, bestBidPrice);
            
            if (bidLevel.isEmpty()) bids.remove(bestBidPrice);
        }
    }
    
    // 實際執行成交
    private void execute(Order aggressor, PriceLevel level, double tradePrice) {
        while (!aggressor.isFilled() && !level.isEmpty()) {
            Order resting = level.peek();  // 取出優先級最高的訂單
            
            // 雙方可成交的數量取較小值
            int tradeQty = Math.min(aggressor.getQuantity(), resting.getQuantity());
            
            aggressor.reduceQuantity(tradeQty);
            resting.reduceQuantity(tradeQty);
            level.getTotalQuantity();  // 更新
            
            Trade trade = new Trade(
                aggressor.getOrderId(),
                resting.getOrderId(),
                tradePrice,
                tradeQty
            );
            trades.add(trade);
            
            System.out.printf("✅ 成交！%s x %s | 價格:%.2f 數量:%d%n",
                aggressor.getOrderId(), resting.getOrderId(), tradePrice, tradeQty);
            
            // 如果掛單方已全部成交，從簿中移除
            if (resting.isFilled()) {
                level.removeOrder(resting);
            }
        }
    }
    
    private void addToBids(Order order) {
        bids.computeIfAbsent(order.getPrice(), PriceLevel::new).addOrder(order);
    }
    
    private void addToAsks(Order order) {
        asks.computeIfAbsent(order.getPrice(), PriceLevel::new).addOrder(order);
    }
    
    // 印出目前訂單簿狀態
    public void printBook() {
        System.out.println("\n=== " + symbol + " 訂單簿 ===");
        System.out.println("--- 賣方 (Ask) ---");
        // 倒序印賣單（讓最低賣價靠近中間）
        new TreeMap<>(Comparator.reverseOrder()).putAll(asks)
            .forEach((price, level) ->
                System.out.printf("  %.2f  %d%n", price, level.getTotalQuantity()));
        System.out.println("── Spread ──");
        bids.forEach((price, level) ->
            System.out.printf("  %.2f  %d%n", price, level.getTotalQuantity()));
        System.out.println("--- 買方 (Bid) ---\n");
    }
}
```

### 成交紀錄

```java
public class Trade {
    private final String aggressorId;  // 主動方（新進來的訂單）
    private final String restingId;    // 被動方（原本在簿中的訂單）
    private final double price;
    private final int quantity;
    private final Instant timestamp;
    
    public Trade(String aggressorId, String restingId, double price, int quantity) {
        this.aggressorId = aggressorId;
        this.restingId   = restingId;
        this.price       = price;
        this.quantity    = quantity;
        this.timestamp   = Instant.now();
    }
    
    @Override
    public String toString() {
        return String.format("Trade{%s<->%s price=%.2f qty=%d}",
            aggressorId, restingId, price, quantity);
    }
}
```

### 測試執行

```java
public class Main {
    public static void main(String[] args) {
        OrderBook book = new OrderBook("BTC/USDT");
        
        // 掛入一些賣單
        book.addOrder(new Order("S1", Order.Side.SELL, 105.0, 100));
        book.addOrder(new Order("S2", Order.Side.SELL, 103.0,  50));
        book.addOrder(new Order("S3", Order.Side.SELL, 103.0,  80));  // 同價格，比 S2 晚
        book.addOrder(new Order("S4", Order.Side.SELL, 102.0, 200));
        
        // 掛入一些買單
        book.addOrder(new Order("B1", Order.Side.BUY, 100.0, 300));
        book.addOrder(new Order("B2", Order.Side.BUY,  99.0, 150));
        
        book.printBook();
        
        // 進來一筆積極買單（願意付 104 元，買 200 股）
        // 應先吃掉 S4（102 元，200股），再吃 S2（103 元 50股），再吃 S3（103 元 80股）
        System.out.println(">>> 新買單進入：B3, 104.0, 250");
        book.addOrder(new Order("B3", Order.Side.BUY, 104.0, 250));
        
        book.printBook();
    }
}
```

## 六、為什麼訂單簿用 TreeMap？

回顧第二章的集合章節，現在可以理解為什麼選 `TreeMap`：

```
需求                     → 選擇
─────────────────────────────────────────
需要「快速找最高買價」   → TreeMap（O(log n) 取最大/最小鍵）
需要「價格排序」         → TreeMap（自動按鍵排序）
需要「快速插入/刪除」    → TreeMap（O(log n)，比陣列排序的 O(n) 快）
同一個價格多個訂單       → PriceLevel 內部用 Queue（FIFO 時間優先）
```

**為什麼不用 HashMap？**  
HashMap 查找是 O(1) 但無序，你無法直接問「最高買價是多少？」，每次都要掃描全部。

## 七、真實撮合引擎的架構

```
                     ┌─────────────────────────────────┐
外部訂單 (FIX/REST)  │         撮合引擎核心             │
──────────────────▶  │  ┌────────────┐                 │
                     │  │ 訂單驗證器  │                 │
                     │  └─────┬──────┘                 │
                     │        ▼                         │
                     │  ┌────────────┐  ┌────────────┐ │
                     │  │  訂單簿    │  │  風控模組  │ │
                     │  │ (Order     │  │ (風險限制) │ │
                     │  │  Book)     │  └────────────┘ │
                     │  └─────┬──────┘                 │
                     │        ▼                         │
                     │  ┌────────────┐                 │
                     │  │ 撮合引擎   │                 │
                     │  │ (Matching  │                 │
                     │  │  Engine)   │                 │
                     │  └─────┬──────┘                 │
                     └────────┼────────────────────────┘
                              ▼
                     ┌─────────────────┐
                     │  成交通知 /     │
                     │  結算系統       │
                     └─────────────────┘
```

**真實系統的額外考量：**
- **序列化**：所有訂單需要有全域唯一的序號，確保順序一致性
- **持久化**：訂單簿狀態要能從崩潰中恢復
- **多商品**：一個引擎可能同時管理數千個商品的訂單簿
- **Co-location**：交易所讓機構客戶把伺服器放在同一機房，減少網路延遲

## 本章練習題

**Q1（理解）：** 訂單簿中有以下買單：
```
價格 100，數量 50，時間 09:00
價格 102，數量 30，時間 09:05
價格 100，數量 80，時間 09:10
```
現在來一筆賣單：賣 100 元，數量 100。請問成交順序和結果為何？

<details>
<summary>答案</summary>
先成交「價格 102，數量 30」（價格優先，102 > 100）。
再成交「價格 100，數量 50（09:00 那筆）」（時間優先，09:00 比 09:10 早）。
再成交「價格 100，數量 20（09:10 那筆，只剩 20）」。
賣單共成交 100 股（30+50+20），09:10 的 100元買單剩 60 股未成交。
</details>

**Q2（實作）：** 修改上面的 OrderBook，加入 `cancelOrder(String orderId)` 方法，支援取消掛單。

**Q3（思考）：** 如果同時有 100 個使用者送單進來，Java 的 TreeMap 是否安全？應該如何解決？
<details>
<summary>提示</summary>
TreeMap 不是 thread-safe。並發環境下需要用 ConcurrentSkipListMap 或加鎖（synchronized / ReentrantLock）。真正的高頻撮合引擎通常用單執行緒 + 無鎖佇列（LMAX Disruptor 模式）來避免鎖競爭。
</details>
