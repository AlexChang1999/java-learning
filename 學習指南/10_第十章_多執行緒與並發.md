# 第十章：多執行緒與並發（Concurrency）
> 為什麼撮合引擎需要理解這章：多個使用者同時送單，你需要安全地處理並發

---

## 一、什麼是執行緒（Thread）？

一般程式是「單執行緒」：一次只做一件事，從上到下。  
多執行緒讓你能「同時」做多件事——但這個「同時」帶來了很多陷阱。

```
單執行緒：
主程式 → A → B → C → D → 結束

多執行緒：
主程式 ─┬─ 執行緒 1：A → B
        ├─ 執行緒 2：C → D
        └─ 執行緒 3：E → F
```

### 建立執行緒的三種方式

```java
// 方式一：繼承 Thread
class MyThread extends Thread {
    @Override
    public void run() {
        System.out.println("執行緒 " + Thread.currentThread().getName() + " 執行中");
    }
}
new MyThread().start();

// 方式二：實作 Runnable（推薦，因為 Java 單繼承限制）
Runnable task = () -> System.out.println("Lambda 執行緒");
new Thread(task).start();

// 方式三：ExecutorService（生產環境推薦，重用執行緒）
ExecutorService pool = Executors.newFixedThreadPool(4);  // 4 個執行緒的池
pool.submit(() -> System.out.println("執行緒池中的任務"));
pool.shutdown();  // 等待任務完成後關閉
```

## 二、競態條件（Race Condition）：最常見的並發 bug

```java
// 問題範例：兩個執行緒同時對同一個計數器加一
public class Counter {
    private int count = 0;

    public void increment() {
        count++;  // 看似一行，其實是三步！
    }
    // 這段程式碼不是「原子的」
}
```

**`count++` 拆開來看有三個步驟：**
```
1. 從記憶體讀取 count 的值（假設是 0）
2. 在 CPU 暫存器中計算 0 + 1 = 1
3. 將 1 寫回記憶體

問題：兩個執行緒同時執行這三步！
─────────────────────────────────────
執行緒 A          執行緒 B
讀取 count = 0
                  讀取 count = 0   ← 還沒等 A 寫回去就讀了！
計算 0 + 1 = 1
                  計算 0 + 1 = 1
寫回 count = 1
                  寫回 count = 1   ← 覆蓋了 A 的結果

最終 count = 1，但應該是 2！
```

這就是**競態條件（Race Condition）**：結果取決於執行緒執行的順序（順序是不可預測的）。

## 三、synchronized：加鎖保護共享資源

```java
public class SafeCounter {
    private int count = 0;

    // synchronized 方法：同一時間只有一個執行緒能進入
    public synchronized void increment() {
        count++;  // 現在是安全的
    }

    public synchronized int getCount() {
        return count;
    }
}
```

**synchronized 的鎖是「物件鎖」：**
```java
// 鎖在整個方法上（鎖住 this 物件）
public synchronized void m1() { ... }

// 等價寫法（鎖在程式碼區塊上）
public void m1() {
    synchronized (this) {
        ...
    }
}

// 鎖在靜態方法上（鎖住 Class 物件，比實例鎖更廣）
public static synchronized void m2() { ... }

// 鎖住指定物件（更細粒度的控制）
private final Object lock = new Object();
public void m3() {
    synchronized (lock) {
        ...
    }
}
```

**synchronized 的代價：**
- 未持有鎖的執行緒會進入 **BLOCKED 狀態**（被 OS 暫停）
- 喚醒被暫停的執行緒需要 **context switch**（約 1~10 μs）
- 鎖競爭激烈時，大部分時間都在切換執行緒，而不是做實際工作
- **這就是為什麼高頻交易不用 synchronized，改用無鎖算法**

## 四、volatile：保證可見性

```java
// 問題：CPU Cache 的可見性問題
public class StopFlag {
    private boolean running = true;  // 沒有 volatile

    public void stop() {
        running = false;
    }

    public void run() {
        while (running) {  // 執行緒 B 可能永遠看到舊的 running = true！
            doWork();
        }
    }
}
```

**為什麼會看到舊值？**
```
現代 CPU 有多個 Core，每個 Core 有自己的 L1/L2 Cache：

Core 1 (執行緒 A)     Core 2 (執行緒 B)
L1 Cache:              L1 Cache:
  running = false  ←   running = true  ← 還在 Core 2 的 cache！
        |                    ↑
        └──── 寫入 RAM ──────┘
                  （但 Core 2 不知道 RAM 更新了）
```

```java
// 解法：volatile 確保每次讀寫都直接存取 RAM（跳過 CPU cache）
public class SafeStopFlag {
    private volatile boolean running = true;

    public void stop() {
        running = false;  // 立刻寫入 RAM，其他 Core 可見
    }

    public void run() {
        while (running) {  // 每次都從 RAM 讀取最新值
            doWork();
        }
    }
}
```

**volatile vs synchronized：**
| | volatile | synchronized |
|---|----------|-------------|
| 解決可見性 | ✅ | ✅ |
| 解決原子性 | ❌（count++ 還是不安全）| ✅ |
| 效能開銷 | 低 | 高（鎖競爭時）|
| 適用場景 | 單一變數的讀/寫 | 多步驟操作 |

## 五、Atomic 類別：無鎖原子操作

```java
import java.util.concurrent.atomic.*;

// AtomicInteger：用 CAS 指令實現原子操作，不需要鎖
AtomicInteger count = new AtomicInteger(0);

// 原子性的加一（底層是 CPU 的 LOCK XADD 指令）
count.incrementAndGet();   // 先加一，回傳新值
count.getAndIncrement();   // 先回傳舊值，再加一

// CAS：Compare-And-Swap（樂觀鎖）
// 「如果當前值是 expected，就更新成 newValue，否則失敗重試」
boolean success = count.compareAndSet(0, 1);  // 如果是 0，就改成 1

// 常用 Atomic 類別
AtomicLong     atomicLong    = new AtomicLong(0L);
AtomicBoolean  atomicBool    = new AtomicBoolean(false);
AtomicReference<Order> atomicRef = new AtomicReference<>(null);
```

**CAS 的工作流程：**
```
執行緒 A 和 B 同時想把 count 從 0 改成 1：

執行緒 A：CAS(expected=0, new=1)
  CPU：記憶體還是 0，更新成功 ✅，count = 1

執行緒 B：CAS(expected=0, new=1)
  CPU：記憶體已經是 1，不等於 expected=0，失敗 ❌
  → 重新讀取 count=1，重試 CAS(expected=1, new=2)
  → 成功 ✅，count = 2
```

## 六、ExecutorService 與執行緒池

```java
import java.util.concurrent.*;

// 建立不同類型的執行緒池
ExecutorService fixedPool   = Executors.newFixedThreadPool(4);     // 固定 4 個執行緒
ExecutorService cachedPool  = Executors.newCachedThreadPool();      // 動態建立，閒置 60 秒後回收
ExecutorService singlePool  = Executors.newSingleThreadExecutor();  // 只有 1 個執行緒（保證順序）
ScheduledExecutorService scheduler = Executors.newScheduledThreadPool(2);  // 支援延遲/定期執行

// 提交任務
// Runnable：無回傳值
fixedPool.submit(() -> System.out.println("無回傳的任務"));

// Callable：有回傳值
Future<Integer> future = fixedPool.submit(() -> {
    Thread.sleep(1000);  // 模擬耗時計算
    return 42;
});

// 取得結果（會阻塞直到計算完成）
try {
    Integer result = future.get();         // 無限等待
    Integer result2 = future.get(2, TimeUnit.SECONDS);  // 最多等 2 秒
} catch (TimeoutException e) {
    future.cancel(true);  // 超時就取消
}

// 關閉執行緒池
fixedPool.shutdown();          // 等待所有任務完成後關閉
fixedPool.shutdownNow();       // 立刻關閉，未完成的任務返回 List
```

## 七、並發集合（Concurrent Collections）

```java
// 普通集合不是執行緒安全的，並發存取會出錯
// Java 提供了線程安全版本：

// ConcurrentHashMap：分段鎖，比 Hashtable 效能好很多
Map<String, Order> orderMap = new ConcurrentHashMap<>();
orderMap.put("O1", order1);                       // 執行緒安全
Order o = orderMap.computeIfAbsent("O2", k -> new Order(...));

// CopyOnWriteArrayList：讀不加鎖，寫時複製一份
// 適合「讀多寫少」的場景（例如訂閱者列表）
List<String> listeners = new CopyOnWriteArrayList<>();

// ConcurrentLinkedQueue：無鎖佇列，適合生產者/消費者
Queue<Order> orderQueue = new ConcurrentLinkedQueue<>();
orderQueue.offer(newOrder);   // 生產者加入
Order next = orderQueue.poll(); // 消費者取出（不阻塞）

// ArrayBlockingQueue：有界阻塞佇列
// 滿了就阻塞生產者，空了就阻塞消費者
BlockingQueue<Order> blockingQ = new ArrayBlockingQueue<>(1000);
blockingQ.put(newOrder);    // 滿了就等（阻塞）
Order o2 = blockingQ.take(); // 空了就等（阻塞）
```

## 八、happens-before：Java 記憶體模型的保證

這是一個重要但常被忽視的概念：**什麼樣的操作能保證「A 執行完才執行 B」？**

Java 記憶體模型（JMM）定義了 happens-before 規則：

```
以下情況，A 保證 happens-before B：

1. 程式順序規則：同一執行緒內，A 寫在 B 前面
2. volatile 規則：volatile 寫入 → 之後任何讀取
3. Monitor 鎖規則：unlock → 下一個 lock 之後的操作
4. start() 規則：t.start() → 執行緒 t 的任何操作
5. join() 規則：執行緒 t 的所有操作 → t.join() 返回後
```

```java
// 實際範例
int x = 0;
volatile boolean ready = false;

// 執行緒 A：
x = 42;          // (1)
ready = true;    // (2) volatile 寫入

// 執行緒 B：
if (ready) {     // (3) volatile 讀取
    // 因為 volatile 規則：(2) happens-before (3)
    // 又因為程式順序規則：(1) happens-before (2)
    // 所以 (1) happens-before (3)
    System.out.println(x);  // 保證看到 42，不會看到 0
}
```

## 九、死鎖（Deadlock）：執行緒互相等待的噩夢

### 什麼是死鎖？

死鎖是指兩個（或多個）執行緒永遠互相等待對方釋放鎖，導致全部卡住、誰都無法繼續執行。

```
典型的死鎖場景（搶鎖順序不同）：

執行緒 A 持有鎖 1，等待鎖 2
執行緒 B 持有鎖 2，等待鎖 1

  執行緒 A ─── 持有 ──▶ 鎖 1
      │                   
   等待鎖 2               
      │                   
      ▼                   
    鎖 2 ◀── 持有 ─── 執行緒 B
                 │
              等待鎖 1
                 │
                 ▼
              （等待 A 釋放鎖 1，但 A 也在等 B）
```

### 死鎖的四個必要條件（缺一不可）

```
1. 互斥（Mutual Exclusion）
   鎖只能同時被一個執行緒持有。
   → synchronized 天生互斥，無法避免。

2. 持有並等待（Hold and Wait）
   執行緒已持有一把鎖，同時等待另一把。
   → 一次只拿一把鎖就能破除。

3. 不可搶占（No Preemption）
   鎖不能強制從執行緒手中搶走，只能主動釋放。
   → tryLock 可以讓你放棄等待，破除此條件。

4. 循環等待（Circular Wait）
   A 等 B，B 等 C，C 等 A，形成環。
   → 規定所有執行緒都按相同順序取鎖，可破除。
```

### 程式碼範例：製造死鎖

```java
public class DeadlockDemo {
    private static final Object LOCK_A = new Object();
    private static final Object LOCK_B = new Object();

    public static void main(String[] args) {
        // 執行緒 1：先取 A，再取 B
        Thread t1 = new Thread(() -> {
            synchronized (LOCK_A) {
                System.out.println("T1 取得鎖 A");
                try { Thread.sleep(50); } catch (InterruptedException e) {}
                synchronized (LOCK_B) {         // 等待 LOCK_B
                    System.out.println("T1 取得鎖 B");
                }
            }
        });

        // 執行緒 2：先取 B，再取 A（順序相反，製造循環等待）
        Thread t2 = new Thread(() -> {
            synchronized (LOCK_B) {
                System.out.println("T2 取得鎖 B");
                try { Thread.sleep(50); } catch (InterruptedException e) {}
                synchronized (LOCK_A) {         // 等待 LOCK_A
                    System.out.println("T2 取得鎖 A");
                }
            }
        });

        t1.start();
        t2.start();
        // 兩個執行緒都卡住，程式永遠不會結束
    }
}
```

### 如何偵測死鎖：jstack

```bash
# 1. 找出 Java 程序的 PID
jps -l

# 2. 輸出執行緒快照
jstack <PID>

# 輸出中會看到：
# Found 1 deadlock.
# "Thread-0":
#   waiting to lock monitor 0x... (object 0x..., a java.lang.Object)
#   which is held by "Thread-1"
# "Thread-1":
#   waiting to lock monitor 0x... (object 0x..., a java.lang.Object)
#   which is held by "Thread-0"
```

### 預防死鎖的實務做法

```java
// 方法一：固定取鎖順序（破除循環等待）
// 所有地方都先取 LOCK_A 再取 LOCK_B，永遠不反過來
synchronized (LOCK_A) {
    synchronized (LOCK_B) {
        // 安全
    }
}

// 方法二：用 tryLock 設定超時（破除不可搶占）
ReentrantLock lockA = new ReentrantLock();
ReentrantLock lockB = new ReentrantLock();

boolean gotA = lockA.tryLock(100, TimeUnit.MILLISECONDS);
boolean gotB = lockB.tryLock(100, TimeUnit.MILLISECONDS);

if (gotA && gotB) {
    try {
        // 執行工作
    } finally {
        lockA.unlock();
        lockB.unlock();
    }
} else {
    // 沒搶到鎖，放棄並稍後重試（不會卡死）
    if (gotA) lockA.unlock();
    if (gotB) lockB.unlock();
}

// 方法三：撮合引擎的做法：根本不用多把鎖
// 所有訂單進入單一 RingBuffer，由「唯一的撮合執行緒」處理
// 沒有競爭 → 沒有死鎖 → 延遲最低
```

## 十、ReentrantLock：比 synchronized 更靈活的鎖

### 為什麼需要 ReentrantLock？

`synchronized` 的限制：
- **無法設定超時**：等鎖就是永遠等，容易死鎖
- **無法中斷等待**：執行緒被 Block 住無法取消
- **無法查詢鎖的狀態**：不知道有多少執行緒在等

`ReentrantLock` 都解決了這些問題：

```java
import java.util.concurrent.locks.ReentrantLock;

public class OrderBook {
    // ReentrantLock：可重入鎖
    // 「可重入」= 同一執行緒可以多次 lock()，不會自己死鎖
    private final ReentrantLock lock = new ReentrantLock();

    public void placeOrder(Order order) {
        lock.lock();  // 取得鎖（如果被佔用就等待）
        try {
            // 在鎖保護下安全操作
            addToBook(order);
            matchOrders();
        } finally {
            lock.unlock();  // 務必放在 finally，確保一定會釋放
        }
    }

    // 可重入示範：同一執行緒呼叫 placeOrder → matchOrders
    // matchOrders 內再次 lock() 不會死鎖，因為是「同一執行緒」
    private void matchOrders() {
        lock.lock();   // 同一執行緒再次取鎖，鎖計數 +1（不死鎖）
        try {
            // ...
        } finally {
            lock.unlock(); // 鎖計數 -1，還沒到 0 不會真的釋放
        }
    }
}
```

### tryLock：嘗試取鎖，失敗就放棄

```java
// 不帶超時：立刻嘗試，失敗直接回傳 false
if (lock.tryLock()) {
    try {
        // 成功取得鎖
    } finally {
        lock.unlock();
    }
} else {
    // 鎖被佔用，改做其他事（不阻塞）
    System.out.println("鎖忙碌，稍後再試");
}

// 帶超時：最多等 200ms，超時就放棄
try {
    if (lock.tryLock(200, TimeUnit.MILLISECONDS)) {
        try {
            // 成功
        } finally {
            lock.unlock();
        }
    } else {
        System.out.println("200ms 內取不到鎖，放棄");
    }
} catch (InterruptedException e) {
    Thread.currentThread().interrupt();
}
```

### 公平鎖 vs 非公平鎖

```java
// 非公平鎖（預設）：新來的執行緒可能插隊
// 效能較好（避免頻繁 context switch），但可能有執行緒餓死
ReentrantLock unfairLock = new ReentrantLock();

// 公平鎖：嚴格按照等待順序，先來先服務
// 效能較差，但保證公平
ReentrantLock fairLock = new ReentrantLock(true);

// 撮合引擎通常用非公平鎖（追求最低延遲）
// 因為我們不在乎哪個執行緒先拿到鎖，只在乎整體吞吐量
```

### synchronized vs ReentrantLock 對比

| 特性 | synchronized | ReentrantLock |
|------|-------------|---------------|
| 語法 | 關鍵字，簡潔 | 需要 try/finally |
| 可重入 | ✅ | ✅ |
| 超時取鎖 | ❌ | ✅ tryLock(timeout) |
| 可中斷等待 | ❌ | ✅ lockInterruptibly() |
| 公平鎖 | ❌（永遠非公平） | ✅ new ReentrantLock(true) |
| 查詢等待數 | ❌ | ✅ getQueueLength() |
| 適用場景 | 簡單同步 | 高並發、複雜鎖策略 |

## 十一、Semaphore：控制同時進入的執行緒數量

### Semaphore 解決什麼問題？

`synchronized` 只允許「一次只有 1 個執行緒」進入。  
但有時候你想讓「最多 N 個執行緒同時進入」，例如：
- 資料庫連線池：最多同時 20 條連線
- 限流：每秒最多處理 100 個請求
- 撮合引擎：限制同時查詢訂單簿的執行緒數

這就是 **Semaphore（號誌）** 的用途。

```
Semaphore 本質是一個「許可證計數器」：

初始：permits = 3（最多 3 個執行緒同時進入）

執行緒 A 呼叫 acquire()：permits = 2，A 進入
執行緒 B 呼叫 acquire()：permits = 1，B 進入
執行緒 C 呼叫 acquire()：permits = 0，C 進入
執行緒 D 呼叫 acquire()：permits = 0，D 阻塞等待...

執行緒 A 呼叫 release()：permits = 1，D 被喚醒進入
```

### 程式碼範例：資料庫連線池限流

```java
import java.util.concurrent.Semaphore;

public class DatabasePool {
    // 最多同時允許 10 條連線
    private final Semaphore semaphore = new Semaphore(10);

    public void query(String sql) throws InterruptedException {
        semaphore.acquire();  // 取得一個許可證（沒有就等）
        try {
            // 這裡最多只有 10 個執行緒同時執行
            executeQuery(sql);
        } finally {
            semaphore.release();  // 用完一定要還許可證！
        }
    }

    // 非阻塞版：試著取得許可，取不到就放棄
    public boolean tryQuery(String sql) {
        if (semaphore.tryAcquire()) {  // 不等待
            try {
                executeQuery(sql);
                return true;
            } finally {
                semaphore.release();
            }
        }
        return false;  // 現在太忙，稍後再試
    }
}
```

## 十二、CountDownLatch 與 CyclicBarrier：等待多執行緒完成

### CountDownLatch：「等所有人都做完，再繼續」

場景：你派出 5 個執行緒同時查詢 5 個不同的資料庫，等全部查完後才能彙總結果。

```
主執行緒 ─── 等待 ──▶ CountDownLatch(5)
                              │
         ┌────────────────────┼────────────────────┐
         ▼                    ▼                    ▼
     執行緒1.done()       執行緒2.done()       ...執行緒5.done()
     countDown()          countDown()          countDown()
         │                    │                    │
         └────────────────────▼────────────────────┘
                     計數器歸零 → 主執行緒繼續
```

```java
import java.util.concurrent.*;

public class ParallelQuery {
    public static void main(String[] args) throws InterruptedException {
        int taskCount = 5;
        CountDownLatch latch = new CountDownLatch(taskCount);  // 計數器設為 5
        ExecutorService pool = Executors.newFixedThreadPool(taskCount);

        List<String> results = new CopyOnWriteArrayList<>();

        for (int i = 0; i < taskCount; i++) {
            final int id = i;
            pool.submit(() -> {
                try {
                    // 模擬查詢不同的資料庫
                    String result = queryDatabase("DB-" + id);
                    results.add(result);
                } finally {
                    latch.countDown();  // 完成一個，計數器 -1
                    // 注意：放在 finally，確保即使例外也會 countDown
                }
            });
        }

        latch.await();  // 主執行緒阻塞，直到計數器歸零
        // 到這裡時，5 個查詢全部完成
        System.out.println("所有查詢完成，結果：" + results);
        pool.shutdown();
    }
}
```

**CountDownLatch 的限制：計數器只能用一次，歸零後無法重置。**

### CyclicBarrier：「所有人到齊後，一起出發」

場景：多執行緒分頭計算，算完後都等在同一個「集合點」，然後一起進行下一階段。

```java
import java.util.concurrent.*;

public class PhaseCalculation {
    public static void main(String[] args) throws Exception {
        int threads = 3;
        // 當 3 個執行緒都到達 barrier，自動執行 Runnable（可選）
        CyclicBarrier barrier = new CyclicBarrier(threads, () -> {
            System.out.println("=== 所有執行緒完成第一階段，開始第二階段 ===");
        });

        for (int i = 0; i < threads; i++) {
            final int id = i;
            new Thread(() -> {
                System.out.println("執行緒 " + id + " 完成第一階段計算");
                try {
                    barrier.await();  // 等其他執行緒也到達這裡
                } catch (Exception e) { e.printStackTrace(); }

                System.out.println("執行緒 " + id + " 開始第二階段");
                // CyclicBarrier 可以重複用，不像 CountDownLatch 只能用一次
            }).start();
        }
    }
}
```

### CountDownLatch vs CyclicBarrier

| | CountDownLatch | CyclicBarrier |
|---|---|---|
| 用途 | 等「別人」做完（主執行緒等子執行緒）| 互相等待（所有執行緒等彼此）|
| 可重置 | ❌（一次性） | ✅（可重複使用）|
| 計數方向 | 倒數到 0 | 計數到 N |
| 典型場景 | 並行查詢後彙總 | 多輪次的階段計算 |

## 十三、CAS 原理與 Lock-Free 資料結構 <!-- 🔴 資深 -->

### CAS（Compare-And-Swap）底層原理

CAS 是現代並發程式設計的基石。`synchronized` 是悲觀鎖（預設衝突一定發生），而 CAS 是**樂觀鎖**（假設衝突少見，先嘗試再說）。

```
CAS(記憶體地址 V, 預期值 A, 新值 B):
  if *V == A:
    *V = B
    return true   // 成功：沒有其他執行緒插手
  else:
    return false  // 失敗：有人改過了，重試
```

這個「讀取-比較-寫入」三步驟在 CPU 層面是**原子指令**（x86: `LOCK CMPXCHG`），作業系統不會在這條指令執行到一半時切換執行緒。

```java
// AtomicInteger 的 incrementAndGet() 底層實作
public final int incrementAndGet() {
    for (;;) {                                      // 自旋重試迴圈
        int current = get();                        // 讀取當前值
        int next = current + 1;                     // 計算新值
        if (compareAndSet(current, next))           // CAS 嘗試
            return next;                            // 成功回傳
        // 失敗：有人在這瞬間改了值，重新讀取再試
    }
}

// compareAndSet 底層呼叫 Unsafe.compareAndSwapInt
// 這是 Java 與 JVM native 原始碼的橋接點
```

### ABA 問題：CAS 的經典陷阱

```
時間線：
  Thread-1 讀到 A（value=1）
  Thread-2 把 1 → 2 → 1（值又回到1）
  Thread-1 的 CAS 看到值還是 1，以為沒人動過 → 成功
  但其實中間狀態已經改變，可能產生邏輯錯誤
```

**解法：加版本號**

```java
AtomicStampedReference<Integer> ref =
    new AtomicStampedReference<>(1, 0);  // (值, 版本號)

int[] stampHolder = new int[1];
Integer val = ref.get(stampHolder);      // 同時取出值和版本號
int stamp = stampHolder[0];

// CAS 必須值和版本號都匹配才成功
ref.compareAndSet(val, newVal, stamp, stamp + 1);
// A→B→A 時，版本號是 0→1→2，即使值回到 A，版本號不同，CAS 失敗
```

### Lock-Free Stack（無鎖堆疊）

```java
<!-- 🔴 資深 -->
public class LockFreeStack<T> {
    private final AtomicReference<Node<T>> top = new AtomicReference<>();

    static class Node<T> {
        final T value;
        Node<T> next;
        Node(T value) { this.value = value; }
    }

    public void push(T value) {
        Node<T> newNode = new Node<>(value);
        Node<T> currentTop;
        do {
            currentTop = top.get();          // 讀目前 top
            newNode.next = currentTop;       // 新節點指向舊 top
        } while (!top.compareAndSet(currentTop, newNode));
        // CAS 失敗代表有人搶先修改了 top，重試
    }

    public T pop() {
        Node<T> currentTop;
        Node<T> newTop;
        do {
            currentTop = top.get();
            if (currentTop == null) return null;  // 空堆疊
            newTop = currentTop.next;
        } while (!top.compareAndSet(currentTop, newTop));
        return currentTop.value;
    }
}
```

### Lock-Free Queue（JDK 的 ConcurrentLinkedQueue）

`ConcurrentLinkedQueue` 是 JDK 內建的無鎖佇列，基於 Michael-Scott 演算法：

```
結構：head → [dummy] → [node1] → [node2] → tail

offer(e)：
  1. 建立新節點 newNode
  2. 讀取 tail.next
  3. 若 tail.next == null，CAS 將其設為 newNode（成功則更新 tail）
  4. 否則先幫忙推進 tail 指針（協助推進 tail）再重試

poll()：
  1. 讀取 head.next（實際第一個元素）
  2. CAS 將 head 前進到 head.next
  3. 回傳舊 head.next 的值
```

**適用場景**：高競爭的生產者-消費者，且不能接受 `synchronized` 的延遲（如撮合引擎的訂單佇列）。

### 效能對比：悲觀鎖 vs 樂觀鎖

| 特性 | synchronized | ReentrantLock | CAS/Lock-Free |
|------|-------------|---------------|---------------|
| 機制 | OS mutex（核心態） | AQS（使用者態 + 核心態） | 純使用者態 |
| 無競爭開銷 | 偏向鎖很低 | 略高 | 最低 |
| 高競爭下 | 執行緒掛起/喚醒（慢） | 可自旋，略好 | 自旋重試，CPU 高但低延遲 |
| 適合場景 | 大多數業務邏輯 | 需要 tryLock/超時 | 極低延遲、高吞吐 |
| ABA 問題 | 無 | 無 | 需 AtomicStampedReference |

### 什麼時候用 Lock-Free？

CAS 自旋在競爭激烈時會浪費大量 CPU（所有執行緒都在原地空轉）。選擇原則：

- **競爭低、操作快**（< 微秒）→ CAS/Lock-Free 勝出
- **競爭高或操作複雜** → 有鎖方案往往更好（執行緒讓出 CPU 比空轉有效率）
- **生產優先**：先用 `ConcurrentLinkedQueue`、`AtomicLong` 等 JDK 成熟實作，不要自己重頭寫 Lock-Free 資料結構（極難驗證正確性）

## 本章練習題

**Q1：下面的程式碼有競態條件嗎？如何修復？**
```java
class BankAccount {
    private double balance = 1000.0;

    public void deposit(double amount) {
        balance += amount;
    }

    public void withdraw(double amount) {
        if (balance >= amount) {
            balance -= amount;
        }
    }
}
```
<details>
<summary>答案</summary>
有競態條件。balance += amount 和 balance -= amount 都不是原子的，而且 if 判斷和減法之間也可能被打斷（TOCTOU 問題）。修復方式：在 deposit 和 withdraw 都加上 synchronized，或者用 ReentrantLock。
</details>

**Q2：volatile 能讓 count++ 變成執行緒安全嗎？**
<details>
<summary>答案</summary>
不能。volatile 只保證可見性（每次讀寫都存取 RAM），但 count++ 是「讀-改-寫」三個步驟，volatile 無法讓這三步變成原子的。需要使用 AtomicInteger 或 synchronized。
</details>

**Q3：在撮合引擎中，訂單簿的 addOrder() 應該如何設計才能讓多執行緒安全？**
<details>
<summary>答案</summary>
方法一（簡單）：整個 addOrder 方法加 synchronized，但高並發時鎖競爭嚴重。方法二（高效能）：單執行緒模型 + 無鎖佇列（如 LMAX Disruptor），所有訂單先進入 RingBuffer，由單一撮合執行緒依序處理。這樣完全不需要鎖，延遲最低。
</details>
