## 一、CPU 流水線（Pipeline）

現代 CPU 不是「執行完一條指令再執行下一條」，而是**同時執行多條指令的不同階段**，就像工廠的流水線。

```
一條指令的執行分為多個階段：
取指 → 解碼 → 取運算元 → 執行 → 寫回

流水線：每個階段同時在處理不同指令
─────────────────────────────────────────────────
時鐘  →  1    2    3    4    5    6    7    8
指令1    取指 解碼 取元 執行 寫回
指令2         取指 解碼 取元 執行 寫回
指令3              取指 解碼 取元 執行 寫回
指令4                   取指 解碼 取元 執行 寫回

→ 理論上每個時鐘週期完成一條指令（CPI = 1）
```

**流水線中斷（Pipeline Stall）的代價很高：**
- 如果執行到一半遇到「分支指令」（if/else），CPU 不知道下一條指令是哪個
- 必須等分支結果出來才能繼續 → 流水線清空 → 損失 10~20 個 cycles

## 二、分支預測（Branch Prediction）

CPU 用「賭」的方式解決分支問題：根據歷史紀錄預測分支走向，提前取指。

```
if (x > 0) {     // ← 分支點
    doA();
} else {
    doB();
}
```

```
CPU 的賭法（動態分支預測）：

預測 x > 0 為 true（因為上次也是 true）
→ 提前載入 doA() 的指令

如果預測正確：
  doA() 的指令已在流水線裡 → 幾乎零延遲

如果預測錯誤（Misprediction）：
  流水線清空 → 損失 10~20 cycles
  現代 CPU 預測準確率約 95%~99%
```

### 對程式碼的影響

```java
// 壞：分支預測困難（每次都不確定走哪邊）
int[] randomData = generateRandom(1_000_000);  // 隨機 0 或 1
long sum = 0;
for (int x : randomData) {
    if (x > 0) sum += x;  // 50% true, 50% false，預測準確率低
}

// 好：先排序，讓分支結果可預測
Arrays.sort(randomData);  // 排序後：先是一堆 0，再是一堆 1
long sum = 0;
for (int x : randomData) {
    if (x > 0) sum += x;  // 先全是 false，後全是 true，預測準確率近 100%
}
```

**實際效果：** 在大量資料上，排序後的版本可能比未排序快 **3~10 倍**，僅僅因為分支預測命中率的差異。

### 在撮合引擎中的應用

```java
// 訂單到來時，大多數是限價單（Limit Order）
// 確保最常走的路徑是「分支預測友好」的
if (order.getType() == OrderType.LIMIT) {  // 通常是 true
    // 主要路徑（被 JIT 優化為快速路徑）
    matchLimitOrder(order);
} else {
    handleSpecialOrder(order);  // 較少走到這裡
}
```

## 三、False Sharing：隱藏的效能殺手

這是多執行緒效能問題中最難察覺的一個。

### CPU Cache Line（快取行）

CPU 不是一次讀一個 byte，而是一次讀一個 **Cache Line（通常 64 bytes）**：

```
Cache Line（64 bytes）：

地址  0    8   16   24   32   40   48   56   64
     [x]  [y]  [z]  [a]  [b]  [c]  [d]  [e]
      ↑                                       ↑
    long x                                  long e
    （8 bytes）

x 和 y 在同一個 Cache Line 裡！
```

### False Sharing 的問題

```java
class SharedCounters {
    long counter1 = 0;  // 執行緒 A 使用
    long counter2 = 0;  // 執行緒 B 使用
}
```

```
記憶體佈局：
[counter1（8 bytes）][counter2（8 bytes）] ← 同一個 Cache Line！

執行緒 A 在 Core 1 上修改 counter1：
→ Core 1 的 Cache Line 被標為「已修改」
→ Core 2 必須把自己的 Cache Line 設為無效，重新從 RAM 讀取
→ 即使 counter2 根本沒有被修改！

這就是 False Sharing：兩個執行緒修改「不同的變數」
但因為它們在同一個 Cache Line 上，互相干擾
```

```
False Sharing 的效能影響：

沒有 False Sharing：2 個執行緒各自計數，每秒約 5 億次
有 False Sharing：  2 個執行緒計數，每秒約 1 千萬次（慢 50 倍！）
```

### 解法：Padding（填充對齊）

```java
// 方法一：手動填充（Java 8 以前）
class PaddedCounters {
    // 讓 counter1 和 counter2 在不同的 Cache Line 上
    long counter1 = 0;
    // 填充 7 個 long（7 × 8 = 56 bytes），把 counter2 推到下一個 Cache Line
    long p1, p2, p3, p4, p5, p6, p7;
    long counter2 = 0;
    long p8, p9, p10, p11, p12, p13, p14;
}

// 方法二：@Contended 注解（Java 8+，需要 -XX:-RestrictContended）
import sun.misc.Contended;

class ModernPaddedCounters {
    @Contended
    long counter1 = 0;

    @Contended
    long counter2 = 0;
    // JVM 自動在這兩個欄位前後填充 128 bytes
}
```

**LMAX Disruptor 的 RingBuffer 大量使用 Padding 技術，這是它效能超群的原因之一。**

## 四、記憶體屏障（Memory Barrier / Memory Fence）

這是 `volatile` 和 `synchronized` 的底層機制。

```
問題：CPU 和編譯器都會對指令重新排序（Reorder）來優化效能

例如：
// 原始碼
x = 1;
y = 2;
z = x + y;

// 編譯器/CPU 可能重排為：
y = 2;
x = 1;
z = x + y;
// 結果一樣，但 y=2 先做了
```

**多執行緒下，重排序可能導致問題：**

```java
// 執行緒 A
value = 42;           // (1)
initialized = true;   // (2) 如果被重排到 (1) 前面...

// 執行緒 B
if (initialized) {    // (3) 可能看到 initialized=true
    use(value);       // (4) 但 value 還是 0！
}
```

### Memory Barrier 的作用

```
Store Barrier（寫屏障）：
  屏障前的所有寫入，必須在屏障後的寫入之前完成
  並且對其他 CPU Core 可見

Load Barrier（讀屏障）：
  屏障前的讀取，必須發生在屏障後的讀取之前
  並且必須看到最新值（不能用 Cache 中的舊值）

Full Barrier（全屏障）：
  = Store Barrier + Load Barrier
  synchronized 就是用全屏障實現的
```

```java
// Java 層面的記憶體屏障對應關係：
synchronized 區塊：
  進入 → Load Barrier（確保看到最新值）
  離開 → Store Barrier（確保寫入對外可見）

volatile 寫入 → Store Barrier
volatile 讀取 → Load Barrier
```

## 五、NUMA 架構（Non-Uniform Memory Access）

現代伺服器通常有多個 CPU Socket，每個 Socket 有自己的「本地記憶體」：

```
┌──────────────────────────────────────────────────────┐
│                    伺服器主機板                       │
│                                                      │
│  ┌─────────────────┐        ┌─────────────────────┐ │
│  │   CPU Socket 0  │        │   CPU Socket 1      │ │
│  │  ┌──────────┐   │        │   ┌──────────┐      │ │
│  │  │ Core 0-7 │   │◀──QPI──▶  │ Core 8-15│      │ │
│  │  └──────────┘   │  橋接  │   └──────────┘      │ │
│  │  ┌──────────┐   │        │   ┌──────────┐      │ │
│  │  │ RAM 32GB │   │        │   │ RAM 32GB │      │ │
│  │  └──────────┘   │        │   └──────────┘      │ │
│  └─────────────────┘        └─────────────────────┘ │
│                                                      │
│  Core 0 存取 Socket 0 的 RAM：~70 ns（本地）          │
│  Core 0 存取 Socket 1 的 RAM：~140 ns（遠端，慢 2 倍）│
└──────────────────────────────────────────────────────┘
```

## 八、ILP：指令級並行（Instruction-Level Parallelism）

### CPU 不是「一次跑一條指令」

你可能以為 CPU 是照順序一條一條執行指令，但現代 CPU 其實像一條有多個工作站的**生產線**，同時在處理好幾條指令：

```
沒有 Pipeline（舊 CPU）：
  時間：  1    2    3    4    5    6
  指令1： [取指][解碼][執行][寫回]
  指令2：                   [取指][解碼]...

有 Pipeline（現代 CPU，4 級為例）：
  時間：  1    2    3    4    5    6    7    8
  指令1： [取指][解碼][執行][寫回]
  指令2：       [取指][解碼][執行][寫回]
  指令3：             [取指][解碼][執行][寫回]
  指令4：                   [取指][解碼][執行][寫回]

第 4 個時間週期開始後，4 條指令同時在不同階段執行！
→ 理論上 4 級 pipeline = 4 倍吞吐量
```

現代高端 CPU（如 Intel Core i9）有 **14~19 級 Pipeline**，且有多個執行單元可同時執行多條無相依性的指令（超純量架構）。

### 什麼會破壞 Pipeline（Pipeline Hazard）？

```
危害一：資料相依（Data Hazard）
  指令 A：r1 = r2 + r3      ← 計算結果存入 r1
  指令 B：r4 = r1 * 2       ← 需要等 A 的結果！CPU 必須暫停（stall）

  解法：亂序執行（Out-of-Order Execution）
  CPU 看到 B 要等 A，就先去執行後面不相依的指令 C、D
  → 這對開發者透明，結果與順序執行相同

危害二：分支（Control Hazard）← 第五節已介紹
  if/while/for → CPU 不知道下一條指令是哪裡
  → 猜錯一次 = 清空 pipeline = 浪費 10~20 個 cycles
  → 這就是「分支預測」要解決的問題

危害三：記憶體存取（Memory Hazard）
  指令 A：store [addr], r1   ← 寫入記憶體
  指令 B：load  r2, [addr]   ← 讀取同一地址！需要等 A 完成
```

### Java 寫法如何影響 ILP

```java
// ❌ 寫法一：有資料相依，CPU 難以並行
double total = 0;
for (int i = 0; i < prices.length; i++) {
    total += prices[i];  // 每次都依賴上一次的 total → 必須串行
}
// CPU 看到每條加法都依賴前一條的結果，無法並行

// ✅ 寫法二：拆成多個累加器，讓 CPU 並行執行
double sum1 = 0, sum2 = 0, sum3 = 0, sum4 = 0;
for (int i = 0; i < prices.length - 3; i += 4) {
    sum1 += prices[i];      // 這四條指令互相獨立！
    sum2 += prices[i + 1];  // CPU 可以同時執行它們
    sum3 += prices[i + 2];
    sum4 += prices[i + 3];
}
double total = sum1 + sum2 + sum3 + sum4;
// 在 4 個執行單元的 CPU 上，理論速度接近 4 倍
// 但注意：JIT 通常會自動做這個優化（迴圈展開），你不一定要手動寫
```

## 九、SIMD：單指令多資料（Single Instruction, Multiple Data）

### 什麼是 SIMD？

傳統指令：**一條指令處理一筆資料**  
SIMD 指令：**一條指令同時處理 N 筆資料**

```
傳統（純量）運算：
  add r1, r2  → 只計算一對數字

SIMD（向量）運算（AVX2，256-bit 暫存器）：
  256 bits ÷ 64 bits(double) = 4 個 double

  [a1, a2, a3, a4]   ← ymm0 暫存器（256-bit）
+
  [b1, b2, b3, b4]   ← ymm1 暫存器
=
  [a1+b1, a2+b2, a3+b3, a4+b4]   ← 一條 vaddpd 指令完成 4 次加法！

SIMD 指令集演進：
  SSE2   (2001)：128-bit，一次處理 2 個 double 或 4 個 float
  AVX    (2011)：256-bit，一次處理 4 個 double 或 8 個 float
  AVX2   (2013)：256-bit + 整數向量支援
  AVX-512(2017)：512-bit，一次處理 8 個 double（現代伺服器 CPU）
```

### JVM 的 Auto-Vectorization（自動向量化）

JIT C2 編譯器會嘗試自動把「符合條件的迴圈」編譯成 SIMD 指令：

```java
// JIT 可以自動向量化的模式（條件：連續陣列、簡單迴圈、無複雜分支）

// ✅ 範例一：陣列批量運算 → JIT 自動用 SIMD
double[] prices = new double[1000];
double factor = 1.05;

for (int i = 0; i < prices.length; i++) {
    prices[i] *= factor;  // 簡單的純量乘法 → JIT 向量化成 4x 並行
}
// 優化前：1000 條乘法指令
// 優化後（AVX2）：250 條 vmuLpd 指令，速度接近 4 倍

// ✅ 範例二：陣列加總 → 可向量化
double sum = 0;
for (double p : prices) {
    sum += p;  // JIT 可能用 SIMD 加速（但需要 reduction 支援）
}

// ❌ 不能向量化的情況：
for (int i = 0; i < prices.length; i++) {
    if (prices[i] > 0) {          // 複雜的條件分支
        prices[i] = Math.log(prices[i]);  // 複雜函式呼叫
    }
}
// 分支太複雜，JIT 放棄向量化
```

### 驗證 JIT 是否做了向量化

```bash
# 查看 JIT 生成的機器碼（需要 hsdis 函式庫）
java -XX:+UnlockDiagnosticVMOptions \
     -XX:+PrintAssembly \
     -XX:CompileOnly=MyClass::myMethod \
     MyApp

# 在輸出中找到 SIMD 指令：
#   vmulpd  → 向量化的雙精度乘法（SIMD）
#   mulsd   → 純量雙精度乘法（沒有向量化）
#   vaddpd  → 向量化的雙精度加法
#   vbroadcastsd → 把一個值廣播到整個向量暫存器
```

### 撮合引擎中 SIMD 的實際應用

```
撮合引擎哪些地方能用到 SIMD？

場景一：批量計算成交金額
  double[] quantities = ...;  // 1000 筆訂單的數量
  double[] prices     = ...;  // 1000 筆訂單的價格
  double[] values     = ...;  // values[i] = quantities[i] * prices[i]

  → 簡單陣列乘法，JIT 自動向量化 ✅
  → AVX2 一次算 4 筆，速度 4x

場景二：批量篩選買賣訂單
  for (int i = 0; i < orders.length; i++) {
      if (orders[i].getPrice() >= bestAsk) { ... }
  }
  → 條件分支，向量化較難，但 AVX-512 有 mask 指令支援

場景三：Market Risk 計算（VaR）
  計算大量持倉的加權平均、標準差
  → 大量浮點數運算，非常適合 SIMD，金融 quant 程式常手動使用

重要提醒：
  在撮合引擎的核心路徑（每筆訂單的撮合邏輯），
  通常一次只處理一筆訂單，SIMD 的優勢不明顯。
  SIMD 的優勢在於「批量處理」，例如：
  - 收盤後批量計算損益
  - 風控系統批量掃描持倉
  - 行情系統批量推送更新
```

## 本章練習題

**Q1：為什麼「先排序再搜尋」有時比直接搜尋快，即使排序本身需要時間？**
<details>
<summary>答案</summary>
排序後的資料有兩個好處：(1) 可以用二分搜尋（O(log n)）代替線性搜尋（O(n)）；(2) 排序後的迴圈中分支方向更規律（先全 false 後全 true），CPU 分支預測命中率大幅提升，減少 pipeline stall。在大量重複搜尋的場景下，一次排序的代價遠小於多次預測錯誤的代價。
</details>

**Q2：以下程式碼存在 False Sharing 問題嗎？如何修復？**
```java
class Stats {
    long totalOrders;    // 執行緒 A 更新
    long totalTrades;    // 執行緒 B 更新
    long totalVolume;    // 執行緒 C 更新
}
```
<details>
<summary>答案</summary>
有嚴重的 False Sharing。三個欄位各 8 bytes，合計 24 bytes，全部在同一個 64-byte Cache Line 裡。任何一個執行緒更新都會讓其他 Core 的 Cache Line 失效。修復：用 @Contended 標記每個欄位，或在每個欄位後手動填充 56 bytes，讓每個欄位獨占一個 Cache Line。
</details>

**Q3：為什麼 LMAX Disruptor 的 RingBuffer 大小必須是 2 的次方（如 1024、65536）？**
<details>
<summary>答案</summary>
索引計算時需要取模（sequence % bufferSize）。當 bufferSize 是 2 的次方時，取模可以用位元運算代替：sequence & (bufferSize - 1)，這是一條指令，比除法（多條指令）快很多。例如 bufferSize=1024 時，1024-1=1023=0b1111111111，sequence & 1023 直接取最低 10 位。
</details>
