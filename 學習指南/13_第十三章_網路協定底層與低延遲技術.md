# 第十三章：網路協定底層與低延遲技術
> 理解封包如何從鍵盤抵達撮合引擎，以及每一層如何影響延遲

---

## 前言：為什麼撮合引擎要學網路底層？

一筆訂單從交易員的鍵盤到撮合完成，**網路延遲往往是最大的瓶頸**。  
理解協定底層，才能知道在哪裡可以「少跑幾步」。

---

## 一、OSI 七層模型 vs TCP/IP 四層

```
OSI 模型（理論）        TCP/IP（實際）        對應技術
─────────────────────────────────────────────────────
7. 應用層               應用層               HTTP, FIX, ITCH（交易協定）
6. 表示層                  ↑
5. 會話層                  ↑
─────────────────────────────────────────────────────
4. 傳輸層               傳輸層               TCP, UDP
─────────────────────────────────────────────────────
3. 網路層               網路層               IP, ICMP
─────────────────────────────────────────────────────
2. 資料連結層           網路介面層            Ethernet, ARP
1. 實體層                  ↑               網路線, 光纖, 電信號
```

**封包的旅程（傳送方）：**
```
你的程式（HTTP/FIX 訊息）
      ↓  加上 TCP 標頭（16 bytes）
   TCP Segment
      ↓  加上 IP 標頭（20 bytes）
   IP Packet
      ↓  加上 Ethernet 標頭（14 bytes）
   Ethernet Frame
      ↓  轉成電信號 / 光信號
   在網路線上傳輸
```

## 二、TCP：可靠但有代價

TCP（Transmission Control Protocol）提供三大保證：
1. **可靠傳輸**：遺失的封包會重送
2. **順序保證**：亂序到達的封包會重組
3. **流量控制**：不讓快速發送方淹沒慢速接收方

### TCP 三次握手（建立連線）

```
客戶端（交易員系統）          伺服器（交易所）
      │                          │
      │──── SYN（seq=100）───────▶│
      │    「我想連線，我的序號是100」
      │                          │
      │◀──── SYN+ACK（seq=300, ack=101）─│
      │    「好，我的序號是300，確認你的101」
      │                          │
      │──── ACK（ack=301）───────▶│
      │    「確認你的301」
      │                          │
         ✅ 連線建立！可以傳資料了
         ⏱ 代價：1.5 個 RTT（Round-Trip Time）
```

```
RTT 範例：
台北 → 紐約：~150 ms × 1.5 = 225 ms 僅建立連線！
台北 → 台灣交易所：~1 ms × 1.5 = 1.5 ms

→ 這就是為什麼高頻交易要把伺服器「co-location」（共址）
  放在交易所機房內，RTT < 0.1 ms
```

### TCP 四次揮手（斷開連線）

```
客戶端                       伺服器
   │──── FIN ──────────────▶│   「我傳完了」
   │◀─── ACK ───────────────│   「收到」
   │◀─── FIN ───────────────│   「我也傳完了」
   │──── ACK ──────────────▶│   「收到」

代價：2 個 RTT，還有 TIME_WAIT 狀態（預設 2 分鐘！）
```

## 三、Nagle 演算法：小封包的效能殺手

Nagle 演算法的目的：**合併小封包，減少網路擁塞**。

```
預設行為（啟用 Nagle）：
發送 1 byte ──▶ 等等……還有沒有更多資料？
                  等待 200ms……
發送 1 byte ──▶ 
                  好了，把這兩個合成一個封包發送

→ 總延遲：200ms
```

**對撮合引擎的影響：災難性！**

```java
// Java 中關閉 Nagle 演算法（交易系統必做）
Socket socket = new Socket();
socket.setTcpNoDelay(true);  // TCP_NODELAY
// 效果：每個 write() 立刻發送，不等待合併

ServerSocket serverSocket = new ServerSocket(9999);
Socket clientSocket = serverSocket.accept();
clientSocket.setTcpNoDelay(true);
```

```
關閉 Nagle 後：
發送 1 byte ──▶ 立刻發送
發送 1 byte ──▶ 立刻發送

→ 總延遲：< 1ms（只有傳播延遲）
```

## 四、TCP vs UDP：低延遲的取捨

```
               TCP                     UDP
─────────────────────────────────────────────────────
連線建立       需要 3 次握手            無連線，直接發
可靠性         保證送達                 不保證送達
順序保證       有                       沒有
流量控制       有（滑動視窗）            沒有
壅塞控制       有（慢啟動、擁塞避免）     沒有
標頭大小       20 bytes                 8 bytes
延遲           較高（重傳等待）          較低
─────────────────────────────────────────────────────
適合場景       訂單傳送（要求可靠）       市場行情廣播
```

### UDP 在交易市場中的應用：行情廣播

```java
// UDP 發送行情（交易所行情廣播）
DatagramSocket socket = new DatagramSocket();
byte[] data = "TSLA 350.00 1000".getBytes();
DatagramPacket packet = new DatagramPacket(
    data, data.length,
    InetAddress.getByName("multicast.exchange.tw"), 5000
);
socket.send(packet);

// UDP 接收行情（客戶端）
DatagramSocket receiver = new DatagramSocket(5000);
byte[] buffer = new byte[1024];
DatagramPacket incoming = new DatagramPacket(buffer, buffer.length);
receiver.receive(incoming);
String quote = new String(incoming.getData(), 0, incoming.getLength());
```

```
Multicast（組播）：
交易所發送一次行情 ──▶ 同時到達所有訂閱者
                       （比 Unicast 一個個發效率高得多）
```

## 五、Java NIO：非阻塞 I/O

傳統 Java I/O（BIO）的問題：

```
每個連線需要一個執行緒：
連線 1 ──▶ 執行緒 1（可能 99% 時間在等待資料）
連線 2 ──▶ 執行緒 2
連線 3 ──▶ 執行緒 3
...
連線 10000 ──▶ 執行緒 10000（記憶體爆炸！）
```

**Java NIO（Non-blocking I/O）的解法：Selector 模型**

```java
import java.nio.*;
import java.nio.channels.*;
import java.net.InetSocketAddress;

// NIO Server 骨架：一個執行緒處理多個連線
Selector selector = Selector.open();

ServerSocketChannel serverChannel = ServerSocketChannel.open();
serverChannel.bind(new InetSocketAddress(9999));
serverChannel.configureBlocking(false);  // 非阻塞模式
serverChannel.register(selector, SelectionKey.OP_ACCEPT);

while (true) {
    selector.select();  // 等待任何一個 channel 就緒
    
    Set<SelectionKey> keys = selector.selectedKeys();
    for (SelectionKey key : keys) {
        if (key.isAcceptable()) {
            // 有新連線
            SocketChannel client = serverChannel.accept();
            client.configureBlocking(false);
            client.register(selector, SelectionKey.OP_READ);
            
        } else if (key.isReadable()) {
            // 某個連線有資料可讀
            SocketChannel client = (SocketChannel) key.channel();
            ByteBuffer buffer = ByteBuffer.allocate(1024);
            int bytesRead = client.read(buffer);
            // 處理資料...
        }
    }
    keys.clear();
}
```

```
NIO Selector 模型：
一個執行緒 ──▶ 同時監控 10000 個連線
              只有「有資料」的連線才會喚醒執行緒
              
→ 這是現代 Web 伺服器（Netty）的核心模型
→ 交易所的行情閘道通常用 Netty 建構
```

## 六、ByteBuffer：直接記憶體操作

```java
// 標準 ByteBuffer
ByteBuffer heapBuffer = ByteBuffer.allocate(1024);    // 在 JVM Heap 上
ByteBuffer directBuffer = ByteBuffer.allocateDirect(1024);  // 在 OS 記憶體（不被 GC 管）

// ByteBuffer 的核心概念：position, limit, capacity
// capacity = 最大容量
// limit    = 可讀/可寫的邊界
// position = 目前讀/寫位置

ByteBuffer buf = ByteBuffer.allocate(10);
// 初始狀態：position=0, limit=10, capacity=10

buf.putInt(42);      // 寫入 4 bytes，position 移到 4
buf.putDouble(3.14); // 寫入 8 bytes，position 移到 12 ← 超過 10 會報錯！

buf.flip();          // 準備讀取：limit=position, position=0
int value = buf.getInt();      // 讀取 4 bytes，position 移到 4
double pi = buf.getDouble();   // 讀取 8 bytes

buf.clear();         // 準備重新寫入：position=0, limit=capacity
```

**為什麼 Direct Buffer 對交易系統重要？**

```
Heap Buffer（普通 ByteBuffer）：
JVM Heap → （複製一份）→ OS 緩衝區 → 網路卡發送
              ↑ 這個複製步驟稱為「中間複製」

Direct Buffer（直接記憶體）：
OS 直接記憶體（不在 JVM Heap） → 網路卡發送
              ↑ 省去一次複製！

→ 這就是 Zero Copy（零拷貝）的一部分
```

## 七、Kernel Bypass：跳過作業系統

這是最激進的低延遲技術，直接讓應用程式存取網路卡硬體。

```
傳統路徑（Software Stack）：
應用程式 → OS Kernel（TCP/IP 處理）→ 驅動程式 → 網路卡
代價：~10 μs（Kernel 處理時間）

Kernel Bypass（DPDK / io_uring）：
應用程式 ────────────────────────────────▶ 網路卡
         跳過 OS Kernel，直接控制硬體！
代價：< 1 μs
```

**Java 中的近似技術：io_uring（JDK 21+ Project Loom）**

```java
// 傳統阻塞 I/O（BIO）
InputStream in = socket.getInputStream();
byte[] data = in.read();  // 阻塞！OS 進入 Kernel 模式，等待資料

// Virtual Thread（JDK 21）+ 底層 io_uring
Thread.startVirtualThread(() -> {
    // 看起來像阻塞，但底層用 io_uring 非阻塞處理
    // OS 不需要每次都切換 Kernel 模式
    byte[] data = socket.getInputStream().read();
});
```

```
io_uring 的原理（Ring Buffer 佇列）：

提交佇列（SQ）：應用程式放入 I/O 請求
完成佇列（CQ）：OS 完成後放入結果

應用程式           OS Kernel
    │──提交請求──▶ SQ
    │              （OS 非同步處理，不阻塞應用程式）
    │◀──完成通知── CQ
    
→ 比 epoll 少一次系統呼叫（syscall）
→ 撮合引擎可以每批次處理 100 個訂單，不是每次都進 Kernel
```

## 八、RDMA：直接記憶體存取（最頂級的技術）

```
傳統 TCP/IP 訊息傳遞：
Server A Memory → (複製) → NIC → 網路 → NIC → (複製) → Server B Memory
代價：CPU 介入、多次複製、延遲 ~50 μs

RDMA（Remote Direct Memory Access）：
Server A Memory ──────────────────────────────────────▶ Server B Memory
              網路卡直接讀寫對方的記憶體，CPU 完全不介入！
代價：~1-3 μs，接近光速傳播延遲
```

**誰在用 RDMA？**
- 高頻交易公司（HFT）
- 大型雲端業者（AWS EFA、Azure InfiniBand）
- 高性能運算叢集（HPC）

## 九、FIX 協定：金融交易的通用語言

FIX（Financial Information eXchange）是交易行業標準協定。

```
一條 FIX 訊息（限價買入 100 股 TSLA @ 350.00）：
8=FIX.4.4|9=176|35=D|49=CLIENT|56=EXCHANGE|34=3|52=20240512-10:30:00.123|
11=ORDER001|55=TSLA|54=1|38=100|40=2|44=350.00|60=20240512-10:30:00.123|10=123|

解析：
  35=D       → MsgType = New Order Single（新訂單）
  55=TSLA    → Symbol（股票代碼）
  54=1       → Side = 1（買入）
  38=100     → OrderQty（數量）
  40=2       → OrdType = 2（限價單）
  44=350.00  → Price（限價）
```

```java
// 在 Java 中處理 FIX 訊息（使用 QuickFIX/J 函式庫）
import quickfix.*;

public class MyApplication implements Application {
    @Override
    public void fromApp(Message message, SessionID sessionID) {
        try {
            // 解析訂單
            String symbol = message.getString(Symbol.FIELD);    // Tag 55
            char side = message.getChar(Side.FIELD);             // Tag 54
            double price = message.getDouble(Price.FIELD);       // Tag 44
            int qty = message.getInt(OrderQty.FIELD);            // Tag 38
            
            // 傳入撮合引擎
            Order order = new Order(symbol, side == Side.BUY ? "BUY" : "SELL",
                                     price, qty);
            orderBook.addOrder(order);
        } catch (FieldNotFound e) {
            // 處理欄位缺失
        }
    }
}
```

## 十、完整的低延遲網路棧設計

```
交易所的低延遲訂單路徑（優化版）：

客戶端 ──(UDP/RDMA)──▶ 交易所 NIC
                           │
                    （Kernel Bypass / DPDK）
                           │
                    Ring Buffer（無鎖佇列）
                           │
                    訂單解析（Zero-copy parsing）
                           │
                    撮合引擎（單執行緒 + CPU 綁定）
                           │
                    成交回報（UDP Multicast 廣播）
                           ↓
                      所有訂閱者收到行情

總延遲（頂級 HFT）：< 10 μs（端到端）
典型 Java 撮合引擎（優化後）：20~100 μs
```

## 本章練習題

**Q1：為什麼高頻交易公司要把伺服器放在交易所機房（co-location）？用網路協定原理解釋。**
<details>
<summary>答案</summary>
TCP 建立連線需要 1.5 個 RTT，每次訂單傳送至少也需要 1 個 RTT。光在光纖中的傳播速度約 200,000 km/s，台北到紐約約 12,000 km，單程需要 60 ms，RTT 約 120 ms。而台北到台灣交易所可能只有 1 km，RTT < 0.01 ms。co-location 讓傳播延遲幾乎為零，成為速度競爭的基礎設施優勢。
</details>

**Q2：TCP_NODELAY 是什麼？什麼情況下必須開啟它？**
<details>
<summary>答案</summary>
TCP_NODELAY 關閉 Nagle 演算法。預設情況下，Nagle 演算法會等待 200ms 或直到有足夠多的資料才發送封包，目的是減少小封包造成的網路擁塞。但對於交易訂單傳送，每條訂單必須立即發送，200ms 的延遲是不可接受的。所以交易系統中必須設定 socket.setTcpNoDelay(true)。
</details>

**Q3：什麼是 Zero Copy？為什麼它對低延遲系統重要？**
<details>
<summary>答案</summary>
Zero Copy 是指資料在傳輸過程中不需要 CPU 進行額外的記憶體複製。傳統方式中，資料從網路到應用程式需要：NIC → Kernel Buffer（複製1）→ User Space Buffer（複製2）→ 應用程式記憶體。Zero Copy 技術（如 sendfile(), mmap(), Direct Buffer）讓資料直接在來源和目的地之間傳輸，CPU 不需要中間搬運資料，不僅減少延遲，也降低 CPU 使用率，讓 CPU 資源可以用在撮合引擎的核心邏輯上。
</details>

**Q4：UDP 和 TCP 各自適合交易系統中的哪些場景？**
<details>
<summary>答案</summary>
TCP 適合「訂單傳送」：訂單必須可靠地到達交易所，不能丟失，需要確認機制和重傳。UDP 適合「行情廣播」：交易所向數千個訂閱者廣播市場行情，即時性比可靠性更重要，丟一筆行情可以等下一筆，不需要重傳機制。而且 UDP 支援 Multicast，一次發送全部人收到，比 TCP 的一對一連線節省大量頻寬。
</details>
