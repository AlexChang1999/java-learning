# 第十五章：設計模式（Design Patterns）
> 工程師讀別人的程式碼時，模式是共同語言；寫自己的程式碼時，模式是解題工具

---

## 前言：模式不是要背的，是要認得的

設計模式是「已被無數人驗證過的解法」。  
你不需要在設計時強行套用，但當你遇到問題時，**你要能認出「這個問題有一個名字」**。

本章用你已經寫過的撮合引擎程式碼來認識這些模式——它們一直都在，只是還沒被命名。

---

## 一、創建型模式（Creational Patterns）

### 1. Singleton（單例）

**問題：** 某個物件只應該存在一個實例（例如設定檔、資料庫連線池）。

```java
// 反面教材：到處 new
OrderBook book1 = new OrderBook("TSLA");
OrderBook book2 = new OrderBook("TSLA");  // 同一個商品兩本書！Bug！

// Singleton 解法：確保只有一個實例
public class ExchangeConfig {
    // volatile 防止 JIT 重排序（第十二章學過的）
    private static volatile ExchangeConfig instance;

    private final int maxOrderQty = 100_000;
    private final double maxPriceDeviation = 0.1;  // 10%

    private ExchangeConfig() {}  // 私有建構子，外部不能 new

    public static ExchangeConfig getInstance() {
        if (instance == null) {
            synchronized (ExchangeConfig.class) {  // Double-Checked Locking
                if (instance == null) {
                    instance = new ExchangeConfig();
                }
            }
        }
        return instance;
    }

    public int getMaxOrderQty() { return maxOrderQty; }
}

// 使用
ExchangeConfig config = ExchangeConfig.getInstance();
int max = config.getMaxOrderQty();
```

**Java 標準庫的 Singleton：**
- `Runtime.getRuntime()` — JVM 執行環境
- `System.out` — 標準輸出（PrintStream 的單例）

**現代 Java 的更好做法（enum Singleton）：**
```java
// 天生執行緒安全，且能防止序列化攻擊
public enum ExchangeConfig {
    INSTANCE;

    private final int maxOrderQty = 100_000;
    public int getMaxOrderQty() { return maxOrderQty; }
}

// 使用
ExchangeConfig.INSTANCE.getMaxOrderQty();
```

### 2. Factory Method（工廠方法）

**問題：** 建立物件的邏輯很複雜，或者需要根據條件建立不同子類別。

```java
// 不用工廠：每個地方都有判斷邏輯，難以維護
if (type.equals("LIMIT")) {
    order = new LimitOrder(symbol, side, price, qty);
} else if (type.equals("MARKET")) {
    order = new MarketOrder(symbol, side, qty);
} else if (type.equals("IOC")) {
    order = new IOCOrder(symbol, side, price, qty);
}
// 每個地方都有這段...
```

```java
// 工廠方法：建立邏輯集中在一個地方
public class OrderFactory {

    public static Order create(String id, String symbol, String side,
                                String type, double price, int qty) {
        return switch (type.toUpperCase()) {
            case "LIMIT"  -> new Order(id, symbol, side, OrderType.LIMIT,  price, qty);
            case "MARKET" -> new Order(id, symbol, side, OrderType.MARKET, 0.0,   qty);
            case "IOC"    -> new Order(id, symbol, side, OrderType.IOC,    price, qty);
            case "FOK"    -> new Order(id, symbol, side, OrderType.FOK,    price, qty);
            default -> throw new IllegalArgumentException("不支援的訂單類型：" + type);
        };
    }
}

// 使用：只需要一行，不需要知道建立細節
Order order = OrderFactory.create("O-001", "TSLA", "BUY", "LIMIT", 350.0, 100);
```

**Java 標準庫的 Factory Method：**
- `List.of(...)`, `Map.of(...)` — 根據元素數量選擇最佳實作
- `Executors.newFixedThreadPool(n)` — 隱藏了 ThreadPoolExecutor 的複雜建構

### 3. Builder（建造者）

**問題：** 物件有很多可選參數，建構子越寫越長，難以閱讀。

```java
// 反面教材：望文生義困難
Order order = new Order("O-001", "TSLA", "BUY", OrderType.LIMIT,
                         350.0, 100, true, false, "CLIENT-A", null);
// 第 7 個 boolean 是什麼意思？第 8 個呢？

// Builder 解法：清晰、可選、可讀
Order order = new Order.Builder("O-001", "TSLA")
    .side("BUY")
    .type(OrderType.LIMIT)
    .price(350.0)
    .quantity(100)
    .clientId("CLIENT-A")
    // .expiry(Instant.now().plusMinutes(30))  // 選填
    .build();
```

```java
// 實作 Builder
public class Order {
    // 必填欄位
    private final String orderId;
    private final String symbol;
    // 選填欄位
    private final String side;
    private final OrderType type;
    private final double price;
    private final int quantity;
    private final String clientId;

    private Order(Builder builder) {
        this.orderId   = builder.orderId;
        this.symbol    = builder.symbol;
        this.side      = builder.side;
        this.type      = builder.type;
        this.price     = builder.price;
        this.quantity  = builder.quantity;
        this.clientId  = builder.clientId;
    }

    public static class Builder {
        private final String orderId;   // 必填
        private final String symbol;    // 必填
        private String side;
        private OrderType type = OrderType.LIMIT;  // 預設值
        private double price = 0.0;
        private int quantity;
        private String clientId = "ANONYMOUS";

        public Builder(String orderId, String symbol) {
            this.orderId = orderId;
            this.symbol = symbol;
        }

        public Builder side(String side)         { this.side = side; return this; }
        public Builder type(OrderType type)      { this.type = type; return this; }
        public Builder price(double price)       { this.price = price; return this; }
        public Builder quantity(int qty)         { this.quantity = qty; return this; }
        public Builder clientId(String clientId) { this.clientId = clientId; return this; }

        public Order build() {
            if (side == null) throw new IllegalStateException("side 為必填");
            if (quantity <= 0) throw new IllegalStateException("數量必須大於 0");
            return new Order(this);
        }
    }
}
```

**Java 標準庫的 Builder：**
- `StringBuilder` — 字串建造者
- `HttpRequest.newBuilder()` — Java 11 HTTP Client
- `ProcessBuilder` — 建立子行程

## 二、結構型模式（Structural Patterns）

### 4. Decorator（裝飾者）

你在第三章已經用過這個模式！

```java
// 第三章的 I/O 流包裝
new BufferedReader(
    new InputStreamReader(
        new FileInputStream("data.txt"),
        "UTF-8"
    )
)
// FileInputStream → InputStreamReader → BufferedReader
// 每層「包裝」增加新功能（解碼 + 緩衝），但核心（讀取位元組）不變
```

**在撮合引擎中應用 Decorator：**
```java
// 基本 OrderBook 介面
public interface IOrderBook {
    List<Trade> addOrder(Order order);
    boolean cancelOrder(String orderId);
}

// 核心實作
public class OrderBook implements IOrderBook { ... }

// 裝飾者一：加上風控檢查
public class RiskCheckOrderBook implements IOrderBook {
    private final IOrderBook wrapped;
    private final ExchangeConfig config;

    public RiskCheckOrderBook(IOrderBook wrapped) {
        this.wrapped = wrapped;
        this.config = ExchangeConfig.INSTANCE;
    }

    @Override
    public List<Trade> addOrder(Order order) {
        // 加入前先做風控
        if (order.getQuantity() > config.getMaxOrderQty()) {
            throw new RiskException("訂單數量超過限制：" + order.getQuantity());
        }
        return wrapped.addOrder(order);  // 通過就繼續
    }

    @Override
    public boolean cancelOrder(String orderId) {
        return wrapped.cancelOrder(orderId);
    }
}

// 裝飾者二：加上日誌記錄
public class LoggingOrderBook implements IOrderBook {
    private final IOrderBook wrapped;

    public LoggingOrderBook(IOrderBook wrapped) {
        this.wrapped = wrapped;
    }

    @Override
    public List<Trade> addOrder(Order order) {
        System.out.printf("[LOG] 收到訂單：%s%n", order.getOrderId());
        List<Trade> trades = wrapped.addOrder(order);
        System.out.printf("[LOG] 成交 %d 筆%n", trades.size());
        return trades;
    }

    @Override
    public boolean cancelOrder(String orderId) {
        System.out.printf("[LOG] 取消訂單：%s%n", orderId);
        return wrapped.cancelOrder(orderId);
    }
}

// 組合使用：包三層
IOrderBook book =
    new LoggingOrderBook(
        new RiskCheckOrderBook(
            new OrderBook("TSLA")
        )
    );
// 每筆訂單都會：記錄 → 風控 → 撮合
```

### 5. Facade（外觀）

**問題：** 系統內部有很多複雜的子系統，外部呼叫者不應該知道細節。

```java
// 沒有 Facade：客戶端需要知道所有細節
OrderValidator validator = new OrderValidator(config);
RiskEngine risk = new RiskEngine(clientLimits);
OrderBook book = OrderBook.getInstance("TSLA");
TradeReporter reporter = new TradeReporter(fixGateway);

validator.validate(order);
risk.check(order);
List<Trade> trades = book.addOrder(order);
reporter.report(trades);
```

```java
// Facade：隱藏複雜性
public class TradingFacade {
    private final OrderValidator validator;
    private final RiskEngine risk;
    private final Map<String, OrderBook> books;
    private final TradeReporter reporter;

    public TradingFacade() {
        this.validator = new OrderValidator(ExchangeConfig.INSTANCE);
        this.risk = new RiskEngine();
        this.books = new HashMap<>();
        this.reporter = new TradeReporter();
    }

    // 外部只需要呼叫這一個方法
    public List<Trade> submitOrder(Order order) {
        validator.validate(order);
        risk.check(order);
        OrderBook book = books.computeIfAbsent(
            order.getSymbol(), OrderBook::new
        );
        List<Trade> trades = book.addOrder(order);
        reporter.report(trades);
        return trades;
    }
}

// 使用：一行搞定
TradingFacade exchange = new TradingFacade();
exchange.submitOrder(order);
```

**Java 標準庫的 Facade：**
- `Collections.sort()` — 隱藏了 TimSort 的實作細節
- `Files.readAllLines()` — 隱藏了 Stream/Reader 的管理

## 三、行為型模式（Behavioral Patterns）

### 6. Strategy（策略）

**問題：** 演算法有多種版本，需要在執行時切換，不想用大量 if/else。

```java
// 不用 Strategy：if/else 散落各處
if (matchingStrategy.equals("PRICE_TIME")) {
    // 價格時間優先邏輯...
} else if (matchingStrategy.equals("PRO_RATA")) {
    // 比例分配邏輯...
}
// 每次新增策略都要改這個方法
```

```java
// Strategy 介面
public interface MatchingStrategy {
    List<Trade> match(Order incomingOrder, TreeMap<Double, PriceLevel> opposite);
}

// 策略一：價格時間優先（FIFO）
public class PriceTimePriorityStrategy implements MatchingStrategy {
    @Override
    public List<Trade> match(Order incomingOrder,
                              TreeMap<Double, PriceLevel> opposite) {
        // 按價格，同價格按時間順序撮合
        // （第十四章的撮合邏輯）
        List<Trade> trades = new ArrayList<>();
        // ... 撮合邏輯 ...
        return trades;
    }
}

// 策略二：比例分配（Pro-Rata，期貨市場常用）
public class ProRataStrategy implements MatchingStrategy {
    @Override
    public List<Trade> match(Order incomingOrder,
                              TreeMap<Double, PriceLevel> opposite) {
        // 同一價格的所有訂單按比例分配成交量
        List<Trade> trades = new ArrayList<>();
        // ... 比例計算邏輯 ...
        return trades;
    }
}

// Context：使用策略的 OrderBook
public class OrderBook {
    private MatchingStrategy strategy;

    public OrderBook(String symbol, MatchingStrategy strategy) {
        this.symbol = symbol;
        this.strategy = strategy;
    }

    // 執行時切換策略（例如開收盤用不同規則）
    public void setStrategy(MatchingStrategy strategy) {
        this.strategy = strategy;
    }

    public List<Trade> addOrder(Order order) {
        // ...
        return strategy.match(incomingOrder, opposite);
    }
}

// 使用
OrderBook equityBook  = new OrderBook("TSLA", new PriceTimePriorityStrategy());
OrderBook futuresBook = new OrderBook("ES",   new ProRataStrategy());
```

**你已經在用 Strategy 的地方：**
- `Comparator.reverseOrder()` 傳入 TreeMap — 排序策略
- `Executors.newFixedThreadPool()` vs `newCachedThreadPool()` — 執行緒池策略

### 7. Observer（觀察者）

**問題：** 某個事件發生後，需要通知多個不同的系統，但不想讓發布者知道訂閱者的細節。

```java
// Observer 介面（訂閱者）
public interface TradeListener {
    void onTrade(Trade trade);
}

// 具體訂閱者一：風控監控
public class RiskMonitor implements TradeListener {
    @Override
    public void onTrade(Trade trade) {
        // 累積當日成交量，超過限額發出警報
        System.out.println("[風控] 成交：" + trade);
    }
}

// 具體訂閱者二：FIX 回報
public class FixReporter implements TradeListener {
    @Override
    public void onTrade(Trade trade) {
        // 發送 FIX ExecutionReport 給對應客戶
        System.out.println("[FIX] 回報成交：" + trade);
    }
}

// 具體訂閱者三：行情廣播
public class MarketDataPublisher implements TradeListener {
    @Override
    public void onTrade(Trade trade) {
        // 透過 UDP Multicast 廣播最新成交
        System.out.println("[行情] 廣播：" + trade);
    }
}

// 發布者：OrderBook
public class OrderBook {
    private final List<TradeListener> listeners = new ArrayList<>();

    public void addListener(TradeListener listener) {
        listeners.add(listener);
    }

    private void notifyTrade(Trade trade) {
        for (TradeListener listener : listeners) {
            listener.onTrade(trade);
        }
    }

    public List<Trade> addOrder(Order order) {
        // ...撮合邏輯...
        for (Trade trade : trades) {
            notifyTrade(trade);  // 通知所有訂閱者
        }
        return trades;
    }
}

// 使用：動態增減訂閱者，OrderBook 不需要修改
OrderBook book = new OrderBook("TSLA");
book.addListener(new RiskMonitor());
book.addListener(new FixReporter());
book.addListener(new MarketDataPublisher());
```

**Java 標準庫的 Observer：**
- `EventListener` 介面系列（GUI 事件）
- `java.util.Observable`（舊，已棄用）
- **現代做法：** `java.util.concurrent.Flow`（Reactive Streams，JDK 9+）

### 8. Command（命令）

**問題：** 操作需要支援「撤銷」（Undo），或者需要把操作本身當成物件傳遞（佇列、排程）。

```java
// Command 介面
public interface OrderCommand {
    void execute(OrderBook book);
    void undo(OrderBook book);   // 撤銷！
}

// 具體命令一：新增訂單
public class AddOrderCommand implements OrderCommand {
    private final Order order;
    private List<Trade> executedTrades;

    public AddOrderCommand(Order order) {
        this.order = order;
    }

    @Override
    public void execute(OrderBook book) {
        executedTrades = book.addOrder(order);
    }

    @Override
    public void undo(OrderBook book) {
        // 若訂單未成交，可以取消
        if (order.getStatus() == OrderStatus.ACTIVE) {
            book.cancelOrder(order.getOrderId());
        }
        // 若已成交，撮合引擎通常不支援撤銷（需要另建反向訂單）
    }
}

// 具體命令二：取消訂單
public class CancelOrderCommand implements OrderCommand {
    private final String orderId;

    public CancelOrderCommand(String orderId) {
        this.orderId = orderId;
    }

    @Override
    public void execute(OrderBook book) {
        book.cancelOrder(orderId);
    }

    @Override
    public void undo(OrderBook book) {
        // 取消的撤銷比較複雜，實務上少見
    }
}

// Invoker：命令歷史（支援 Undo）
public class OrderCommandHistory {
    private final Deque<OrderCommand> history = new ArrayDeque<>();
    private final OrderBook book;

    public OrderCommandHistory(OrderBook book) {
        this.book = book;
    }

    public void execute(OrderCommand command) {
        command.execute(book);
        history.push(command);
    }

    public void undo() {
        if (!history.isEmpty()) {
            history.pop().undo(book);
        }
    }
}
```

**Command 模式在撮合引擎中的另一個用途：**
```java
// 把訂單命令序列化後存入佇列，支援重播（Replay）
// 交易所系統當機後，從命令日誌重建訂單簿狀態
Queue<OrderCommand> commandLog = new LinkedBlockingQueue<>();
// 系統重啟時：
commandLog.forEach(cmd -> cmd.execute(orderBook));
// 訂單簿狀態完整恢復
```

### 9. Template Method（模板方法）

**問題：** 演算法的骨架固定，但某些步驟的實作會因子類別而不同。

```java
// 抽象類別：定義骨架
public abstract class OrderProcessor {

    // 模板方法：骨架固定（final 防止子類別覆寫流程）
    public final List<Trade> process(Order order) {
        validate(order);        // 步驟 1：固定
        List<Trade> trades = doMatch(order);  // 步驟 2：子類別實作
        postProcess(trades);    // 步驟 3：固定
        return trades;
    }

    private void validate(Order order) {
        if (order.getQuantity() <= 0)
            throw new IllegalArgumentException("數量必須 > 0");
    }

    protected abstract List<Trade> doMatch(Order order);  // 子類別決定撮合方式

    private void postProcess(List<Trade> trades) {
        trades.forEach(t -> System.out.println("成交：" + t));
    }
}

// 子類別一：股票（價格時間優先）
public class EquityOrderProcessor extends OrderProcessor {
    private final OrderBook book;

    @Override
    protected List<Trade> doMatch(Order order) {
        return book.addOrder(order);
    }
}

// 子類別二：期貨（比例分配）
public class FuturesOrderProcessor extends OrderProcessor {
    private final OrderBook book;

    @Override
    protected List<Trade> doMatch(Order order) {
        // 期貨的特殊撮合邏輯
        return book.addOrder(order);
    }
}
```

**Java 標準庫的 Template Method：**
- `AbstractList`：已實作 `contains()`, `indexOf()` 等方法，只要你實作 `get()` 和 `size()`
- `HttpServlet`：已定義 HTTP 請求流程，你只需要覆寫 `doGet()`, `doPost()`

## 三(續)、補充兩個必學模式

### 10. Proxy（代理）— Spring AOP 的底層

代理模式在「真實物件」外面包一層，攔截所有呼叫，在前後加邏輯：

```java
// 介面
public interface OrderService {
    Order createOrder(OrderRequest req);
}

// 真實實作
public class OrderServiceImpl implements OrderService {
    public Order createOrder(OrderRequest req) {
        return orderRepository.save(Order.from(req));
    }
}

// JDK 動態代理（只能代理介面）
OrderService proxy = (OrderService) Proxy.newProxyInstance(
    OrderServiceImpl.class.getClassLoader(),
    new Class[]{OrderService.class},
    (proxyObj, method, args) -> {
        System.out.println("[LOG] 呼叫 " + method.getName());    // 前置邏輯
        long start = System.currentTimeMillis();
        Object result = method.invoke(new OrderServiceImpl(), args);  // 呼叫真實方法
        System.out.println("[LOG] 耗時 " + (System.currentTimeMillis() - start) + "ms");  // 後置邏輯
        return result;
    }
);
proxy.createOrder(req);  // 實際上先走代理，再走真實方法
```

**Spring AOP 怎麼運作的？**

```
你寫的 @Service OrderServiceImpl
    ↓ Spring 啟動時，偵測到有 @Transactional / @Cacheable
    ↓ 用 CGLIB 建立一個子類別（代理）
OrderServiceImplCGLIB$$xxx（代理物件，Spring 容器存的是這個）
    ↓ 你呼叫 orderService.createOrder(...)
    ↓ 代理攔截 → 開始交易 → 呼叫真實方法 → 提交交易
```

這就是為什麼 `@Transactional` 加在 private 方法上沒效果（代理無法覆寫 private），也是為什麼在同一個類別內部呼叫自己的方法不走 AOP（`this.method()` 跳過了代理）。

### 11. Chain of Responsibility（責任鏈）— Filter Chain 的底層

把多個處理者串成一條鏈，請求依序通過每個處理者，每個處理者決定是否繼續傳遞：

```java
// 抽象處理者
public abstract class RequestHandler {
    protected RequestHandler next;

    public RequestHandler setNext(RequestHandler next) {
        this.next = next;
        return next;  // 支援鏈式呼叫
    }

    public abstract void handle(HttpRequest request);
}

// 具體處理者 1：身份驗證
public class AuthHandler extends RequestHandler {
    public void handle(HttpRequest request) {
        if (!isAuthenticated(request)) {
            throw new UnauthorizedException();
        }
        System.out.println("[Auth] 驗證通過");
        if (next != null) next.handle(request);  // 交給下一個
    }
}

// 具體處理者 2：限流
public class RateLimitHandler extends RequestHandler {
    public void handle(HttpRequest request) {
        if (isRateLimited(request)) {
            throw new TooManyRequestsException();
        }
        System.out.println("[RateLimit] 未超限");
        if (next != null) next.handle(request);
    }
}

// 具體處理者 3：日誌
public class LogHandler extends RequestHandler {
    public void handle(HttpRequest request) {
        System.out.println("[Log] 記錄請求: " + request.getPath());
        if (next != null) next.handle(request);
    }
}

// 組裝鏈條（順序很重要！）
RequestHandler chain = new AuthHandler();
chain.setNext(new RateLimitHandler())
     .setNext(new LogHandler());

chain.handle(request);  // Auth → RateLimit → Log
```

**Spring Security 的 SecurityFilterChain 就是這個模式：**

```
請求 → UsernamePasswordAuthenticationFilter
     → JwtFilter（你加的）
     → ExceptionTranslationFilter
     → FilterSecurityInterceptor
     → Controller
```

每個 Filter 可以決定放行（`chain.doFilter()`）或攔截（直接回應）。

## 四、模式總覽

| 模式 | 解決什麼問題 | 你程式碼裡的例子 |
|------|------------|----------------|
| Singleton | 確保只有一個實例 | ExchangeConfig |
| Factory Method | 隱藏物件建立邏輯 | OrderFactory.create() |
| Builder | 複雜物件的清晰建構 | Order.Builder |
| Decorator | 動態增加功能，不改原始類別 | LoggingOrderBook、BufferedReader |
| Facade | 簡化複雜子系統的介面 | TradingFacade |
| Strategy | 可切換的演算法族 | MatchingStrategy、Comparator |
| Observer | 一對多的事件通知 | TradeListener |
| Command | 操作物件化，支援 Undo/Queue | OrderCommand |
| Template Method | 固定骨架，子類別填空 | OrderProcessor |

---

## 五、如何判斷要用哪個模式

```
「我需要確保只有一個實例」         → Singleton
「建立邏輯很複雜或有很多參數」      → Factory Method / Builder
「想動態增加功能，不改原始類別」     → Decorator
「想隱藏複雜的子系統」             → Facade
「有多種演算法要切換」              → Strategy
「事件發生後要通知多個地方」        → Observer
「操作需要支援 Undo 或排隊執行」    → Command
「流程固定，但某些步驟要讓子類別決定」 → Template Method
```

## 本章練習題

**Q1：Java I/O 中 `new BufferedReader(new FileReader("file.txt"))` 用了什麼模式？**
<details>
<summary>答案</summary>
Decorator 模式。BufferedReader「包裝」了 FileReader，在不修改 FileReader 的情況下增加了緩衝功能。兩者都實現了 Reader 介面，Decorator 持有被包裝物件的引用，並在呼叫前後增加額外行為。
</details>

**Q2：`Comparator.reverseOrder()` 在 TreeMap 中用了什麼模式？**
<details>
<summary>答案</summary>
Strategy 模式。Comparator 就是「排序策略」的介面，TreeMap 的 Context 在建構時注入策略（Comparator），之後用這個策略來決定鍵的順序。你可以在建構時傳入不同的 Comparator 來改變行為，完全符合 Strategy 的定義。
</details>

**Q3：請設計一個 `PriceAlertObserver`，當最新成交價超過 400.0 時印出警告，並接入第十四章的 OrderBook。**
<details>
<summary>答案</summary>

```java
public class PriceAlertObserver implements TradeListener {
    private final double threshold;

    public PriceAlertObserver(double threshold) {
        this.threshold = threshold;
    }

    @Override
    public void onTrade(Trade trade) {
        if (trade.getPrice() > threshold) {
            System.out.printf("[警告] 成交價 %.2f 超過警戒線 %.2f！%n",
                trade.getPrice(), threshold);
        }
    }
}

// 接入
OrderBook book = new OrderBook("TSLA");
book.addListener(new PriceAlertObserver(400.0));
```
</details>

**Q4：Builder 和 Factory Method 都能建立物件，什麼時候選哪個？**
<details>
<summary>答案</summary>
Factory Method：建立「同一介面的不同子類別」，重點在於「選擇建立哪個類別」，例如根據訂單類型字串決定建立 LimitOrder 還是 MarketOrder。Builder：建立「同一個複雜類別」，重點在於「如何設定這個物件的各種參數」，參數多、有預設值、組合靈活時使用。簡單說：Factory = 我幫你決定要什麼；Builder = 我幫你組裝你要的東西。
</details>
