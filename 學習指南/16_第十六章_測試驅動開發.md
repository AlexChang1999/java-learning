# 第十六章：測試驅動開發（TDD with JUnit 5 + Mockito）
> 工程師不靠「應該沒問題」，靠的是「測試說它沒問題」

---

## 前言：為什麼工程師必須會寫測試

當你的撮合引擎只有 100 行，你可以靠眼睛看。  
當它長到 10,000 行，且 FOK、IOC、市價單互相影響時，**你唯一能信任的只有測試**。

測試讓你能：
- **放心重構**：改完程式碼，跑測試，全綠 = 沒破壞現有功能
- **快速除錯**：失敗的測試直接告訴你哪裡壞了
- **說清楚規格**：「FOK 數量不足時，返回空列表」——測試比文件更精確

### 最小可跑的測試

```java
import org.junit.jupiter.api.*;
import static org.junit.jupiter.api.Assertions.*;

class OrderBookTest {

    private OrderBook book;

    @BeforeEach           // 每個測試前都執行（建立乾淨的初始狀態）
    void setUp() {
        book = new OrderBook("TSLA");
    }

    @AfterEach            // 每個測試後執行（可選，清理資源）
    void tearDown() {
        // book = null;  // GC 自動處理，通常不需要
    }

    @Test
    @DisplayName("兩個相互匹配的限價單應該產生一筆成交")
    void limitOrderShouldMatch() {
        // 安排（Arrange）
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));

        // 行動（Act）
        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100)
        );

        // 斷言（Assert）
        assertEquals(1, trades.size(), "應該產生 1 筆成交");
        assertEquals(350.0, trades.get(0).getPrice(), "成交價應為 350.0");
        assertEquals(100, trades.get(0).getQuantity(), "成交量應為 100");
    }
}
```

## 二、JUnit 5 核心注解

```java
class JUnit5AnnotationsDemo {

    @BeforeAll              // 整個測試類別只執行一次（必須 static）
    static void initAll() {
        System.out.println("所有測試開始前");
    }

    @BeforeEach             // 每個 @Test 之前
    void init() {
        System.out.println("單一測試開始前");
    }

    @Test
    void normalTest() { }

    @Test
    @Disabled("等待功能開發完成")  // 暫時停用，不計入失敗
    void notYetImplemented() { }

    @Test
    @Timeout(value = 100, unit = TimeUnit.MILLISECONDS)  // 超時就失敗
    void performanceTest() throws InterruptedException {
        // 如果超過 100ms 就失敗
        Thread.sleep(50);
    }

    @AfterEach              // 每個 @Test 之後
    void tearDown() { }

    @AfterAll               // 整個測試類別結束後（必須 static）
    static void tearDownAll() { }
}
```

## 三、Assertions：各種斷言

```java
class AssertionsDemo {

    @Test
    void demonstrateAssertions() {
        // 基本斷言
        assertEquals(42, 6 * 7);
        assertNotEquals(0, 42);
        assertTrue(350.0 > 0);
        assertFalse("".length() > 0);
        assertNull(null);
        assertNotNull(new Object());

        // 浮點數比較（永遠不要用 assertEquals 比較 double！）
        assertEquals(0.3, 0.1 + 0.2, 1e-9);  // 第三個參數是允許誤差

        // 集合斷言
        List<Trade> trades = List.of(new Trade("B1", "S1", 350.0, 100));
        assertFalse(trades.isEmpty());
        assertEquals(1, trades.size());

        // 例外斷言（確認某個操作一定拋出例外）
        IllegalArgumentException ex = assertThrows(
            IllegalArgumentException.class,
            () -> new Order("", "TSLA", "BUY", OrderType.LIMIT, 350.0, -1)
        );
        assertTrue(ex.getMessage().contains("數量必須 > 0"));

        // 多個斷言一起執行（即使前面失敗，後面也繼續執行）
        assertAll("訂單欄位",
            () -> assertEquals("TSLA", order.getSymbol()),
            () -> assertEquals("BUY", order.getSide()),
            () -> assertEquals(350.0, order.getPrice(), 1e-9)
        );
    }
}
```

## 四、ParameterizedTest：一個測試，多組資料

```java
import org.junit.jupiter.params.*;
import org.junit.jupiter.params.provider.*;

class ParameterizedDemo {

    // 測試各種有效訂單類型
    @ParameterizedTest
    @ValueSource(strings = { "LIMIT", "MARKET", "IOC", "FOK" })
    void shouldAcceptAllOrderTypes(String type) {
        assertDoesNotThrow(() ->
            OrderFactory.create("O-1", "TSLA", "BUY", type, 350.0, 100)
        );
    }

    // 測試邊界條件：各種數量
    @ParameterizedTest
    @ValueSource(ints = { 1, 100, 1000, 99999 })
    void shouldAcceptValidQuantities(int qty) {
        Order order = new Order("O-1", "TSLA", "BUY", OrderType.LIMIT, 350.0, qty);
        assertEquals(qty, order.getRemainingQty());
    }

    // 測試負數、零應該拋出例外
    @ParameterizedTest
    @ValueSource(ints = { 0, -1, -100 })
    void shouldRejectInvalidQuantities(int qty) {
        assertThrows(IllegalArgumentException.class,
            () -> new Order("O-1", "TSLA", "BUY", OrderType.LIMIT, 350.0, qty)
        );
    }

    // 用 @CsvSource 測試多個欄位組合
    @ParameterizedTest
    @CsvSource({
        "TSLA, BUY,  350.0, 100",
        "TSLA, SELL, 351.0, 200",
        "AAPL, BUY,  180.0, 50"
    })
    void shouldCreateOrderWithCorrectFields(
            String symbol, String side, double price, int qty) {
        Order order = new Order("O-1", symbol, side, OrderType.LIMIT, price, qty);
        assertEquals(symbol, order.getSymbol());
        assertEquals(side,   order.getSide());
        assertEquals(price,  order.getPrice(), 1e-9);
        assertEquals(qty,    order.getRemainingQty());
    }
}
```

## 五、完整的 OrderBook 測試套件

```java
class OrderBookMatchingTest {

    private OrderBook book;

    @BeforeEach
    void setUp() {
        book = new OrderBook("TSLA");
    }

    // ────────────────── 限價單撮合 ──────────────────

    @Test
    @DisplayName("限價買單 >= 限價賣單時應撮合")
    void limitOrdersMatchWhenPricesOverlap() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));
        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100)
        );
        assertEquals(1, trades.size());
        assertEquals(100, trades.get(0).getQuantity());
    }

    @Test
    @DisplayName("限價買單 < 限價賣單時不撮合，訂單留在簿中")
    void limitOrdersDoNotMatchWhenPricesDontOverlap() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 351.0, 100));
        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100)
        );
        assertTrue(trades.isEmpty());
        assertEquals(350.0, book.getBestBid(), 1e-9);
        assertEquals(351.0, book.getBestAsk(), 1e-9);
    }

    @Test
    @DisplayName("部分撮合後剩餘數量留在訂單簿")
    void partialFillLeavesRemainingInBook() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));
        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 60)
        );
        assertEquals(1, trades.size());
        assertEquals(60, trades.get(0).getQuantity());
        // 賣單剩 40 股，仍在訂單簿
        assertEquals(351.0, book.getBestAsk(), 1e-9);  // 沒有更低的賣單了
        // 實際上 350.0 的賣單應該還有 40 股
    }

    // ────────────────── 時間優先（FIFO）──────────────────

    @Test
    @DisplayName("同一價格的賣單按時間順序成交（FIFO）")
    void samepriceOrdersMatchInFIFOOrder() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 50));
        book.addOrder(new Order("S2", "TSLA", "SELL", OrderType.LIMIT, 350.0, 50));

        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 50)
        );

        assertEquals(1, trades.size());
        assertEquals("S1", trades.get(0).getSellOrderId());  // S1 先到，S1 先成交
    }

    // ────────────────── 市價單 ──────────────────

    @Test
    @DisplayName("市價買單應吃掉最低賣價的訂單")
    void marketOrderFillsAtBestAvailablePrice() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));
        book.addOrder(new Order("S2", "TSLA", "SELL", OrderType.LIMIT, 351.0, 100));

        List<Trade> trades = book.addOrder(
            new Order("B1", "TSLA", "BUY", OrderType.MARKET, 0, 150)
        );

        assertEquals(2, trades.size());
        assertEquals(350.0, trades.get(0).getPrice(), 1e-9);  // 先以 350 成交
        assertEquals(351.0, trades.get(1).getPrice(), 1e-9);  // 再以 351 成交
        assertEquals(100, trades.get(0).getQuantity());
        assertEquals(50,  trades.get(1).getQuantity());
    }

    // ────────────────── IOC ──────────────────

    @Test
    @DisplayName("IOC 部分成交後剩餘取消（不進訂單簿）")
    void iocOrderCancelsUnfilledRemainder() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 50));

        List<Trade> trades = book.addOrder(
            new Order("IOC1", "TSLA", "BUY", OrderType.IOC, 350.0, 100)
        );

        assertEquals(1, trades.size());
        assertEquals(50, trades.get(0).getQuantity());

        // IOC 的剩餘 50 股不應進入訂單簿
        assertEquals(0.0, book.getBestBid(), 1e-9);  // 買單簿應為空
    }

    // ────────────────── FOK ──────────────────

    @Test
    @DisplayName("FOK 數量不足時全部取消，不產生任何成交")
    void fokOrderCancelsIfCannotFullyFill() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 50));

        List<Trade> trades = book.addOrder(
            new Order("FOK1", "TSLA", "BUY", OrderType.FOK, 350.0, 100)
        );

        assertTrue(trades.isEmpty(), "FOK 無法全部成交時不應有任何成交");
        // 賣單應仍在訂單簿中（未被消耗）
        assertEquals(350.0, book.getBestAsk(), 1e-9);
    }

    @Test
    @DisplayName("FOK 數量足夠時全部成交")
    void fokOrderFillsCompletelyWhenSufficientQuantity() {
        book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));

        List<Trade> trades = book.addOrder(
            new Order("FOK1", "TSLA", "BUY", OrderType.FOK, 350.0, 100)
        );

        assertEquals(1, trades.size());
        assertEquals(100, trades.get(0).getQuantity());
    }

    // ────────────────── 取消訂單 ──────────────────

    @Test
    @DisplayName("取消存在的訂單應成功")
    void cancelExistingOrderShouldSucceed() {
        book.addOrder(new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 349.0, 100));
        boolean result = book.cancelOrder("B1");
        assertTrue(result);
        assertEquals(0.0, book.getBestBid(), 1e-9);  // 訂單簿應為空
    }

    @Test
    @DisplayName("取消不存在的訂單應返回 false")
    void cancelNonexistentOrderShouldFail() {
        boolean result = book.cancelOrder("NONEXISTENT");
        assertFalse(result);
    }
}
```

## 六、Mockito：模擬依賴

**什麼時候要 Mock？**  
當你要測試的類別依賴外部系統（資料庫、FIX 閘道、網路），而你不想在單元測試中真的連接這些系統。

```java
import org.mockito.*;
import org.mockito.junit.jupiter.*;
import static org.mockito.Mockito.*;

@ExtendWith(MockitoExtension.class)
class TradingFacadeTest {

    @Mock
    TradeReporter mockReporter;    // 建立假的回報器（不真的發 FIX 訊息）

    @Mock
    RiskEngine mockRisk;           // 建立假的風控引擎

    @InjectMocks
    TradingFacade facade;          // 被測試的類別（自動注入上面的 Mock）

    @Test
    void shouldCallReporterWhenTradeOccurs() {
        // 安排：設定風控引擎的行為（不拋出例外 = 通過）
        doNothing().when(mockRisk).check(any(Order.class));

        // 行動
        Order buy  = new Order("B1", "TSLA", "BUY",  OrderType.LIMIT, 350.0, 100);
        Order sell = new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100);
        facade.submitOrder(sell);
        facade.submitOrder(buy);

        // 斷言：回報器被呼叫了至少一次
        verify(mockReporter, atLeastOnce()).report(any(Trade.class));
    }

    @Test
    void shouldNotMatchWhenRiskCheckFails() {
        // 安排：讓風控引擎拋出例外
        doThrow(new RiskException("超過限額"))
            .when(mockRisk).check(any(Order.class));

        Order order = new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100);

        // 斷言：例外應該被傳播出來
        assertThrows(RiskException.class, () -> facade.submitOrder(order));

        // 斷言：回報器不應被呼叫（風控沒通過就不該成交）
        verify(mockReporter, never()).report(any());
    }

    @Test
    void shouldReturnTradeFromReporter() {
        // 安排：讓風控通過
        doNothing().when(mockRisk).check(any(Order.class));

        // 行動 + 斷言
        Order order = new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 99999.0, 999999);
        // 市價單吃掉所有賣單...
        // 確認 submitOrder 回傳的 Trade 和 reporter.report 接收的一致
    }
}
```

## 七、TDD 工作流程：Red → Green → Refactor

這是工程師最重要的習慣之一。

```
1. 🔴 Red：先寫一個失敗的測試
   → 確認你理解規格，測試也真的在測你以為它測的東西

2. 🟢 Green：寫最少的程式碼讓測試通過
   → 不要過度設計，先讓測試綠燈

3. 🔵 Refactor：重構程式碼
   → 測試保護你，大膽清理重複、命名、結構
   → 重構後重跑測試，確認還是綠燈

然後回到步驟 1，加下一個測試
```

**實際示範：TDD 實作「買單匹配」**

```java
// 第一輪 Red：最基本的匹配
@Test
void basicMatch() {
    book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));
    List<Trade> trades = book.addOrder(
        new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100)
    );
    assertFalse(trades.isEmpty());  // ← 這個測試先失敗
}
// → 去實作最基本的撮合邏輯讓它通過

// 第二輪 Red：加入邊界條件
@Test
void noMatchWhenBuyPriceTooLow() {
    book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 351.0, 100));
    List<Trade> trades = book.addOrder(
        new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100)
    );
    assertTrue(trades.isEmpty());  // ← 這個測試先失敗
}
// → 修正程式碼加入價格比較邏輯

// 第三輪 Red：部分成交
@Test
void partialMatch() {
    book.addOrder(new Order("S1", "TSLA", "SELL", OrderType.LIMIT, 350.0, 100));
    List<Trade> trades = book.addOrder(
        new Order("B1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 60)
    );
    assertEquals(60, trades.get(0).getQuantity());
}
// 每一輪只解決一個問題，永遠有測試守護你
```

## 八、測試覆蓋率與常見陷阱

```
好的測試覆蓋率 ≠ 100%

測試的目標是：覆蓋所有重要的「行為」，而不是覆蓋所有的「程式碼行」

重要行為（一定要測）：
  ✅ 正常路徑（Happy Path）
  ✅ 邊界值（0, 1, 最大值）
  ✅ 錯誤情況（例外、無效輸入）
  ✅ 狀態轉換（ACTIVE → FILLED → CANCELLED）

不值得測的：
  ❌ Getter / Setter（除非有邏輯）
  ❌ toString() / equals()（除非有特殊邏輯）
  ❌ 純 Facade（測了等於測了兩次子系統）
```

```
常見陷阱：

1. 測試依賴執行順序
   → 每個測試必須獨立，@BeforeEach 確保乾淨狀態

2. 一個測試測太多東西
   → 一個 @Test 只驗證一個行為，名字說清楚它在測什麼

3. Mock 過度
   → Mock 太多，測試和實際行為脫節。
   → 撮合引擎的核心邏輯（OrderBook）不應該 Mock，直接測

4. 測試名稱不清楚
   → 壞：test1(), testMatch()
   → 好：limitOrdersMatchWhenPricesOverlap()
```

## 本章練習題

**Q1：以下測試有什麼問題？**
```java
@Test
void testOrder() {
    Order o = new Order("O1", "TSLA", "BUY", OrderType.LIMIT, 350.0, 100);
    assertEquals("TSLA", o.getSymbol());
    assertEquals("BUY", o.getSide());
    assertEquals(350.0, o.getPrice());
    assertEquals(100, o.getRemainingQty());
    assertEquals(OrderStatus.PENDING, o.getStatus());
    book.addOrder(o);
    assertEquals(OrderStatus.ACTIVE, o.getStatus());
    book.cancelOrder("O1");
    assertEquals(OrderStatus.CANCELLED, o.getStatus());
}
```
<details>
<summary>答案</summary>
問題一：測試名稱 testOrder 完全沒說明在測什麼。問題二：一個測試同時測了多個行為（欄位初始值、狀態轉換、取消），應該拆成三個獨立測試。問題三：浮點數 350.0 的比較沒有給誤差值（雖然這個例子不會出問題，但養成習慣應寫 assertEquals(350.0, o.getPrice(), 1e-9)）。應拆成：shouldInitializeFieldsCorrectly、shouldBecomeActiveWhenAddedToBook、shouldBeCancelledAfterCancelOrder。
</details>

**Q2：為什麼不應該用 `System.out.println` 來驗證測試結果？**
<details>
<summary>答案</summary>
因為 println 需要人眼看輸出才能判斷對錯，而測試框架的 assertEquals 等斷言是機器自動判斷——測試通過就是綠燈，不需要人工閱讀。自動化測試的價值在於 CI/CD 流程中自動驗證，println 無法做到這點。此外，當有 100 個測試同時執行時，印出的內容難以追溯是哪個測試的結果。
</details>

**Q3：什麼是 Mock？什麼情況下應該使用 Mock，什麼情況下不應該？**
<details>
<summary>答案</summary>
Mock 是模擬物件，讓你可以控制它的行為（例如讓它回傳特定值或拋出特定例外），目的是隔離被測試的類別。應該使用 Mock 的情況：依賴外部系統（資料庫、網路、FIX 閘道）、依賴難以建立的物件（需要複雜設定）、需要測試特定的錯誤情境（例如資料庫斷線）。不應該使用 Mock 的情況：被測試的核心邏輯本身（Mock 掉 OrderBook 後就不是在測撮合引擎了）、簡單的 POJO 或 Value Object。過度 Mock 會讓測試和真實行為脫節，反而測不到真正的問題。
</details>
