## 前言：println 的三個致命問題

```java
// 你現在的程式碼
System.out.println("收到訂單：" + order.getOrderId());
System.out.println("成交：" + trade);

// 問題一：無法關閉
// 上線後想關掉這些輸出？只能逐行刪除，然後除錯時再加回來

// 問題二：沒有時間戳、沒有等級、沒有執行緒資訊
// 出了問題，你根本不知道這條訊息是幾點幾分哪個執行緒輸出的

// 問題三：無法寫到檔案、無法根據等級過濾
// 線上系統出問題，你要 SSH 進去看 console？不存在的
```

## 一、SLF4J + Logback：業界標準組合

```
SLF4J（Simple Logging Facade for Java）：
  → 日誌的「介面」，你的程式碼只和這個介面互動
  → 可以在不改程式碼的情況下，替換底層日誌實作

Logback：
  → 日誌的「實作」，真正把訊息寫到哪裡、怎麼格式化
  → SLF4J 的預設最佳夥伴（同一個作者）

其他選擇：
  Log4j2、java.util.logging（不推薦新專案使用）
```

### 加入依賴

```xml
<!-- pom.xml -->
<dependency>
    <groupId>ch.qos.logback</groupId>
    <artifactId>logback-classic</artifactId>
    <version>1.5.6</version>
</dependency>
<!-- logback-classic 已包含 slf4j-api，不需要額外加 -->
```

## 二、基本用法

```java
import org.slf4j.Logger;
import org.slf4j.LoggerFactory;

public class OrderBook {

    // 每個類別建立自己的 Logger（static final，不需要每個實例各自建立）
    private static final Logger log = LoggerFactory.getLogger(OrderBook.class);

    public List<Trade> addOrder(Order order) {
        // 五個日誌等級（由低到高）
        log.trace("進入 addOrder，訂單 ID：{}", order.getOrderId());   // 最細，通常關閉
        log.debug("訂單詳情：{} {} {} @ {}", order.getSide(),
                  order.getQuantity(), order.getSymbol(), order.getPrice());
        log.info("收到新訂單：{}", order.getOrderId());               // 正常業務流程
        log.warn("訂單數量接近上限：{}", order.getQuantity());         // 警告，但不是錯誤
        log.error("撮合發生例外", e);                                   // 錯誤，需要關注

        // 注意：用 {} 佔位符，不要用字串拼接！
        // 好：log.info("訂單：{}", orderId);
        // 壞：log.info("訂單：" + orderId);  ← 即使日誌等級關閉，字串拼接仍然執行（浪費 CPU）
    }
}
```

**輸出範例（預設格式）：**
```
2024-05-12 10:30:00.123 [main] INFO  tw.brad.tutor.OrderBook - 收到新訂單：O-001
2024-05-12 10:30:00.124 [main] WARN  tw.brad.tutor.OrderBook - 訂單數量接近上限：95000
2024-05-12 10:30:00.125 [pool-1-thread-3] ERROR tw.brad.tutor.OrderBook - 撮合發生例外
java.lang.NullPointerException: Cannot invoke "Order.getPrice()" because "order" is null
    at tw.brad.tutor.OrderBook.matchOrder(OrderBook.java:87)
    ...
```

## 三、Logback 設定檔（logback.xml）

放在 `src/main/resources/logback.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>

    <!-- Console Appender：輸出到終端機 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <!-- 格式：時間 [執行緒] 等級 類別 - 訊息 -->
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- File Appender：輸出到檔案（每天自動切割） -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/exchange.log</file>
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/exchange.%d{yyyy-MM-dd}.log</fileNamePattern>
            <maxHistory>30</maxHistory>     <!-- 保留最近 30 天 -->
            <totalSizeCap>1GB</totalSizeCap> <!-- 總大小上限 1GB，超過自動刪舊的 -->
        </rollingPolicy>
        <encoder>
            <pattern>%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n</pattern>
        </encoder>
    </appender>

    <!-- 交易核心：DEBUG 等級（詳細記錄） -->
    <logger name="tw.brad.tutor" level="DEBUG" additivity="false">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </logger>

    <!-- HikariCP 連線池：只顯示 WARN 以上（避免太多雜訊） -->
    <logger name="com.zaxxer.hikari" level="WARN"/>

    <!-- 預設：INFO 等級 -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
    </root>

</configuration>
```

## 四、MDC：追蹤一筆請求的完整流程

**問題：** 系統有 100 個並發請求，日誌全混在一起，怎麼追蹤某一筆訂單的完整流程？

```java
import org.slf4j.MDC;

// MDC（Mapped Diagnostic Context）：執行緒本地的 key-value 儲存
// 放進去後，同一執行緒的所有日誌都自動附上這個資訊

public class OrderProcessor {
    private static final Logger log = LoggerFactory.getLogger(OrderProcessor.class);

    public void processOrder(Order order) {
        // 在處理開始時放入追蹤 ID
        MDC.put("orderId", order.getOrderId());
        MDC.put("symbol",  order.getSymbol());

        try {
            log.info("開始處理訂單");              // 自動帶上 orderId 和 symbol
            validate(order);
            log.debug("驗證通過");
            List<Trade> trades = matchingEngine.match(order);
            log.info("撮合完成，成交 {} 筆", trades.size());

        } catch (Exception e) {
            log.error("訂單處理失敗", e);          // 發生錯誤時，立刻能找到是哪筆訂單
        } finally {
            MDC.clear();  // 執行緒還回池子前清除，避免污染下一個請求
        }
    }
}
```

在 `logback.xml` 中加入 MDC 欄位：

```xml
<pattern>%d{HH:mm:ss.SSS} [%thread] %-5level [orderId=%X{orderId} symbol=%X{symbol}] %logger{20} - %msg%n</pattern>
```

**輸出：**
```
10:30:00.123 [pool-1-thread-1] INFO  [orderId=O-001 symbol=TSLA] OrderProcessor - 開始處理訂單
10:30:00.124 [pool-1-thread-2] INFO  [orderId=O-002 symbol=AAPL] OrderProcessor - 開始處理訂單
10:30:00.125 [pool-1-thread-1] INFO  [orderId=O-001 symbol=TSLA] OrderProcessor - 撮合完成，成交 1 筆
```

即使多執行緒交錯輸出，過濾 `orderId=O-001` 就能完整追蹤這筆訂單的生命週期。

## 五、日誌等級策略

```
TRACE：最細粒度，通常只在本地除錯時開啟
        例：每個 bytecode 指令、每個 cache line 存取

DEBUG：開發和測試環境開啟，生產環境通常關閉
        例：訂單的詳細欄位、撮合的中間步驟

INFO： 生產環境的正常流程記錄（預設等級）
        例：收到訂單、成交完成、服務啟動/停止

WARN： 不影響功能但需要關注的情況
        例：訂單數量接近限額、連線池快滿了、重試第 2 次

ERROR：需要立即處理的問題（通常觸發告警通知）
        例：資料庫連線失敗、撮合邏輯拋出例外、資料不一致
```

## 六、常見反模式（別這樣做）

```java
// ❌ 用字串拼接
log.debug("訂單 ID：" + order.getOrderId() + " 價格：" + order.getPrice());
// 即使 DEBUG 等級被關閉，字串拼接仍然執行！

// ✅ 用 {} 佔位符（惰性求值，等級關閉時完全不執行）
log.debug("訂單 ID：{} 價格：{}", order.getOrderId(), order.getPrice());

// ❌ 吃掉例外（最常見的反模式）
try {
    matchOrder(order);
} catch (Exception e) {
    log.error("出錯了");  // 沒有 e！日誌裡看不到任何 stack trace
}

// ✅ 一定要把例外傳給 log.error
try {
    matchOrder(order);
} catch (Exception e) {
    log.error("撮合訂單 {} 時發生例外", order.getOrderId(), e);
}

// ❌ 重複記錄（每層都 log，造成同一個錯誤出現三次）
// 原則：只在「最頂層」（進入點）記錄，底層讓例外向上傳播
```

## 七、分散式追蹤：從一行 Log 到跨服務全鏈路 <!-- 🔴 資深 -->

> 在微服務架構中，一筆 HTTP 請求可能跨越 API Gateway → OrderService → InventoryService → PaymentService → Kafka → EmailService，光靠單機日誌找不到問題。分散式追蹤讓你把「所有服務的局部視角」拼成「完整的請求旅程地圖」。

### 核心概念：Trace、Span、Context Propagation

```
Trace（追蹤）：一整條請求鏈路的唯一識別
  └── Span（跨度）：鏈路中的一個工作單元（一次 RPC call、一次 DB query）
        ├── traceId：整條鏈路共享的 ID
        ├── spanId：本段工作的 ID
        ├── parentSpanId：上游 Span 的 ID
        ├── startTime / duration：計時
        └── tags / logs：附加資訊（HTTP status、SQL 語句等）

視覺化（Gantt 圖）：
  [──────────────────────── Trace ────────────────────────]
  [── API Gateway (root span) ──────────────────────────]
       [── OrderService ──]  [─── PaymentService ──────]
            [─ DB query ─]        [─ Kafka produce ─]
```

### Spring Boot 整合（Micrometer Tracing + Zipkin）

```xml
<!-- pom.xml -->
<dependency>
    <groupId>io.micrometer</groupId>
    <artifactId>micrometer-tracing-bridge-brave</artifactId>  <!-- Brave 實作 -->
</dependency>
<dependency>
    <groupId>io.zipkin.reporter2</groupId>
    <artifactId>zipkin-reporter-brave</artifactId>            <!-- 傳送到 Zipkin -->
</dependency>
```

```yaml
# application.yml
management:
  tracing:
    sampling:
      probability: 1.0   # 開發環境 100% 採樣；生產建議 0.1（10%）
spring:
  zipkin:
    base-url: http://localhost:9411
```

**自動注入**：一旦加入依賴，Spring 會自動為所有 HTTP 請求、`@Async` 方法、Kafka 消費者注入 `traceId` 和 `spanId`。

### traceId 出現在 Log 中

```java
// Micrometer Tracing 自動將 traceId/spanId 放入 MDC
// logback.xml 加入 %X{traceId} 即可印出
<pattern>%d{HH:mm:ss} [%thread] [%X{traceId}] %-5level %logger{36} - %msg%n</pattern>
```

```
13:42:01 [http-nio-8080] [3f2a1b4c8d9e0f12] INFO  OrderController - 收到訂單請求 #1001
13:42:01 [http-nio-8080] [3f2a1b4c8d9e0f12] INFO  OrderService   - 建立訂單，扣減庫存
13:42:01 [http-nio-8080] [3f2a1b4c8d9e0f12] INFO  PaymentService - 發起付款請求
```

同一個 traceId 串起所有服務的 Log，即使分散在不同機器的 Log 檔，用 ELK 的 `traceId: 3f2a1b4c8d9e0f12` 一搜即得。

### 手動建立 Span

```java
import io.micrometer.tracing.Tracer;
import io.micrometer.tracing.Span;

@Service
public class InventoryService {

    private final Tracer tracer;

    // Spring 自動注入
    public InventoryService(Tracer tracer) {
        this.tracer = tracer;
    }

    public void deductStock(Long productId, int quantity) {
        // 為這段業務邏輯建立獨立的 Span
        Span span = tracer.nextSpan().name("inventory.deduct").start();
        span.tag("product.id", String.valueOf(productId));
        span.tag("quantity", String.valueOf(quantity));

        try (Tracer.SpanInScope ws = tracer.withSpan(span)) {
            // 執行業務邏輯
            doDeduct(productId, quantity);
        } catch (Exception e) {
            span.error(e);   // 標記此 Span 有錯誤
            throw e;
        } finally {
            span.end();      // 一定要 end，否則 Span 不會送出
        }
    }
}
```

### 採樣策略：不要記錄每一筆

生產環境 100% 採樣會產生大量資料（追蹤本身也有開銷）：

| 策略 | 設定 | 適合場景 |
|------|------|----------|
| 100% 採樣 | `probability: 1.0` | 開發 / 除錯 |
| 10% 採樣 | `probability: 0.1` | 一般生產環境 |
| 基於規則 | 自定義 `Sampler` | 錯誤 100%、正常 1% |
| 基於 Header | `X-B3-Sampled: 1` | 強制追蹤特定請求 |

## 本章練習題

**Q1：為什麼 `log.debug("訂單：" + orderId)` 比 `log.debug("訂單：{}", orderId)` 效能差？**
<details>
<summary>答案</summary>
字串拼接（"訂單：" + orderId）在 Java 中會立即執行，無論 DEBUG 等級是否開啟。若 DEBUG 等級被關閉（生產環境常見），這個字串建立的 CPU 和記憶體都被白白浪費了。用 {} 佔位符的版本，SLF4J 只有在確認該等級開啟時才會執行字串格式化（惰性求值）。在高吞吐量系統（如撮合引擎）中，關閉 DEBUG 日誌但保留拼接語法，可能造成顯著的效能損失。
</details>

**Q2：MDC 是什麼？在多執行緒系統中它解決了什麼問題？**
<details>
<summary>答案</summary>
MDC（Mapped Diagnostic Context）是執行緒本地（ThreadLocal）的 key-value 儲存，同一執行緒的所有日誌輸出都會自動附上 MDC 中的值。在多執行緒系統（如同時處理 100 個請求的 Web 伺服器）中，日誌交錯輸出無法區分哪條日誌屬於哪個請求。透過在請求入口設置 MDC.put("requestId", uuid)，可以在日誌中過濾出特定請求的完整流程，大幅提高問題排查效率。注意 finally 中要呼叫 MDC.clear()，避免污染執行緒池中的下一個請求。
</details>

**Q3：如何設定讓撮合引擎核心程式碼輸出 DEBUG，但第三方函式庫只輸出 WARN？**
<details>
<summary>答案</summary>
在 logback.xml 中為不同 package 設定不同等級：
&lt;logger name="tw.brad.tutor" level="DEBUG"/&gt;（你的程式碼，詳細）
&lt;logger name="com.zaxxer.hikari" level="WARN"/&gt;（HikariCP，只看警告）
&lt;logger name="org.springframework" level="WARN"/&gt;（Spring，只看警告）
&lt;root level="INFO"&gt;（其他所有 package，INFO 以上）
這樣可以精確控制雜訊，在需要詳細診斷的模組開啟 DEBUG，避免被框架日誌淹沒。
</details>
