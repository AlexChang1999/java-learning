## 前言：從電到程式

## 一、電腦的組成：五大部件

```
┌──────────────────────────────────────────────────────────┐
│                        電腦                               │
│                                                          │
│  ┌──────────────────────────┐   ┌────────────────────┐  │
│  │          CPU             │   │      記憶體（RAM）   │  │
│  │  ┌──────┐  ┌──────────┐  │   │  存放執行中的資料   │  │
│  │  │控制單元│  │ ALU 算術 │  │   │  和程式碼          │  │
│  │  │（CU） │  │ 邏輯單元 │  │   └────────────────────┘  │
│  │  └──────┘  └──────────┘  │                            │
│  │  ┌──────────────────────┐│   ┌────────────────────┐  │
│  │  │    暫存器（Registers）││   │   儲存裝置（SSD/HDD）│  │
│  │  │  PC, SP, AX, BX...  ││   │  長期保存資料        │  │
│  │  └──────────────────────┘│   └────────────────────┘  │
│  └──────────────────────────┘                            │
│                                                          │
│  ┌──────────────────────────────────────────────────┐   │
│  │              輸入/輸出裝置（I/O）                  │   │
│  │         鍵盤、滑鼠、螢幕、網路卡                   │   │
│  └──────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────┘
```

### 控制單元（CU）：指揮官

負責「告訴所有部件該做什麼」：
- 從記憶體取下一條指令
- 解讀指令的意義（這是加法？還是跳轉？）
- 協調 ALU 和暫存器的動作

### ALU（Arithmetic Logic Unit）：計算工廠

做所有的計算：
- 整數加減乘除
- 位元運算（AND、OR、XOR、左移、右移）
- 大小比較（a > b？）

### 暫存器（Registers）：CPU 的口袋

**速度最快、容量最小**的記憶體，就在 CPU 晶片內：

```
常見暫存器（x86-64 架構）：
  rax, rbx, rcx, rdx  — 通用暫存器（存放計算中的數值）
  rsp                  — Stack Pointer（指向當前 Stack 頂部）
  rbp                  — Base Pointer（指向當前 Stack Frame 底部）
  rip                  — Instruction Pointer（下一條要執行的指令在哪）
  rflags               — 旗標暫存器（存放比較結果：零、負數、進位...）

64 位元 CPU：每個暫存器可存放 64 bits = 8 bytes
現代 CPU 通常有 16~32 個通用暫存器
```

## 二、一條指令的一生（Fetch-Decode-Execute Cycle）

這是 CPU 做的事，無止無休地重複：

```
┌─────────────────────────────────────────────────────────┐
│                  指令週期（Instruction Cycle）            │
│                                                         │
│  1. FETCH（取指）                                        │
│     rip 暫存器存著下一條指令的記憶體地址                   │
│     CPU 去那個地址取回指令的位元組                         │
│     rip 自動加 1（指向下一條指令）                         │
│                          ↓                              │
│  2. DECODE（解碼）                                       │
│     控制單元解讀位元組的意義                              │
│     「0x83 0xC0 0x01」→ 把 rax 暫存器加 1               │
│                          ↓                              │
│  3. EXECUTE（執行）                                      │
│     ALU 做實際計算                                       │
│     把結果寫回暫存器或記憶體                               │
│                          ↓                              │
│     回到步驟 1，重複……                                   │
└─────────────────────────────────────────────────────────┘
```

**一個簡單的 Java 加法被編譯成什麼：**

```java
int a = 5;
int b = 3;
int c = a + b;
```

```asm
; 對應的 x86-64 組語（概念示意）
mov DWORD PTR [rbp-4],  5    ; 把 5 存到 Stack（a 的位置）
mov DWORD PTR [rbp-8],  3    ; 把 3 存到 Stack（b 的位置）
mov eax, DWORD PTR [rbp-4]   ; 把 a 載入暫存器 eax
add eax, DWORD PTR [rbp-8]   ; eax = eax + b（實際加法）
mov DWORD PTR [rbp-12], eax  ; 把結果存到 Stack（c 的位置）
```

CPU 看的是最底層的這些指令，不是你寫的 Java。

## 三、記憶體層次（Memory Hierarchy）：速度 vs 容量的取捨

**這是理解所有效能問題的根本圖表。請把這個數字記住。**

```
速度          容量        延遲（存取時間）    頻寬
──────────────────────────────────────────────────────────
暫存器         < 1 KB      ~ 0.3 ns（1 cycle）  超高
L1 快取         32~64 KB   ~ 1 ns（4 cycles）   ~1 TB/s
L2 快取        256 KB~1MB  ~ 4 ns（12 cycles）  ~400 GB/s
L3 快取         8~64 MB    ~ 15 ns（40 cycles）  ~200 GB/s
RAM（主記憶體） 8~64 GB    ~ 100 ns             ~50 GB/s
NVMe SSD       1~4 TB      ~ 100 μs（10萬 ns）   ~7 GB/s
SATA SSD       1~4 TB      ~ 500 μs             ~550 MB/s
HDD 硬碟        2~18 TB    ~ 10 ms（1000萬 ns）  ~150 MB/s
網路（同機房）   —          ~ 100 μs             ~10 GB/s
──────────────────────────────────────────────────────────
```

**關鍵比例（以 L1 命中 = 1 秒為基準，換算成人類時間）：**

```
L1 快取命中    =  1 秒
L2 快取命中    =  4 秒
L3 快取命中    =  40 秒
RAM 存取       =  6 分鐘
SSD 存取       =  5 天
HDD 存取       =  3 個月
網路（跨洋）   =  4 年
```

→ **這就是為什麼「把資料放在快取裡」如此重要。**

## 四、CPU Cache 的工作原理

CPU 不會「一個位元組一個位元組」地讀 RAM，而是一次抓一整個 **Cache Line（快取行，通常 64 bytes）**。

```
RAM 中的陣列：
地址  0    8   16   24   32   40   48   56   64   ...
     [a0] [a1] [a2] [a3] [a4] [a5] [a6] [a7] [a8] ...

第一次存取 a[0]：Cache miss（快取沒有）
  → CPU 去 RAM 取整個 Cache Line（a[0]~a[7] 全部進快取）
  → 延遲：~100 ns

存取 a[1]：Cache hit（快取有！）
  → 延遲：~1 ns（快 100 倍）

存取 a[2], a[3], ... a[7]：全部 Cache hit
存取 a[8]：Cache miss，再去 RAM 取下一個 Cache Line
```

### 空間局部性（Spatial Locality）

```java
// 快：循序存取陣列（每個 Cache Line 都被充分利用）
int[] arr = new int[10_000_000];
long sum = 0;
for (int i = 0; i < arr.length; i++) {
    sum += arr[i];  // i=0 時 Cache miss，i=1~15 全是 Cache hit
}

// 慢：隨機跳躍存取（每次幾乎都是 Cache miss）
for (int i = 0; i < arr.length; i++) {
    sum += arr[random.nextInt(arr.length)];  // 幾乎每次都要去 RAM
}

// 實測差距：循序版通常比隨機版快 5~20 倍！
// 原因不是演算法不同，而是 Cache 命中率的差異
```

### 時間局部性（Temporal Locality）

```java
// 快：同一資料短時間內多次使用（留在快取裡）
int x = heavyComputation();
use(x);   // x 在暫存器/L1 裡
use(x);   // 直接用
use(x);   // 直接用

// 慢：每次都讀不同資料（快取一直被刷新）
```

## 五、Cache Miss 的三種類型

```
1. Cold Miss（冷啟動 Miss）
   第一次存取某塊記憶體，快取當然沒有
   → 不可避免，但可以用 prefetch 提前載入

2. Capacity Miss（容量 Miss）
   工作集太大，超過 Cache 容量，資料被踢出後又要用
   → 解法：減少工作集大小（第十五章 Flyweight 模式）

3. Conflict Miss（衝突 Miss）
   Cache 是用「組相聯（Set Associative）」方式組織的
   兩個記憶體地址映射到同一個 Cache 組，互相擠掉對方
   → 解法：調整資料結構對齊（進階話題）
```

## 六、從「大 O」到「實際速度」：Cache 改寫了排名

**教科書告訴你的：**

| 演算法 | Big O |
|--------|-------|
| 二元搜尋 | O(log n) |
| 線性搜尋 | O(n) |

**現實告訴你的（n = 1000, 已排序陣列 vs 樹）：**

```
線性搜尋陣列：
  存取模式：a[0], a[1], a[2], ... （循序）
  Cache 效率：極高（Cache Line 充分利用）
  實測：比理論快很多

二元搜尋樹（BST）：
  存取模式：根 → 左子樹 → 右右子樹 → ...（跳躍）
  每個節點是獨立的 Heap 物件，分散在記憶體各處
  Cache 效率：極低（幾乎每次都 Cache miss）
  實測：比線性搜尋陣列慢！（當 n 不夠大時）

結論：O(log n) 的樹，在 n < 幾千 時，輸給 O(n) 的陣列掃描
      原因：Cache miss 的代價（~100 ns）遠大於多幾次比較的代價（~1 ns）
```

## 七、數字系統基礎（補充第五章）

CPU 只認識 0 和 1，所有資料都是二進位的：

```
位元（bit）：0 或 1（電壓高或低）
位元組（byte）：8 個 bits

二進位 → 十進位換算：
  1010₂ = 1×2³ + 0×2² + 1×2¹ + 0×2⁰ = 8 + 0 + 2 + 0 = 10₁₀

int 在 CPU 中的儲存（32 bits）：
  42   = 0000 0000 0000 0000 0000 0000 0010 1010
  -1   = 1111 1111 1111 1111 1111 1111 1111 1111（二補數）
  最大  = 0111 1111 1111 1111 1111 1111 1111 1111 = 2,147,483,647

為什麼 int + 1 可能變負數（第五章的整數溢位）：
  0111...1111 (MAX_VALUE)
+ 0000...0001
= 1000...0000 (MIN_VALUE, -2,147,483,648)  ← 進位把符號位翻轉了！
```

## 八、現代 CPU 的三大效能技術

### 超純量（Superscalar）：一個時鐘週期執行多條指令

```
傳統 CPU：一個時鐘週期，1 條指令
現代 CPU：一個時鐘週期，4~6 條指令（同時！）

前提：指令之間沒有依賴關係
  指令 A：rax = rbx + 1
  指令 B：rcx = rdx + 2   ← 和 A 無關，可以同時執行
  指令 C：rax = rax + rcx ← 依賴 A 和 B 的結果，必須等待
```

### 亂序執行（Out-of-Order Execution）

```java
// 你寫的程式碼順序
int a = arr[i];        // (1) 可能 Cache miss，等待 ~100 ns
int b = a + 1;         // (2) 必須等 (1) 完成
int c = arr2[j];       // (3) 和 a/b 無關！

// CPU 實際執行順序（亂序）：
// 同時發出 (1) 和 (3) 的記憶體請求
// 等待 (1) 返回時，(3) 的結果可能已經就緒
// 大幅隱藏記憶體延遲
```

### 推測執行（Speculative Execution）

```
這就是「分支預測」（第十二章）的底層機制：
CPU 不等 if 條件的結果出來，直接「猜」並開始執行一條路徑
如果猜對了：零額外代價
如果猜錯了：丟棄結果，重新執行正確路徑（代價：10~20 cycles）

2018 年的 Spectre/Meltdown 漏洞：
  攻擊者利用推測執行，讓 CPU 推測性地存取不該讀的記憶體，
  再透過 Cache timing 側通道洩漏資料。
  → 這是整個 CPU 架構設計的缺陷，需要韌體和 OS 更新修補。
```

## 九、對 Java 程式碼的實際影響

```java
// 場景：遍歷 100 萬筆訂單

// ❌ 慢：物件分散在 Heap 各處（pointer chasing）
List<Order> orders = new ArrayList<>();  // Order 是 Heap 物件
for (Order o : orders) {
    sum += o.getPrice();  // 每個 Order 物件可能在不同 Cache Line
    // 可能每次都 Cache miss！
}

// ✅ 快：把資料打包成連續陣列（AoS vs SoA）
// 如果只需要 price 欄位，提取成 double[]
double[] prices = orders.stream()
    .mapToDouble(Order::getPrice)
    .toArray();
for (double price : prices) {
    sum += price;  // 完美的循序存取，Cache 命中率近 100%
}

// 這個技巧在 HFT 系統（第十四章）中極為重要
// 也是為什麼「原始型別陣列代替物件集合」出現在優化清單裡
```

## 本章練習題

**Q1：為什麼說「選對資料結構比選對演算法更重要」？**
<details>
<summary>答案</summary>
因為現代 CPU 的效能瓶頸往往在記憶體存取，而非 ALU 計算。一個演算法複雜度更高但 Cache 友好的方案（如循序掃描陣列），可能比複雜度更低但 Cache 不友好的方案（如遍歷鏈結串列或樹）快得多。L1 Cache 命中需要 ~1 ns，RAM 存取需要 ~100 ns，差距 100 倍。數據結構的記憶體佈局（連續 vs 分散）直接決定 Cache 命中率，進而影響實際效能遠超理論複雜度。
</details>

**Q2：如果你的程式碼在測試時很快，但換到另一台機器卻慢很多，可能的 Cache 相關原因是什麼？**
<details>
<summary>答案</summary>
(1) L3 Cache 大小不同：你的測試機有 32 MB L3，新機只有 8 MB，工作集放不進去，產生大量 Capacity Miss；(2) NUMA 架構差異：多 Socket 伺服器的跨 Node 記憶體存取延遲是本地的 2 倍，如果執行緒被調度到不同 Socket 就會慢；(3) Cache Line 大小差異：舊架構 32 bytes，新架構 64 bytes，某些 Padding 策略失效；(4) 超執行緒（Hyper-threading）共享 L1/L2，競爭加劇。
</details>

**Q3：「亂序執行」和「程式碼執行順序」的關係是什麼？這對多執行緒有什麼影響？**
<details>
<summary>答案</summary>
CPU 會重新排列指令執行順序（只要結果等效），這在單執行緒中完全安全。但在多執行緒中，如果沒有記憶體屏障（Memory Barrier），一個執行緒的「寫入順序」對另一個執行緒來說可能是亂序的。這就是第十章 volatile 和 happens-before 規則存在的原因——volatile 寫入/讀取會插入 Store/Load Barrier，強制 CPU 不能重排這些操作前後的指令，確保多執行緒間的可見性和順序性。
</details>
