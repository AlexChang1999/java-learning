# 第二十四章：作業系統基礎
> Java 的 Thread、GC、I/O 都建立在 OS 上——不懂 OS，就不懂 Java 為什麼這樣設計

---

## 前言：OS 是什麼？

作業系統（Operating System）是你的程式和硬體之間的**翻譯官 + 資源管理員**：

```
你的 Java 程式
      ↓  呼叫 Java API
   JVM
      ↓  呼叫 System Call（系統呼叫）
   作業系統（OS Kernel）
      ↓  操控硬體
   CPU / RAM / 硬碟 / 網路卡
```

OS 負責管理：
- **CPU**：決定誰現在可以跑（排程器）
- **記憶體**：給每個程式一塊「虛擬的」記憶體空間
- **I/O**：統一管理磁碟讀寫、網路收發
- **安全**：程式 A 不能偷看程式 B 的記憶體

---

## 一、Process vs Thread：OS 的視角

### Process（行程）

```
Process = 一個執行中的程式 + 它的所有資源

每個 Process 擁有：
  - 獨立的虛擬記憶體空間（別的 Process 看不到）
  - 獨立的檔案描述符（開啟的檔案列表）
  - 至少一個 Thread

當你在終端機執行 java ExchangeApp：
  OS 建立一個新 Process
  → 分配虛擬記憶體（JVM 的 Heap、Stack、Metaspace 都在這裡）
  → 載入 JVM 的可執行檔
  → 建立第一個 Thread（main Thread）
  → 開始執行 main()
```

### Thread（執行緒）

```
Thread = Process 內的執行路徑

多個 Thread 共享：
  同一個 Process 的 Heap 記憶體（這就是 Race Condition 的根源！）
  同一份開啟的檔案列表
  同一個 Code 段

每個 Thread 擁有（私有）：
  自己的 Stack（方法呼叫棧）
  自己的 PC Register（目前執行到哪一條指令）
  自己的暫存器狀態

Java 的 Thread 和 OS Thread 的關係（JDK 21 以前）：
  一個 Java Thread = 一個 OS Thread（1:1 對應）
  建立/銷毀成本：~1 ms（需要向 OS 申請）
  記憶體成本：預設 Stack 大小 512 KB~1 MB
```

**這就是為什麼第十章要用 ExecutorService（執行緒池）：**

```java
// 壞：每個請求都建立一個新 Thread
// 1000 個並發請求 = 建立/銷毀 1000 個 OS Thread
// 每個 Thread 1 MB Stack = 直接佔掉 1 GB 記憶體
for (Request req : requests) {
    new Thread(() -> handle(req)).start();  // 非常浪費！
}

// 好：Thread 池，重用已建立的 Thread
ExecutorService pool = Executors.newFixedThreadPool(16);
for (Request req : requests) {
    pool.submit(() -> handle(req));  // 從池子借 Thread，用完還回去
}
```

## 二、OS 排程器（Scheduler）：誰得到 CPU？

你的電腦同時有幾十個程式在跑，但 CPU 核心的數量有限（例如 8 核）。  
**OS 排程器** 決定每個時刻哪個 Thread 能用 CPU。

### 時間片（Time Slice / Quantum）

```
OS 的排程策略（搶占式多工 Preemptive Multitasking）：

每個 Thread 得到一小段 CPU 時間（時間片），用完了就強制換人：
─────────────────────────────────────────────────────────────
時間  → 0ms    1ms    2ms    3ms    4ms    5ms
Core 0   [A]    [A]    [B]    [B]    [A]    [C]
         Thread A 跑了 2ms，換 B 跑 2ms，再換回 A，再換 C...

使用者感覺 A、B、C「同時在跑」，其實是快速輪流
時間片通常 1~10 ms（Linux 預設約 4ms）
```

### Context Switch（上下文切換）

每次換人，OS 必須：

```
儲存（Save）被換走的 Thread 狀態：
  → 所有 CPU 暫存器的值（16+ 個暫存器）
  → PC（下次從哪裡繼續）
  → Stack Pointer
  → FPU/SIMD 暫存器（浮點運算狀態）

載入（Restore）新 Thread 的狀態：
  → 把上次儲存的暫存器全部恢復

代價：
  純 Context Switch：~ 1~3 μs（微秒）
  加上 Cache/TLB 失效：~ 5~30 μs（快取被污染）

一個忙碌的系統每秒可能發生數千次 Context Switch
```

**這是為什麼大量 Thread 反而讓效能下降：**

```
16 核心 CPU，1000 個 Thread：
每個 Thread 平均 1000/16 = 62.5 個 Thread 競爭一個核心
每秒 250 次 Context Switch per core（假設每個時間片 4ms）
Context Switch 佔用的 CPU 時間本身就是浪費

最佳化策略（LMAX Disruptor 第十四章）：
  核心數 = Thread 數（每個 Core 綁一個 Thread）
  讓 Thread 永遠不需要被換出
  → 零 Context Switch！延遲從 μs 降到 ns 等級
```

## 三、Thread 的六個狀態（Java 視角）

```
                    new Thread()
                         │
                         ▼
                      NEW（新建）
                         │
                    thread.start()
                         │
                         ▼
                   RUNNABLE（就緒/執行中）
                    │         │         │
          synchronized    Object.wait()  Thread.sleep()
          等不到鎖         /join()        /等待 I/O
                    │         │         │
                    ▼         ▼         ▼
               BLOCKED    WAITING   TIMED_WAITING
                    │         │         │
               鎖釋放了    notify()   時間到/notifyAll()
                    │         │         │
                    └────────▶▼◀────────┘
                          RUNNABLE
                              │
                        run() 方法結束
                              │
                              ▼
                        TERMINATED（終止）
```

**重要：BLOCKED vs WAITING 的差異**

```
BLOCKED：等待「鎖」（synchronized 進不去）
  → OS 不安排 CPU 給它（放棄 CPU）
  → 鎖釋放後，OS 安排它重新競爭 CPU

WAITING：等待「通知」（Object.wait() 或 join()）
  → 同樣放棄 CPU
  → 必須有另一個 Thread 呼叫 notify() 才會醒來

TIMED_WAITING：有時限的等待（sleep(1000) 或 wait(500)）
  → 時間到自動醒來
```

## 四、System Call（系統呼叫）：用戶空間 vs 核心空間

### 兩個特權層

```
Ring 0（Kernel Space，核心空間）：
  OS Kernel 執行的地方
  可以直接控制硬體
  可以存取所有記憶體
  任何錯誤可能當機

Ring 3（User Space，用戶空間）：
  你的程式（JVM）執行的地方
  不能直接存取硬體（例如不能直接讀磁碟）
  不能存取別的程式的記憶體
  錯誤只影響這個 Process
```

### System Call 的代價

```java
// 這行 Java 程式碼
System.out.println("Hello");

// 底層發生什麼：
// 1. Java 呼叫 FileOutputStream.write()
// 2. JVM 呼叫 OS 的 write() 系統呼叫
// 3. CPU 從 Ring 3 切換到 Ring 0（代價 ~1 μs）
// 4. OS 把資料寫到輸出緩衝區
// 5. CPU 從 Ring 0 切換回 Ring 3（代價 ~1 μs）

// 每次 System Call 代價 ~1~10 μs（包含 User/Kernel 切換）
// 這就是為什麼 BufferedOutputStream 很重要：
// 把 1000 次小 write() 合併成 1 次大 write()
// 從 1000 次 System Call 降為 1 次
```

**Java I/O 的底層：**

```
Java 的 FileInputStream.read()
  → JNI（Java Native Interface）
  → OS 的 read() 系統呼叫
  → OS 進入 Kernel Mode
  → OS 的 VFS（Virtual File System）層
  → 實際的檔案系統（ext4、NTFS...）
  → 發出磁碟 I/O 請求
  → 等待磁碟回應（HDD: ~10 ms，SSD: ~100 μs）
  → 資料從 Kernel Buffer 複製到 JVM Buffer（一次記憶體複製）
  → 返回 User Space
```

## 五、虛擬記憶體（Virtual Memory）

### 問題：多個程式共享有限的 RAM

```
假設：RAM = 8 GB，同時執行 20 個程式
每個程式說「我需要 2 GB 記憶體」
2 GB × 20 = 40 GB > 8 GB，怎麼辦？
```

### 解法：給每個程式一個「虛假的」完整記憶體空間

```
每個 Process 以為自己有整個記憶體空間（64 位元系統：128 TB 虛擬地址空間）：

Process A 的虛擬記憶體：
  地址 0x0000 ~ 0xFFFF...（128 TB 的「假想空間」）
  A 以為自己獨占了整個空間，其實大部分是假的

Process B 的虛擬記憶體：
  同樣，地址 0x0000 ~ 0xFFFF...
  B 也以為自己獨占，但和 A 看到的「地址相同的位置」其實是不同的實體記憶體

OS 的頁表（Page Table）負責翻譯：
  Process A 的虛擬地址 0x1000 → 實體 RAM 地址 0x50000
  Process B 的虛擬地址 0x1000 → 實體 RAM 地址 0xA0000
  （相同虛擬地址，不同實體地址）
```

### 分頁（Paging）

```
RAM 被分成固定大小的「頁框（Page Frame）」，通常 4 KB：

虛擬記憶體（無限大）
  [Page 0] [Page 1] [Page 2] [Page 3] [Page 4] ...
      ↓ Page Table 翻譯 ↓
實體 RAM（有限）
  [Frame 5] [Frame 2]  —— 不存在 ——  [Frame 9] ...

不在 RAM 裡的頁 → 存在 SSD/HDD 上（Page File / Swap）
```

### Page Fault（缺頁中斷）

```
你的程式存取一個在虛擬空間中「存在」但目前不在 RAM 裡的地址：

1. CPU 發現這個虛擬地址沒有映射到 RAM → 觸發 Page Fault 中斷
2. OS 接管（Ring 0）
3. OS 從 SSD 讀取對應的 Page（~100 μs）
4. 把 Page 載入 RAM（可能需要先踢出一個不常用的 Page）
5. 更新頁表，映射到新的實體地址
6. 返回你的程式繼續執行

代價：一次 Page Fault = ~100 μs~10 ms（取決於在 SSD 還是 HDD）

在高效能系統（如撮合引擎）中：
  啟動時用 -XX:+AlwaysPreTouch 讓 JVM 預先觸發所有 Page Fault
  確保在收單前 Heap 的所有頁都已在 RAM 中
  避免在交易時間突然出現 Page Fault 造成延遲抖動
```

### TLB（Translation Lookaside Buffer）

```
每次記憶體存取都需要查頁表（Page Table），
但頁表本身也在記憶體中 → 多一次記憶體存取 → 太慢了！

解法：TLB（快取頁表的快取）
  TLB 就是「頁表的 L1 Cache」
  容量很小（通常 32~4096 個條目）
  TLB Hit：虛擬地址 → 實體地址翻譯只需 ~1 ns
  TLB Miss：需要查真正的頁表 → 幾十 ns

Context Switch 的隱藏代價：
  切換 Process 時必須清空 TLB（不同 Process 的頁表完全不同）
  新 Process 的前幾百次記憶體存取全是 TLB Miss → 效能暫時下降
  這也是 Context Switch 實際代價遠超「儲存/恢復暫存器」的原因
```

## 六、OS 如何影響 Java 效能

```
現象                    OS 層面原因               解法

GC Stop-the-World      OS 調度器可能在 GC 開始
時間不穩定              時切換 Thread，讓 GC         ZGC/Shenandoah（低停頓）
                       Thread 等待 CPU

Thread.sleep(1) 實際   OS 時間片通常 4ms，          換用 spin-wait（busy-wait）
睡眠時間 > 1 ms         最小精度受限                  但會一直佔 CPU

高並發下 synchronized  大量 Context Switch，         無鎖算法（CAS）
效能崩塌               每次 Unpark/Park 都是          或 LMAX Disruptor
                       System Call

                       （JVM Heap 的 Page 需要載入 RAM）-XX:+AlwaysPreTouch
```

## 本章練習題

**Q1：一個有 8 個 CPU 核心的伺服器上，執行 100 個 Java Thread 和執行 8 個 Java Thread，哪個吞吐量更高？為什麼？**
<details>
<summary>答案</summary>
視工作類型而定。若 Thread 是 CPU-bound（純計算，不等待 I/O）：8 個 Thread 接近最佳，100 個 Thread 反而因為大量 Context Switch（每次切換 ~5~30 μs）而效能下降。若 Thread 是 I/O-bound（等待磁碟、網路）：100 個 Thread 可能更好，因為等待 I/O 時 Thread 進入 WAITING 狀態讓出 CPU，讓其他 Thread 有機會跑，CPU 利用率更高。規則：CPU-bound 任務，Thread 數 ≈ CPU 核心數；I/O-bound 任務，Thread 數可以是 CPU 核心數的幾倍到幾十倍。
</details>

**Q2：為什麼「鎖競爭」在 Java 中這麼昂貴？從 OS 的角度解釋。**
<details>
<summary>答案</summary>
當 Thread A 持有鎖，Thread B 嘗試獲取鎖失敗時：(1) Thread B 呼叫 OS 的 park()（System Call，~1 μs）進入 BLOCKED 狀態；(2) OS 排程器把 B 從 run queue 移走；(3) Context Switch 發生（~5~30 μs），包括保存 B 的所有暫存器狀態；(4) 當 A 釋放鎖時，呼叫 OS 的 unpark()（另一次 System Call）喚醒 B；(5) OS 把 B 放回 run queue；(6) 再次 Context Switch 恢復 B 的狀態。一次鎖競爭 = 至少 2 次 System Call + 2 次 Context Switch，總代價 ~10~60 μs。這就是為什麼 Disruptor 用無鎖設計，把每筆訂單的處理延遲從 μs 降到 ns。
</details>

**Q3：Java 的 `Thread.sleep(1)` 實際上會睡多久？為什麼？**
<details>
<summary>答案</summary>
實際睡眠時間通常大於 1 ms，在 Windows 上可能是 10~15 ms，在 Linux 上是 1~4 ms。原因：(1) sleep() 的精度受 OS 調度器的時間片限制，OS 不會在 1 ms 後立刻喚醒（除非它恰好輪到你的 Thread）；(2) 即使 Thread 準時醒來，它必須等到被 OS 排程器選中才能真正執行，若 CPU 正忙，可能再等一個時間片；(3) Windows 預設時鐘精度是 15.6 ms，需要 timeBeginPeriod(1) 才能提高到 1 ms 精度。結論：不要用 sleep() 做精確的時序控制，高精度場景要用忙等待（busy-wait）或系統特定的計時器。
</details>
