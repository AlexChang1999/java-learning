# 第二十五章：GC 深度剖析
> 為什麼你的程式每隔幾秒就卡一下？為什麼 HFT 系統不用預設 GC？

---

## 前言：GC 是魔法，也是陷阱

Java 不需要手動 `free()` 記憶體——GC 自動回收沒用的物件。  
但「自動」不代表「無代價」。不懂 GC，你永遠不知道你的系統為什麼會抖動。

---

## 一、GC 要解決什麼問題

```java
// 沒有 GC 的語言（C/C++）：程式設計師手動管理
Order* order = new Order("O1", "TSLA", 350.0, 100);  // 分配記憶體
order->match();
delete order;   // 必須手動釋放！
// 忘記 delete → 記憶體洩漏（程式越跑越慢）
// delete 後繼續用 → 懸空指標（undefined behavior，可能當機）

// 有 GC 的語言（Java）：GC 自動回收
Order order = new Order("O1", "TSLA", 350.0, 100);
order.match();
// 不需要 delete，GC 會在適當時機回收不再被引用的物件
```

**GC 需要回答兩個問題：**
1. **哪些物件是垃圾？**（不再被任何地方引用）
2. **什麼時候回收？如何回收？**

---

## 二、如何判斷垃圾：可達性分析（Reachability Analysis）

```
GC Roots（根物件，永遠不是垃圾）：
  - 所有執行中 Thread 的 Stack 上的局部變數
  - 靜態欄位（static fields）
  - JNI 引用

從 GC Roots 出發，能「走到」的物件 = 存活
走不到的物件 = 垃圾，可以回收

例子：
GC Root → OrderBook → [PriceLevel A] → [Order 1, Order 2]
                    → [PriceLevel B] → [Order 3]

Order 4（沒有任何引用指向它）→ 垃圾！

「引用計數（Reference Counting）」為什麼不用：
  int a = new A();  // A 的引用計數 = 1
  int b = a;        // A 的引用計數 = 2
  a = null;         // A 的引用計數 = 1
  b = null;         // A 的引用計數 = 0 → 回收

  但如果 A 引用 B，B 又引用 A（循環引用）：
  a.next = b; b.prev = a;  // 互相引用，引用計數永遠 >= 1，永遠不回收！
  這就是為什麼 Java 不用引用計數。
```

## 三、GC 演算法：三種基本方式

### Mark-Sweep（標記-清除）

```
Phase 1 - Mark（標記）：
  從 GC Roots 出發，把所有可達物件標記為「存活」

Phase 2 - Sweep（清除）：
  掃描整個 Heap，把沒被標記的物件對應的記憶體標記為「可用」

Before：[A][B][C][D][E][F]  A/C/E 存活，B/D/F 垃圾
After： [A][  ][C][  ][E][  ] ← 有很多記憶體碎片！

問題：記憶體碎片（Fragmentation）
  即使總空閒記憶體夠，但沒有連續的大塊空間分配大物件
```

### Mark-Compact（標記-整理）

```
Phase 1 - Mark：和 Mark-Sweep 相同
Phase 2 - Compact（整理）：把存活物件移到記憶體一端

Before：[A][B][C][D][E][F]
After： [A][C][E][         ] ← 存活物件壓縮在一起，後面是連續空閒空間

優點：沒有碎片，分配新物件很快（只需移動 top pointer）
缺點：移動物件需要更新所有引用（代價高）
```

### Copying（複製）

```
把 Heap 分成兩半：From Space 和 To Space

每次 GC：
  把 From Space 中的存活物件複製到 To Space
  From Space 整個清空

From:  [A][B][C][D][E]  B/D 垃圾
To:    [空]

After: [空]            From 變空
       [A][C][E][    ] ← To Space 整齊，沒有碎片

缺點：同一時間只能用一半的 Heap
優點：速度快，分配物件超快，沒有碎片
適用：短命物件多的場景（Young Generation！）
```

## 四、分代假說：大多數物件「朝生暮死」

Java 的 GC 設計者觀察到：

```
絕大多數物件的生命週期非常短：
  計算中間結果的臨時物件 → 用完立刻變垃圾
  方法呼叫中的局部物件   → 方法返回就變垃圾
  Stream 中間操作的物件  → 用完就拋

少數物件存活很久：
  OrderBook（整個程式都在用）
  資料庫連線池（一直保持）
  靜態設定物件（永不回收）
```

### Heap 的分代結構

```
┌────────────────────────────────────────────────────────┐
│                      JVM Heap                          │
│                                                        │
│  ┌──────────────────────────┐  ┌─────────────────────┐│
│  │      Young Generation    │  │   Old Generation    ││
│  │  ┌────────┐  ┌───┐ ┌───┐│  │  （也叫 Tenured）   ││
│  │  │  Eden  │  │S0 │ │S1 ││  │  存活很久的物件     ││
│  │  └────────┘  └───┘ └───┘│  │  佔 Heap 的 2/3     ││
│  │  新物件先在 Eden 出生   │  └─────────────────────┘│
│  │  Eden 佔 Young Gen 的  │                           │
│  │  80%，S0/S1 各 10%     │  Minor GC 後還活著的物件  │
│  └──────────────────────────┘  會晉升（Promote）到 Old  │
└────────────────────────────────────────────────────────┘
```

### Minor GC：只清理 Young Generation

```
步驟：
1. Eden 滿了，觸發 Minor GC
2. 標記 Eden + 當前 Survivor（S0）中的存活物件
3. 把存活物件複製到另一個 Survivor（S1）
   → 每個物件帶一個「年齡計數器」，每次 Minor GC +1
4. Eden 和 S0 整個清空
5. S0 和 S1 角色對調（下次 GC 時從 S1 複製到 S0）

當物件年齡 >= 15（預設），晉升到 Old Generation

特點：
  速度快（通常 < 10 ms）
  只掃 Young Gen（佔 Heap 的 1/3）
  頻率高（數秒到數十秒觸發一次）
```

### Major GC / Full GC：清理整個 Heap

```
觸發條件：
  Old Generation 空間不足
  顯式呼叫 System.gc()（不建議！）
  Meta Space 不足（Class 載入太多）

代價遠高於 Minor GC：
  掃描整個 Heap（包含 Old Gen）
  通常需要 100 ms ~ 數秒
  這就是「Stop-the-World（STW）」停頓！

Stop-the-World（STW）：
  GC 開始時，所有 Java Thread 強制暫停
  JVM 不能讓物件在 GC 標記期間移動（否則引用會失效）
  對撮合引擎來說：STW = 訂單處理停止 = 不可接受！
```

## 五、現代 GC 收集器對比

### Serial GC（序列 GC）

```
特點：單執行緒執行 GC，執行時所有 Java Thread 停止
適合：小堆（< 100 MB）、單核 CPU
不適合：生產環境
STW 時間：100ms ~ 數秒（依 Heap 大小）
```

### Parallel GC（並行 GC，Java 8 預設）

```
特點：多個 GC Thread 同時工作（但 Java Thread 仍然停止）
Young Gen：多執行緒 Parallel Copy
Old Gen：多執行緒 Mark-Compact
適合：吞吐量優先（批次處理、科學計算）
STW 時間：比 Serial 快，但仍可能有幾百 ms 停頓
```

### G1 GC（Garbage-First，Java 9+ 預設）

```
革命性改變：把 Heap 切成很多小塊（Region，通常 1~32 MB）：

傳統：[     Young Gen     ] [          Old Gen          ]
G1：  [Y][O][Y][O][Y][Y][O][Y][O][O][Y][O][Y][H][O][Y]
      Y=Young, O=Old, H=Humongous（大物件）

好處：
  可以只選「垃圾最多的 Region」先回收（Garbage First）
  可預測停頓時間：-XX:MaxGCPauseMillis=200（目標最多停 200 ms）

代價：
  比 Parallel GC 消耗更多 CPU 來維護 Region 元資料
  適合：需要低延遲且 Heap > 4 GB 的應用
STW 時間：通常 50~200 ms，可設定目標
```

### ZGC（Z Garbage Collector，Java 15 正式）

```
革命性設計：幾乎所有 GC 工作都在 Java Thread 執行時「並發」完成

核心技術：Colored Pointers + Load Barriers
  每個物件引用帶有元資料位元（顏色位）
  CPU 存取物件時，Load Barrier 自動檢查並更新引用
  GC 可以在物件還在被使用時移動它，並在 CPU 讀取時修正引用

STW 時間：< 1 ms（無論 Heap 多大！即使 1 TB Heap）
代價：
  每次物件存取有微小的 Load Barrier 開銷（~1~3%）
  適合：低延遲系統（金融、遊戲後端）

-XX:+UseZGC
```

### Shenandoah GC

```
類似 ZGC，也是幾乎完全並發
由 Red Hat 開發，OpenJDK 內建
STW 時間：< 1 ms（和 ZGC 同等級）
適合：與 ZGC 類似，可以對比測試選擇
-XX:+UseShenandoahGC
```

### 選擇指南

```
吞吐量優先（批次處理、資料分析）：
  → Parallel GC（-XX:+UseParallelGC）

延遲敏感、Heap < 8 GB（一般 Web 後端）：
  → G1 GC（預設，不需要設定）

超低延遲（金融、撮合引擎、遊戲）：
  → ZGC（-XX:+UseZGC）
  → Shenandoah（-XX:+UseShenandoahGC）

撮合引擎最佳實踐：
  → ZGC + -XX:+AlwaysPreTouch（啟動時預先觸發 Page Fault）
  → -Xms == -Xmx（固定 Heap 大小，避免動態擴展時的停頓）
  → -XX:ZUncommitDelay=600（GC 後不立刻把記憶體還給 OS）
```

## 六、GC 調優實戰：常用 JVM 參數

```bash
# 基本 Heap 設定
-Xms4g          # 初始 Heap 大小（和 Xmx 設相同，避免動態擴展停頓）
-Xmx4g          # 最大 Heap 大小

# GC 選擇
-XX:+UseG1GC        # 使用 G1（Java 9+ 預設）
-XX:+UseZGC         # 使用 ZGC（低延遲首選）
-XX:MaxGCPauseMillis=100  # G1 的停頓目標（毫秒）

# GC 日誌（診斷必備）
-Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m
# 記錄所有 GC 事件到 gc.log，保留最近 5 個，每個 20 MB

# 效能調優
-XX:+AlwaysPreTouch    # 啟動時預先填充 Heap（避免執行時 Page Fault）
-XX:+DisableExplicitGC # 禁用 System.gc()（防止程式碼意外觸發 Full GC）

# 物件年齡晉升
-XX:MaxTenuringThreshold=15  # 物件存活幾次 Minor GC 後晉升 Old Gen（預設 15）
```

## 七、記憶體洩漏的四大陷阱

```java
// 陷阱一：靜態集合持有物件引用
public class Cache {
    private static final Map<String, Order> cache = new HashMap<>();

    public static void put(String key, Order order) {
        cache.put(key, order);  // 放進去了，什麼時候移出來？
    }
    // 沒有 remove 機制 → 訂單永遠留在靜態 Map → OOM
}
// 解法：用 WeakHashMap（value 是弱引用，GC 可以回收）
//       或設定 TTL（存活時間），用 Caffeine/Guava Cache

// 陷阱二：非靜態內部類別持有外部類別引用
public class OrderBook {
    class TradeListener {
        void onTrade(Trade t) {
            // 即使 TradeListener 不用了，OrderBook 也不能被 GC
            // 因為 TradeListener 隱含持有 OrderBook.this 的引用
        }
    }
}
// 解法：改用 static 內部類別，或 Lambda

// 陷阱三：ThreadLocal 未清理
public void handleRequest(Order order) {
    MDC.put("orderId", order.getOrderId());
    process(order);
    // 忘記 MDC.clear()！
    // 執行緒回到執行緒池後，ThreadLocal 值依然存在
    // 下個請求看到的 MDC 是上個請求的殘留
}
// 解法：finally { MDC.clear(); }

// 陷阱四：未關閉的資源（第十九章有提）
Connection conn = getConnection();
// 沒有 try-with-resources，Connection 物件的 GC 不會關閉底層的 TCP 連線
// 解法：永遠用 try-with-resources
```

## 本章練習題

**Q1：Minor GC 和 Major GC 的觸發條件和代價有何不同？**
<details>
<summary>答案</summary>
Minor GC：當 Eden 區滿時觸發，只回收 Young Generation（佔 Heap 1/3），使用複製演算法，速度快（通常 < 10 ms），Stop-the-World 時間短。Major GC（Full GC）：當 Old Generation 空間不足時觸發，需要回收整個 Heap，使用 Mark-Compact 演算法，速度慢（可能 100ms~數秒），Stop-the-World 時間長。優化方向：透過調整 Heap 比例（Young/Old）和物件晉升閾值，減少 Full GC 的頻率。
</details>

**Q2：ZGC 怎麼能做到「幾乎不停頓」？它的代價是什麼？**
<details>
<summary>答案</summary>
ZGC 使用「彩色指標（Colored Pointers）」和「讀屏障（Load Barrier）」。每個物件引用包含元資料位元（記錄物件是否被移動、是否已標記）。當 Java Thread 存取物件時，CPU 的 Load Barrier 會自動檢查這些位元，如果物件已被 GC 移動，就自動修正引用。因此 GC 可以在 Java Thread 繼續執行時移動物件，不需要完全停頓。代價：每次物件存取都有約 1~3% 的 Load Barrier 開銷（多一次條件判斷），整體吞吐量比 Parallel GC 略低。對延遲敏感但能接受微小吞吐量損失的系統（如撮合引擎），這個代價完全值得。
</details>

**Q3：你的程式從不 new 大物件，Heap 設得也不小，為什麼還是發生了 OutOfMemoryError？**
<details>
<summary>答案</summary>
可能是記憶體洩漏而非記憶體不足：(1) 靜態 Map/List 持續累積物件，GC 無法回收（因為仍有引用）；(2) ThreadLocal 沒有清理，執行緒池中每個執行緒都持有一份資料累積；(3) 大量 Class 載入導致 Metaspace 溢出（-XX:MetaspaceSize 設定不足）；(4) 直接記憶體（Direct Buffer）洩漏（不在 JVM Heap，但也受記憶體限制）。診斷工具：heap dump（jmap -dump:format=b,file=heap.hprof PID），再用 Eclipse MAT 或 VisualVM 分析哪些物件佔用最多記憶體、誰持有它們的引用。
</details>
