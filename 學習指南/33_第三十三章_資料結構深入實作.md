## 一、為什麼要懂資料結構？

第六章學完 Big O 之後，你已經知道：

| 操作 | ArrayList | LinkedList | TreeMap | HashMap |
|------|-----------|------------|---------|---------|
| 查詢 | O(1)      | O(n)       | O(log n)| O(1)    |
| 插入 | O(n)*     | O(1)       | O(log n)| O(1)    |

但這張表只告訴你「結果」，沒有告訴你「為什麼」。

### 只知道 Big O 還不夠

考慮這個面試問題：

> 「你的撮合引擎訂單簿用 `TreeMap`，為什麼不用 `PriorityQueue`？  
>  如果每秒有 10 萬筆訂單進來，兩者效能差在哪？」

如果你只知道「TreeMap 是 O(log n)，PriorityQueue 也是 O(log n)」，你沒辦法回答。

你需要知道：
- `TreeMap` 底層是**紅黑樹**，支援按 key 範圍查詢（取最大買價/最小賣價），也支援 O(log n) 刪除任意節點
- `PriorityQueue` 底層是**Heap**，只能高效取出「最值」，刪除任意節點是 O(n)
- 當你需要「取消任意一筆訂單」時，PriorityQueue 是災難

這就是本章的目標：**讓你從「知道結果」進化到「理解原因」**。

### 撮合引擎的資料結構地圖

```
撮合引擎
│
├── 訂單簿（Order Book）
│   ├── TreeMap<Double, Queue<Order>>   ← 紅黑樹，按價格排序
│   └── HashMap<String, Order>          ← 雜湊表，按訂單ID查詢
│
├── 行情快取（Market Data Cache）
│   └── Redis Sorted Set（ZSet）        ← 跳表，支援並發讀取
│
└── 優先撮合（Priority Matching）
    └── 偶爾用 PriorityQueue            ← Heap，快速取最優價
```

每一種資料結構都有它最適合的使用場景。現在讓我們一個個拆解。

## 二、二元搜尋樹（Binary Search Tree，BST）

### 2.1 定義

BST 是一種樹狀結構，滿足一個規則：

> 對任意節點 N：
> - N 的**左子樹**中，所有節點的值 **< N**
> - N 的**右子樹**中，所有節點的值 **> N**

這個規則讓「搜尋」變得像猜數字遊戲：每次比較都能排除一半。

### 2.2 圖解：插入 5、3、7、1、4、6、8

```
插入順序：5 → 3 → 7 → 1 → 4 → 6 → 8

        5
       / \
      3   7
     / \ / \
    1  4 6  8
```

查詢 4 的路徑：5 → 3（4 > 3，往右）→ 4（找到！），只走了 3 步。

### 2.3 Java 實作

```java
// BST 節點定義
class BSTNode {
    int val;
    BSTNode left, right;

    BSTNode(int val) {
        this.val = val;
    }
}

class BST {

    BSTNode root;

    // 插入節點（遞迴版）
    // 時間複雜度：O(h)，h 是樹的高度
    public BSTNode insert(BSTNode node, int val) {
        // 基礎情況：到達空節點，建立新節點
        if (node == null) return new BSTNode(val);

        if (val < node.val) {
            // 比當前節點小，往左走
            node.left = insert(node.left, val);
        } else if (val > node.val) {
            // 比當前節點大，往右走
            node.right = insert(node.right, val);
        }
        // val == node.val：BST 通常不插入重複值，直接回傳
        return node;
    }

    // 查詢節點（遞迴版）
    public boolean search(BSTNode node, int val) {
        // 基礎情況：節點不存在 → 找不到
        if (node == null) return false;

        if (val == node.val) return true;           // 找到了
        if (val < node.val) return search(node.left, val);   // 往左找
        return search(node.right, val);             // 往右找
    }

    // 刪除節點（遞迴版）— 最複雜的操作
    public BSTNode delete(BSTNode node, int val) {
        if (node == null) return null;

        if (val < node.val) {
            // 目標在左子樹
            node.left = delete(node.left, val);
        } else if (val > node.val) {
            // 目標在右子樹
            node.right = delete(node.right, val);
        } else {
            // 找到目標節點，三種情況：
            if (node.left == null) return node.right;  // 情況1：只有右子樹
            if (node.right == null) return node.left;  // 情況2：只有左子樹

            // 情況3：有兩個子樹
            // 策略：找右子樹中的最小值（中序後繼）來替換
            BSTNode successor = findMin(node.right);
            node.val = successor.val;                  // 用後繼值替換
            node.right = delete(node.right, successor.val); // 刪掉後繼
        }
        return node;
    }

    // 找子樹中的最小值（一路向左）
    private BSTNode findMin(BSTNode node) {
        while (node.left != null) node = node.left;
        return node;
    }
}
```

### 2.4 最壞情況：BST 退化

如果你把已排序的資料依序插入，BST 會退化成鏈結串列：

```
插入順序：1 → 2 → 3 → 4 → 5（已排序）

退化的 BST：         平衡的 BST（理想狀態）：
1                          3
 \                        / \
  2                      1   5
   \                      \  /
    3                      2 4
     \
      4
       \
        5

查詢 5：需要走 5 步          查詢 5：只需要走 2 步
時間複雜度 O(n)              時間複雜度 O(log n)
```

這就是為什麼我們需要平衡樹。

## 三、平衡樹（AVL 樹 / 紅黑樹）

### 3.1 為什麼需要平衡？

BST 的效能完全取決於樹的**高度 h**。

- 完全平衡的樹：h = log₂(n)，查詢 O(log n)
- 退化成鏈結串列：h = n，查詢 O(n)

平衡樹的目標就是**強制限制高度**，保證 h 維持在 O(log n)。

### 3.2 AVL 樹

AVL 樹是最嚴格的平衡樹：

> **AVL 規則**：每個節點的左右子樹高度差（平衡因子 Balance Factor）的絕對值 ≤ 1。

```
合法的 AVL 樹：              不合法（需要旋轉）：
    5                             5
   / \                             \
  3   7    ← 高度差 = 0             7     ← 5的平衡因子 = -2（不合法）
 /                                   \
1                                     9
```

當插入節點破壞平衡時，AVL 樹會進行**旋轉**（Rotation）來修復。

**右旋（Right Rotation）範例**：

```
插入 1 後，5 的平衡因子 = 2（左偏），需要右旋：

插入前：        右旋後：
    5              3
   /              / \
  3              1   5
 /
1

右旋操作：「把 3 提上來，5 降下去」
```

AVL 保證高度 ≤ 1.44 * log₂(n)，非常嚴格。

### 3.3 紅黑樹（Red-Black Tree）

紅黑樹是 **Java `TreeMap` / `TreeSet` 的底層實作**，也是工業界最廣泛使用的平衡樹。

**五條規則**（理解大意即可，不需要背）：

1. 每個節點是紅色或黑色
2. 根節點是黑色
3. 葉節點（null 節點）是黑色
4. 紅色節點的子節點必須是黑色（不能有兩個連續紅節點）
5. 從任意節點到其所有後代葉節點的路徑，包含相同數量的黑色節點

```
紅黑樹示意（R=紅, B=黑）：

        7(B)
       /    \
     3(R)   11(R)
    /  \    /  \
  1(B) 5(B) 9(B) 13(B)
```

**直覺理解**：顏色 = 寬鬆的高度約束

- 規則 4 說「不能有兩個連續紅」→ 紅色節點不計入「有效高度」
- 規則 5 說「黑色節點數量相同」→ 黑色節點構成的骨架是平衡的
- 結合起來：最長路徑（紅黑交替）≤ 2 × 最短路徑（全黑）

因此紅黑樹的高度保證 ≤ 2 * log₂(n)，比 AVL 稍寬鬆，但仍是 O(log n)。

### 3.4 為什麼 Java 選紅黑樹而不是 AVL？

| 特性 | AVL 樹 | 紅黑樹 |
|------|--------|--------|
| 平衡嚴格度 | 非常嚴格（高度差≤1） | 較寬鬆（高度差≤2x） |
| 查詢效能 | 略快（樹更矮） | 略慢 |
| 插入/刪除旋轉次數 | 多（每次可能多次旋轉） | 少（最多 3 次旋轉） |
| 適合場景 | 讀多寫少 | 讀寫均衡 |

**結論**：TreeMap 需要頻繁插入和刪除（每次下單、成交、撤單都是修改），紅黑樹的旋轉成本更低，整體吞吐量更高。

### 3.5 撮合引擎：訂單簿設計

```java
import java.util.*;

// 訂單簿用 TreeMap：按價格排序，O(log n) 找最優報價
class OrderBook {
    // 買方：價格越高越優先，用 reverseOrder() 讓最大值在 first()
    private TreeMap<Double, Queue<Order>> bids =
        new TreeMap<>(Comparator.reverseOrder());

    // 賣方：價格越低越優先，預設升序，最小值在 first()
    private TreeMap<Double, Queue<Order>> asks = new TreeMap<>();

    // 加入買單，O(log n)
    public void addBid(Order order) {
        // computeIfAbsent：如果這個價格的 Queue 不存在，就建立一個新的
        bids.computeIfAbsent(order.price, k -> new LinkedList<>())
            .offer(order);
    }

    // 取得最高買價，O(log n)
    public double getBestBid() {
        return bids.isEmpty() ? 0 : bids.firstKey();
    }

    // 取得最低賣價，O(log n)
    public double getBestAsk() {
        return asks.isEmpty() ? Double.MAX_VALUE : asks.firstKey();
    }
}
```

## 四、堆積（Heap / Priority Queue）

### 4.1 定義

Heap 是一種特殊的**完全二元樹**（Complete Binary Tree），滿足：

- **最大堆（Max Heap）**：每個節點的值 ≥ 其子節點的值
- **最小堆（Min Heap）**：每個節點的值 ≤ 其子節點的值

**完全二元樹**：節點從上到下、從左到右填滿，不允許有空缺（除了最後一層最右側）。

### 4.2 用陣列表示 Heap

完全二元樹有個神奇的性質：可以直接用陣列存，不需要指標！

```
陣列：[10, 9, 8, 7, 6, 5, 4]
索引：[ 0, 1, 2, 3, 4, 5, 6]

對應的樹形：
            10 (index 0)
           /            \
        9 (1)           8 (2)
       /    \          /    \
    7 (3)  6 (4)    5 (5)  4 (6)

計算公式：
- 節點 i 的父節點：(i - 1) / 2
- 節點 i 的左子節點：2 * i + 1
- 節點 i 的右子節點：2 * i + 2

驗證：節點 3（值為 7），父節點 = (3-1)/2 = 1（值為 9）✓
      節點 1（值為 9），左子 = 2*1+1 = 3（值為 7）✓
```

陣列表示比指標節省記憶體，也更 **Cache 友好**（相鄰元素在連續記憶體，參考第二十三章）。

### 4.3 核心操作

**heapify-up（插入後往上修復）**：

```java
// 插入新元素 → 放到陣列末尾 → 往上「冒泡」直到滿足 Heap 性質
private void heapifyUp(int[] heap, int index) {
    while (index > 0) {
        int parent = (index - 1) / 2;     // 計算父節點索引

        // 最大堆：如果比父節點大，就往上交換
        if (heap[index] > heap[parent]) {
            swap(heap, index, parent);
            index = parent;                // 繼續往上比較
        } else {
            break;                         // 不需要繼續，已滿足 Heap 性質
        }
    }
}
```

**heapify-down（取出根節點後往下修復）**：

```java
// 取出最大值 → 把末尾元素移到根 → 往下「沉降」直到滿足 Heap 性質
private void heapifyDown(int[] heap, int size, int index) {
    while (true) {
        int left = 2 * index + 1;          // 左子節點索引
        int right = 2 * index + 2;         // 右子節點索引
        int largest = index;               // 先假設當前節點最大

        // 找出當前節點、左子、右子中最大的
        if (left < size && heap[left] > heap[largest]) largest = left;
        if (right < size && heap[right] > heap[largest]) largest = right;

        if (largest != index) {
            swap(heap, index, largest);    // 把最大的換上來
            index = largest;               // 繼續往下比較
        } else {
            break;                         // 已滿足 Heap 性質
        }
    }
}
```

### 4.4 Java PriorityQueue 使用範例

```java
import java.util.PriorityQueue;

// Java 的 PriorityQueue 預設是最小堆
PriorityQueue<Integer> minHeap = new PriorityQueue<>();
minHeap.offer(5);
minHeap.offer(1);
minHeap.offer(3);
System.out.println(minHeap.poll()); // 輸出 1（最小值）

// 最大堆：傳入 Comparator.reverseOrder()
PriorityQueue<Integer> maxHeap =
    new PriorityQueue<>(Comparator.reverseOrder());
maxHeap.offer(5);
maxHeap.offer(1);
maxHeap.offer(3);
System.out.println(maxHeap.poll()); // 輸出 5（最大值）
```

### 4.5 撮合引擎：為什麼訂單簿不用 PriorityQueue？

這是個非常好的設計問題：

```
需求：
1. 快速取得最優買價（最高）/ 最優賣價（最低）→ PriorityQueue 可以做到 O(log n)
2. 按訂單 ID 取消任意一筆訂單                   → PriorityQueue 做不到！

PriorityQueue 的弱點：
- poll() 取出根節點：O(log n) ✓
- remove(element)：需要先線性搜尋找到元素 O(n)，然後再 heapify O(log n)
- 總計取消訂單：O(n) ✗

解決方案：TreeMap + HashMap 組合

TreeMap<Double, Queue<Order>> priceLevel   → O(log n) 按價格排序、取最優
HashMap<String, Order> orderIndex          → O(1) 按訂單 ID 查找

取消訂單：
1. HashMap 用 orderId 找到 Order：O(1)
2. 從 Order 取得 price，去 TreeMap 找到那層 Queue
3. 從 Queue 移除該 Order：O(n)，但同價格的訂單數量通常很少

整體效果遠比純 PriorityQueue 好。
```

## 五、Hash Table（雜湊表）

### 5.1 原理

雜湊表的核心思想：

```
key（字串 "ORD-001"）
    ↓ 雜湊函數（Hash Function）
    ↓ 計算出整數，例如 12345
    ↓ 12345 % 陣列長度（例如 16）= 9
bucket[9] → 存放 Order 物件
```

理想情況下，查詢就是：
1. 計算 hash：O(1)
2. 直接存取 `array[hash]`：O(1)

總計 O(1)！

### 5.2 碰撞（Collision）

問題：不同的 key 可能計算出相同的 bucket index。

例如 `"ORD-001"` 和 `"ORD-999"` 都映射到 bucket[9]，這就是**碰撞**。

**解法一：鏈結法（Chaining）**— Java HashMap 採用此方法

```
bucket[9] → LinkedList[ ("ORD-001", Order1) → ("ORD-999", Order9) ]

查詢時：
1. 計算 hash，找到 bucket[9]：O(1)
2. 在鏈結串列中逐一比較 key：O(k)，k 是串列長度
```

```
bucket 陣列（假設長度為 8）：

index 0: → null
index 1: → [A → D]    ← A 和 D 碰撞，用鏈結串列串起來
index 2: → [B]
index 3: → null
index 4: → [C → E → G] ← 三個碰撞
index 5: → [F]
index 6: → null
index 7: → null
```

**解法二：開放定址（Open Addressing）**

碰撞時往下找下一個空位。Python dict 採用此方法。

```
想插入 key K，計算出 hash = 3，但 bucket[3] 已被佔用
→ 嘗試 bucket[4]，還是被佔
→ 嘗試 bucket[5]，空的！存進去

查詢時也要同樣的探測步驟。
```

### 5.3 Java HashMap 的內部細節

```java
// HashMap 的底層是陣列 + 鏈結串列（桶很深時升級為紅黑樹）
//
// 預設陣列長度：16
// 預設 Load Factor：0.75
//
// 當 size > capacity * loadFactor 時，觸發 rehash：
// - 容量翻倍（16 → 32 → 64...）
// - 重新計算所有 key 的位置（O(n) 操作！）
//
// 當某個 bucket 的鏈結串列長度 > 8 時，
// 且整個陣列長度 >= 64，升級為紅黑樹（O(log n) 查詢）

// 為什麼要預估初始容量？
// 不指定初始大小：如果要存 10000 筆資料
// 16 → 32 → 64 → ... → 16384，需要 rehash 10 次
// 每次 rehash 都是 O(n)，浪費大量時間！

// 正確做法：預估容量 / loadFactor + 1
int expectedSize = 10000;
HashMap<String, Order> map = new HashMap<>(
    (int)(expectedSize / 0.75) + 1  // 約 13334，HashMap 會選下一個 2 的冪
);
```

### 5.4 撮合引擎：快速查單

```java
// 用 HashMap 實現 O(1) 取消訂單
class MatchingEngine {
    // orderId → Order 映射，查詢和取消都是 O(1)
    private HashMap<String, Order> orderIndex = new HashMap<>(65536);

    // 新增訂單
    public void addOrder(Order order) {
        orderIndex.put(order.orderId, order);    // O(1)
        // 同時加入訂單簿的 TreeMap...
    }

    // 取消訂單 by ID
    public boolean cancelOrder(String orderId) {
        Order order = orderIndex.remove(orderId);  // O(1) 查找並移除
        if (order == null) return false;           // 訂單不存在

        // 從訂單簿移除（TreeMap + Queue 操作）
        removeFromBook(order);
        return true;
    }
}
```

## 六、跳表（Skip List）

### 6.1 問題：鏈結串列的困境

鏈結串列有個根本缺陷：

```
搜尋值 7 的過程（有序鏈結串列）：

head → 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8 → 9
        ↑       ↑       ↑       ↑
       比較1   比較2   比較3   找到！

必須從頭開始，一個一個比較，O(n)
即使是有序串列，也無法像陣列那樣做二元搜尋（沒有索引）
```

### 6.2 Skip List 的思想：建立快速通道

解法：在鏈結串列上面加「快速通道層」，每層跳過一些節點：

```
4 層跳表結構（n = 8 個節點）：

層 3（最頂層，跨度最大）：
head ─────────────────────────→ 6 ──────────→ null

層 2：
head ──────────→ 3 ───────────→ 6 ──→ 8 ──→ null

層 1：
head ──→ 1 ──→ 3 ──→ 5 ──→ 6 ──→ 7 ──→ 8 ──→ null

層 0（底層，完整串列）：
head → 1 → 2 → 3 → 4 → 5 → 6 → 7 → 8 → null

搜尋 7 的路徑：
層 3：head → 6（7 > 6，繼續）→ null（超過，往下一層）
層 2：從 6 → 8（7 < 8，往下一層）
層 1：從 6 → 7（找到！）

只走了 4 步，而不是 7 步！
```

### 6.3 查詢複雜度分析

- 每層大約是下層的一半節點（概率 50% 晉升到上層）
- 總層數 ≈ log₂(n)
- 每層最多比較幾次 → 期望查詢時間：**O(log n)**

```java
// 跳表節點
class SkipListNode {
    int val;
    SkipListNode[] forward;  // forward[0] = 下一層，forward[i] = 第 i 層的下一個

    SkipListNode(int val, int level) {
        this.val = val;
        this.forward = new SkipListNode[level + 1];
    }
}

class SkipList {
    private static final int MAX_LEVEL = 16;  // 最多 16 層，支援 2^16 ≈ 65536 個節點
    private static final double PROBABILITY = 0.5; // 晉升概率

    private SkipListNode head;   // 哨兵頭節點
    private int currentLevel;    // 當前最高層數
    private Random random = new Random();

    public SkipList() {
        // 頭節點不存真實資料，只作為起點
        head = new SkipListNode(Integer.MIN_VALUE, MAX_LEVEL);
        currentLevel = 0;
    }

    // 搜尋，O(log n) 期望時間
    public boolean search(int target) {
        SkipListNode cur = head;

        // 從最高層開始往下找
        for (int i = currentLevel; i >= 0; i--) {
            // 在當前層向右移動，直到下一個節點 >= target
            while (cur.forward[i] != null && cur.forward[i].val < target) {
                cur = cur.forward[i];
            }
            // 下降到更低的層（更細緻的搜尋）
        }

        // 此時 cur 是第 0 層中最後一個 < target 的節點
        cur = cur.forward[0];
        return cur != null && cur.val == target;
    }

    // 隨機決定新節點的層高（幾何分布）
    private int randomLevel() {
        int level = 0;
        while (random.nextDouble() < PROBABILITY && level < MAX_LEVEL) {
            level++;
        }
        return level;
    }
}
```

### 6.4 Redis Sorted Set（ZSet）底層用跳表

Redis 的 Sorted Set（ZSet）同時用了**跳表 + 哈希表**：

```
ZSet 結構：
- 跳表（Skip List）：按 score 排序，支援範圍查詢
- 哈希表（Hash Table）：按 member 名稱查找 score

為什麼 Redis 選跳表而不是紅黑樹？

1. 並發讀取更友好
   - 紅黑樹旋轉需要鎖住整棵樹（或複雜的 CAS 操作）
   - 跳表修改只影響少數節點，更容易實現無鎖並發

2. 範圍查詢更簡單
   - 紅黑樹做範圍查詢需要中序遍歷
   - 跳表的底層就是有序鏈結串列，範圍查詢 = 找到起點後往右掃

3. 實作簡單，易於維護
   - 紅黑樹的旋轉邏輯複雜，容易出 bug
   - 跳表結構直觀，調試方便
```

## 七、綜合比較：撮合引擎的選擇理由

| 需求 | 資料結構 | 原因 |
|------|----------|------|
| 訂單簿按價格排序，取最優報價 | `TreeMap`（紅黑樹） | O(log n) 有序存取，支援範圍查詢 |
| 按訂單 ID 取消訂單 | `HashMap`（雜湊表） | O(1) 查找，不需要排序 |
| 行情快取，支援並發讀取 | Redis ZSet（跳表） | 並發友好，範圍查詢簡單 |
| 偶爾需要快速取單一最值 | `PriorityQueue`（Heap） | O(1) 取根，O(log n) 插入 |

### 記憶技巧

```
需要「找最大/最小的那一個」  → Heap（PriorityQueue）
需要「按順序排列，也能取消任意一個」 → 平衡樹（TreeMap）
需要「用名字立刻找到那個人」 → Hash Table（HashMap）
需要「有序但要支援並發」     → Skip List（Redis ZSet）
```

## 八、硬體關聯：為什麼陣列比鏈結串列快？

（延伸自第二十三章計算機組成原理）

```
陣列的記憶體佈局（連續）：
[元素0][元素1][元素2][元素3][元素4]...
 ↑ CPU 讀取 元素0 時，Cache Line 會預先載入後面的元素
   → Cache Hit！下一次存取 元素1 非常快

鏈結串列的記憶體佈局（分散）：
元素0 在地址 0x100
元素1 在地址 0x8F4   ← 跳來跳去，記憶體不連續
元素2 在地址 0x2C0

 → 每次存取都可能 Cache Miss！需要重新從主記憶體載入

Heap（完全二元樹存在陣列）的優點：
- 節點按 2i+1、2i+2 規律分布
- 父子節點在陣列中距離很近
- Cache Hit 率高，實際效能比理論 O(log n) 更好

紅黑樹的缺點（相對 Heap）：
- 每個節點單獨 new 出來，散落在 Heap 記憶體各處
- 樹越深，Cache Miss 越多
- 這是為什麼在 CPU 效能敏感的場景，有人改用 B-Tree 或 Cache-Oblivious 結構
```


<details>
<summary>點我展開答案</summary>

```java
int findKthLargest(int[] nums, int k) {
    // 最小堆，大小維持為 k
    PriorityQueue<Integer> minHeap = new PriorityQueue<>();

    for (int num : nums) {
        minHeap.offer(num);         // 加入元素：O(log k)

        if (minHeap.size() > k) {
            minHeap.poll();          // 超過 k 個就丟掉最小的：O(log k)
        }
        // 此時堆裡永遠是「遇到過的最大 k 個數字」
        // 堆頂（最小）= 這 k 個數中最小的 = 第 k 大
    }

    return minHeap.peek();           // 回傳堆頂，就是第 k 大
}

// 時間複雜度：O(n log k)
// 空間複雜度：O(k)
```

**直覺**：堆裡放著「截至目前遇過的前 K 大數字」。最小堆的頂端就是這 K 個數中最小的——也就是全局的第 K 大。

</details>

### 題目三：設計 LRU 快取

設計一個 LRU（最近最少使用）快取，支援 O(1) 的 `get` 和 `put` 操作。

- `put(key, value)`：插入鍵值對；如果容量超過上限，移除最久未使用的

```java

    int capacity;
    // 請填入實作
}
```

<details>
<summary>點我展開答案</summary>

```java
class LRUCache {

    // 雙向鏈結串列節點
    class Node {
        int key, val;
        Node prev, next;
        Node(int k, int v) { key = k; val = v; }
    }

    private int capacity;
    private HashMap<Integer, Node> map = new HashMap<>();  // key → Node

    // 用哨兵頭尾節點，避免 null 檢查
    private Node head = new Node(0, 0);  // 哨兵頭：最近使用端
    private Node tail = new Node(0, 0);  // 哨兵尾：最久未使用端

    public LRUCache(int capacity) {
        this.capacity = capacity;
        head.next = tail;
        tail.prev = head;
    }

    public int get(int key) {
        if (!map.containsKey(key)) return -1;

        Node node = map.get(key);
        moveToHead(node);   // 最近使用，移到頭部
        return node.val;
    }

    public void put(int key, int val) {
        if (map.containsKey(key)) {
            // 已存在：更新值並移到頭部
            Node node = map.get(key);
            node.val = val;
            moveToHead(node);
        } else {
            // 新增
            Node node = new Node(key, val);
            map.put(key, node);
            addToHead(node);

            if (map.size() > capacity) {
                // 超過容量，移除最久未使用（尾部）
                Node lru = removeTail();
                map.remove(lru.key);
            }
        }
    }

    // 將節點加到頭部（最近使用）
    private void addToHead(Node node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }

    // 從串列中移除節點
    private void removeNode(Node node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void moveToHead(Node node) {
        removeNode(node);
        addToHead(node);
    }

    private Node removeTail() {
        Node lru = tail.prev;
        removeNode(lru);
        return lru;
    }
}

// 使用範例：
// LRUCache cache = new LRUCache(2);
// cache.put(1, 1);
// cache.put(2, 2);
// cache.get(1);    // 回傳 1，1 變成最近使用
// cache.put(3, 3); // 容量超過，移除最久未使用的 key=2
// cache.get(2);    // 回傳 -1（已被移除）
```

**原理**：HashMap 負責 O(1) 查找，雙向串列負責 O(1) 維護使用順序。這是「用兩種資料結構互補彼此弱點」的經典範例。

</details>
