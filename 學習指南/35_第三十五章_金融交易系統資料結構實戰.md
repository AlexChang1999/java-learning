## 一、金融系統的核心挑戰

```
普通系統的要求：「能跑就好」
金融系統的要求：
  ✅ 延遲：撮合引擎 p99 < 1ms（高頻交易甚至要 < 1μs）
  ✅ 正確性：一分錢都不能錯（銀行對帳）
  ✅ 吞吐量：交易所每秒百萬筆訂單
  ✅ 可稽核性：所有操作都要有記錄（合規要求）

這四個要求決定了資料結構的選型。
```

## 二、訂單簿（Order Book）— 交易所的心臟

### 訂單簿是什麼？

```
訂單簿 = 買賣雙方所有未成交訂單的集合

BTC/USD 訂單簿（某時刻）：

ASKS（賣方，升序——最低賣價最優先）：
  $50,005  │ 2.0 BTC
  $50,003  │ 0.5 BTC
  $50,001  │ 1.2 BTC  ← Best Ask（最低賣價）
  ─────────────────── Spread（買賣價差）
  $50,000  │ 3.0 BTC  ← Best Bid（最高買價）
  $49,998  │ 1.5 BTC
  $49,995  │ 5.0 BTC
BIDS（買方，降序——最高買價最優先）

當 Best Bid >= Best Ask 時 → 撮合成交！
```

### 為什麼用 TreeMap（紅黑樹）？

```
需求分析：
  ✅ 取最高買價（firstKey on bids）→ 需要有序
  ✅ 取最低賣價（firstKey on asks）→ 需要有序
  ✅ 取消任意訂單（按訂單ID刪除）→ 需要 O(log n) 任意刪除
  ✅ 顯示盤口深度（前5檔）→ 需要範圍查詢

PriorityQueue 為什麼不行？
  只支援 O(1) 取最值，刪除任意節點是 O(n)
  每次撤單都是 O(n) → 高頻撤單場景災難

HashMap 為什麼不行？
  O(1) 查找，但無序，無法取「最高/最低」
```

### 完整訂單簿實作

```java
import java.util.*;
import java.math.BigDecimal;

/**
 * 生產級訂單簿
 * 注意：使用 BigDecimal 而非 double 避免浮點精度問題
 * 例如：0.1 + 0.2 ≠ 0.3 in double！金融系統不能容忍這個
 */
public class OrderBook {

    // 買方訂單：價格降序（最高買價在最前面）
    private final TreeMap<BigDecimal, LinkedList<Order>> bids =
        new TreeMap<>(Comparator.reverseOrder());

    // 賣方訂單：價格升序（最低賣價在最前面）
    private final TreeMap<BigDecimal, LinkedList<Order>> asks =
        new TreeMap<>();

    // 訂單 ID → 訂單：O(1) 按 ID 查詢（用於撤單）
    private final HashMap<String, Order> orderIndex = new HashMap<>();

    // 訂單 ID → 所在價格（撤單時知道去哪個佇列刪）
    private final HashMap<String, BigDecimal> priceIndex = new HashMap<>();

    /** 加入訂單，O(log n) */
    public void addOrder(Order order) {
        TreeMap<BigDecimal, LinkedList<Order>> book =
            order.isBuy() ? bids : asks;

        book.computeIfAbsent(order.getPrice(), k -> new LinkedList<>())
            .addLast(order);   // FIFO：同價格先到先撮合

        orderIndex.put(order.getId(), order);
        priceIndex.put(order.getId(), order.getPrice());
    }

    /** 撤銷訂單，O(log n) */
    public boolean cancelOrder(String orderId) {
        Order order = orderIndex.remove(orderId);
        if (order == null) return false;

        BigDecimal price = priceIndex.remove(orderId);
        TreeMap<BigDecimal, LinkedList<Order>> book =
            order.isBuy() ? bids : asks;

        LinkedList<Order> queue = book.get(price);
        if (queue != null) {
            queue.remove(order);            // O(n) 在 LinkedList 中刪除
            if (queue.isEmpty()) book.remove(price);  // 該價格沒有訂單了
        }
        return true;
    }

    /** 撮合：嘗試讓最優買賣單成交，O(log n) */
    public List<Trade> match() {
        List<Trade> trades = new ArrayList<>();

        while (!bids.isEmpty() && !asks.isEmpty()) {
            BigDecimal bestBid = bids.firstKey();
            BigDecimal bestAsk = asks.firstKey();

            if (bestBid.compareTo(bestAsk) < 0) break;  // 無法成交

            Order buyOrder  = bids.get(bestBid).peekFirst();
            Order sellOrder = asks.get(bestAsk).peekFirst();

            // 成交量 = min(買單數量, 賣單數量)
            BigDecimal matchQty = buyOrder.getRemainQty()
                .min(sellOrder.getRemainQty());

            // 成交價：通常以先到的一方為準（Price Priority）
            BigDecimal tradePrice = buyOrder.getTimestamp() < sellOrder.getTimestamp()
                ? bestBid : bestAsk;

            trades.add(new Trade(buyOrder.getId(), sellOrder.getId(),
                                 tradePrice, matchQty));

            // 更新剩餘數量
            buyOrder.fillQty(matchQty);
            sellOrder.fillQty(matchQty);

            // 完全成交則移除
            if (buyOrder.getRemainQty().compareTo(BigDecimal.ZERO) == 0) {
                bids.get(bestBid).pollFirst();
                if (bids.get(bestBid).isEmpty()) bids.remove(bestBid);
                orderIndex.remove(buyOrder.getId());
                priceIndex.remove(buyOrder.getId());
            }
            if (sellOrder.getRemainQty().compareTo(BigDecimal.ZERO) == 0) {
                asks.get(bestAsk).pollFirst();
                if (asks.get(bestAsk).isEmpty()) asks.remove(bestAsk);
                orderIndex.remove(sellOrder.getId());
                priceIndex.remove(sellOrder.getId());
            }
        }
        return trades;
    }

    /** 取得盤口深度（前 N 檔），O(N log N) */
    public OrderBookSnapshot getSnapshot(int depth) {
        List<PriceLevel> bidLevels = new ArrayList<>();
        List<PriceLevel> askLevels = new ArrayList<>();

        int count = 0;
        for (Map.Entry<BigDecimal, LinkedList<Order>> entry : bids.entrySet()) {
            if (count++ >= depth) break;
            BigDecimal totalQty = entry.getValue().stream()
                .map(Order::getRemainQty)
                .reduce(BigDecimal.ZERO, BigDecimal::add);
            bidLevels.add(new PriceLevel(entry.getKey(), totalQty));
        }

        count = 0;
        for (Map.Entry<BigDecimal, LinkedList<Order>> entry : asks.entrySet()) {
            if (count++ >= depth) break;
            BigDecimal totalQty = entry.getValue().stream()
                .map(Order::getRemainQty)
                .reduce(BigDecimal.ZERO, BigDecimal::add);
            askLevels.add(new PriceLevel(entry.getKey(), totalQty));
        }

        return new OrderBookSnapshot(bidLevels, askLevels);
    }
}
```

## 三、風控系統 — Bloom Filter + 滑動視窗 <!-- 💡 進階 -->

### 場景：防止重複訂單提交

```
問題：交易所每秒數十萬筆訂單，需要：
  1. 防止同一筆訂單被提交兩次（冪等性）
  2. 防止用戶在短時間內下太多訂單（限流）

方案：
  Bloom Filter → 快速判斷訂單 ID 是否已存在
  Sliding Window Counter（滑動視窗計數器）→ 限流
```

```java
@Service
public class RiskControlService {

    // Bloom Filter：預計 1 億筆訂單，0.01% 誤判率
    private final BloomFilter<String> processedOrders = BloomFilter.create(
        Funnels.stringFunnel(StandardCharsets.UTF_8),
        100_000_000L,
        0.0001
    );

    // 滑動視窗：每個用戶每秒最多 100 筆訂單
    // key = userId, value = 滑動視窗（含時間戳記的佇列）
    private final Map<String, Deque<Long>> userOrderWindows = new ConcurrentHashMap<>();

    /**
     * 風控檢查：O(1) 操作
     */
    public RiskResult check(Order order) {
        // 1. 重複訂單檢查
        if (processedOrders.mightContain(order.getId())) {
            // Bloom Filter 說「可能存在」，再去 Redis 確認（避免 0.01% 誤判）
            if (redisService.exists("order:" + order.getId())) {
                return RiskResult.DUPLICATE_ORDER;
            }
        }

        // 2. 限流：滑動視窗
        if (!checkRateLimit(order.getUserId())) {
            return RiskResult.RATE_LIMIT_EXCEEDED;
        }

        // 3. 通過風控，記錄訂單
        processedOrders.put(order.getId());
        return RiskResult.PASS;
    }

    private boolean checkRateLimit(String userId) {
        long now = System.currentTimeMillis();
        long windowMs = 1000L;  // 1 秒視窗
        int maxOrders = 100;    // 每秒最多 100 筆

        Deque<Long> window = userOrderWindows.computeIfAbsent(
            userId, k -> new ArrayDeque<>());

        synchronized (window) {
            // 移除視窗外的舊記錄
            while (!window.isEmpty() && window.peekFirst() < now - windowMs) {
                window.pollFirst();
            }
            if (window.size() >= maxOrders) return false;  // 超過限制
            window.addLast(now);
            return true;
        }
    }
}
```

## 四、高頻交易（HFT）的極致優化 <!-- 🔴 資深 -->

### 問題：為什麼 TreeMap 還不夠快？

```
高頻交易（HFT）的延遲要求：< 1μs（微秒！）

TreeMap（紅黑樹）的問題：
  1. 每個節點是獨立的 Java 物件 → GC 壓力
  2. 樹遍歷時指標跳躍 → Cache Miss（~100ns × 幾十次 = μs 級）
  3. JVM GC Pause（Stop-the-World）→ 偶爾 ms 級停頓 → 不可接受

HFT 的解法：
  1. 用「價格陣列」替代 TreeMap（假設價格範圍有限）
  2. 預先分配所有物件，避免 GC
  3. 使用 Off-Heap 記憶體（不受 GC 管理）
```

### 陣列訂單簿（Array-based Order Book）

```java
/**
 * 超低延遲訂單簿：用陣列索引代替 TreeMap
 * 假設：股票價格範圍 0.01~999.99，最小價格單位 0.01（tick size）
 * 則最多 99999 個價格等級 → 一個 long[] 陣列就夠了！
 *
 * 效能：所有操作 O(1)（陣列索引），Cache 友好，無 GC
 */
public class ArrayOrderBook {
    private static final int TICK_SIZE_CENTS = 1;   // 最小變動單位：1 分
    private static final int MAX_PRICE_CENTS = 9_999_999;  // 最高價：99999.99

    // 每個價格對應的總掛單量（正數=買方，負數=賣方）
    private final long[] bidQty = new long[MAX_PRICE_CENTS + 1];
    private final long[] askQty = new long[MAX_PRICE_CENTS + 1];

    // 維護最優買賣價的指標（避免每次掃描）
    private volatile int bestBidPrice = 0;
    private volatile int bestAskPrice = MAX_PRICE_CENTS;

    /** O(1) 插入 */
    public void addBid(int priceCents, long quantity) {
        bidQty[priceCents] += quantity;
        if (priceCents > bestBidPrice) bestBidPrice = priceCents;
    }

    /** O(1) 取最高買價 */
    public int getBestBid() {
        // 往下掃到有量的價格（通常只掃幾格，接近 O(1)）
        while (bestBidPrice > 0 && bidQty[bestBidPrice] == 0) bestBidPrice--;
        return bestBidPrice;
    }

    // 記憶體佔用：2 × 10M × 8 bytes = 160MB（可接受，換來的是 ns 級延遲）
}
```

### 延遲比較

```
操作：插入訂單 + 取最優報價

TreeMap（Java 物件）：
  插入：~200-500ns（含 Object 建立、GC）
  取最優：~50-100ns

陣列訂單簿：
  插入：~5-10ns（純陣列寫入，Cache Friendly）
  取最優：~2-5ns（直接陣列索引）

差距：~50-100 倍
```

## 五、清算與結算 — BigDecimal 與精度 <!-- 💡 進階 -->

### 為什麼絕對不能用 double 算錢？

```java
// double 的浮點誤差
System.out.println(0.1 + 0.2);          // 0.30000000000000004（不是 0.3！）
System.out.println(1_000_000 * 0.1);    // 99999.99999999999（不是 100000！）

// 每天結算 100 萬筆交易，每筆誤差 0.0001 元
// 日積月累：100萬 × 0.0001 = 100元/天，一年 = 36500 元
// 金融監管單位：這是財務造假！

// 正確做法：BigDecimal
BigDecimal price = new BigDecimal("50000.00");   // 用字串建立，不用 double
BigDecimal qty   = new BigDecimal("1.5");
BigDecimal total = price.multiply(qty);           // 75000.00，精確
```

### 定點數（Fixed-Point）— 更高效的方案

```java
/**
 * 實務上 BigDecimal 太慢（GC + 物件建立）
 * 高效做法：用 long 儲存，約定最後 N 位是小數
 *
 * 例如：$50,000.12 存為 5_000_012（放大 100 倍，單位：分）
 *       $0.001 存為 1（最小單位：0.001 元）
 */
public class FixedPoint {

    private static final long SCALE = 1_000_000L;  // 小數後 6 位

    public static long fromString(String s) {
        return new BigDecimal(s).multiply(BigDecimal.valueOf(SCALE))
                .longValueExact();  // 若有精度損失則拋例外
    }

    public static long multiply(long a, long b) {
        // 注意：兩個 long 相乘可能溢出，用 Math.multiplyExact
        return Math.multiplyExact(a, b) / SCALE;
    }

    public static String toString(long value) {
        return BigDecimal.valueOf(value).divide(BigDecimal.valueOf(SCALE))
                .toPlainString();
    }
}

// 使用：
long price = FixedPoint.fromString("50000.123456");  // 50_000_123_456
long qty   = FixedPoint.fromString("1.500000");      // 1_500_000
long total = FixedPoint.multiply(price, qty);         // 75_000_185_184
// toString(total) → "75000.185184"
```

## 六、WAL（Write-Ahead Log）— 崩潰恢復 <!-- 🔴 資深 -->

### 場景：撮合引擎如果中途崩潰怎麼辦？

```
問題：撮合引擎做到一半當機，重啟後怎麼知道哪些訂單已撮合？

解法：WAL（Write-Ahead Log）
  核心原則：「先寫日誌，再做操作」
  即使崩潰，重啟後重放日誌可以還原狀態

類比：記帳本
  不管有沒有空腹，先在本子上記「晚上吃了 100 元的飯」
  就算忘了，翻帳本可以回想

WAL 的資料結構：Append-Only Log（只追加，不修改）
  這正是 LSM-Tree 的思想！
```

```java
@Service
public class OrderBookWAL {

    private static final String WAL_FILE = "/data/orderbook.wal";

    /**
     * 每個操作先寫 WAL，再執行
     * WAL entry format：[timestamp][operation][orderId][data]
     */
    public void writeAheadLog(WalEntry entry) {
        // FileChannel + MappedByteBuffer：OS 負責定期 flush，延遲最低
        // 或 fileChannel.force(true) 強制 fsync（延遲高但持久化保證強）
        byte[] bytes = entry.serialize();
        walChannel.write(ByteBuffer.wrap(bytes));
        // 不需要立即 fsync，OS Page Cache 會定期刷盤
        // 若需要強一致性：walChannel.force(false)
    }

    /** 系統重啟後，重放 WAL 恢復狀態 */
    public void replay() {
        try (FileInputStream fis = new FileInputStream(WAL_FILE)) {
            WalEntry entry;
            while ((entry = readNextEntry(fis)) != null) {
                switch (entry.getOperation()) {
                    case ADD_ORDER    -> orderBook.addOrder(entry.getOrder());
                    case CANCEL_ORDER -> orderBook.cancelOrder(entry.getOrderId());
                    case MATCH        -> orderBook.applyTrade(entry.getTrade());
                }
            }
        }
    }
}
```

## 七、本章總結

| 場景 | 資料結構 | 理由 |
|------|----------|------|
| 訂單簿按價格排序 | TreeMap（紅黑樹） | 有序 + 任意刪除 O(log n) |
| 訂單 ID 查詢/撤單 | HashMap | O(1) 查詢 |
| 防重複訂單 | Bloom Filter | 極小記憶體，「一定不存在」快速排除 |
| 高頻限流 | 滑動視窗 + Deque | O(1) 操作 |
| 超低延遲（HFT） | 陣列訂單簿 | O(1)，Cache Friendly，無 GC |
| 金額計算 | BigDecimal / long 定點數 | 避免浮點誤差 |
| 崩潰恢復 | WAL（Append-Only Log） | 先寫日誌再操作，可回放 |

**金融系統的核心哲學**：
```
正確性 > 效能 > 功能
  先保證錢算對，再考慮快不快，最後才考慮功能多不多
```
