## 一、作業系統排程器 — 紅黑樹

### Linux CFS（完全公平排程器）

```
問題：Linux 同時跑著幾百個進程，CPU 只有幾個核心
      如何決定「下一個讓誰跑」？

Linux CFS（Completely Fair Scheduler）的設計：
  目標：每個進程都應獲得「公平」的 CPU 時間
  衡量標準：vruntime（虛擬運行時間）
    vruntime 越小 → 被剝奪 CPU 時間越多 → 應該先跑

資料結構：
  所有可運行的進程組成一棵「以 vruntime 為 key 的紅黑樹」
  每次排程：取 vruntime 最小的進程（樹的最左節點）

為什麼用紅黑樹？
  ✅ O(1) 取最小 vruntime（最左節點，直接存在 rb_leftmost 指標）
  ✅ O(log n) 插入（進程進入就緒狀態）
  ✅ O(log n) 刪除（進程讓出 CPU 或被阻塞）
  ✅ n = 幾百個進程，log n ≈ 8-9，幾乎是常數
```

```java
// Java 的 TreeMap 就是紅黑樹，可以模擬 CFS 排程概念：

class CfsScheduler {

    // key = vruntime，value = 進程列表（相同 vruntime 的進程）
    TreeMap<Long, Queue<Process>> runQueue = new TreeMap<>();

    // 讓進程進入就緒狀態（加入紅黑樹）
    void enqueue(Process p) {
        runQueue.computeIfAbsent(p.getVruntime(), k -> new LinkedList<>())
                .offer(p);
    }

    // 取下一個要運行的進程（vruntime 最小）
    Process pickNext() {
        if (runQueue.isEmpty()) return null;
        Map.Entry<Long, Queue<Process>> entry = runQueue.firstEntry();  // 最左節點
        Process p = entry.getValue().poll();
        if (entry.getValue().isEmpty()) runQueue.pollFirstEntry();
        return p;
    }

    // 進程運行一段時間後，更新 vruntime 並重新入隊
    void tick(Process p, long deltaTime) {
        // vruntime 增量 = 實際運行時間 × (nice_value_weight / weight)
        // 低優先級進程的 vruntime 增長更快 → 更快被其他進程搶佔
        p.addVruntime(deltaTime);
        enqueue(p);
    }
}
```

### CPU 上下文切換的代價（硬體視角）

```
進程切換的成本：
  1. 保存 CPU 暫存器（~30 個 64-bit 暫存器）到 PCB：~50ns
  2. 切換記憶體映射（TLB flush，如果跨進程）：~1000ns
  3. Cold Cache（新進程的資料不在 L1/L2）：~數 ms 重新暖機

上下文切換總成本：5-30μs（第十二章提到的數字）

這就是為什麼 1 個進程開 10,000 個 Thread 很糟：
  10,000 個 Thread × 5-30μs 切換 = 大量 CPU 浪費在「切換」而非「工作」
  → 這正是 Netty、WebFlux 用 Event Loop（少量 Thread 處理大量連線）的原因
```

## 二、資料庫 Join 算法 — Hash Join vs Sort-Merge Join <!-- 🔴 資深 -->

### 三種 Join 算法

```sql
-- 假設：orders(1,000,000 行) JOIN users(100,000 行) ON orders.user_id = users.id

-- 演算法選擇由資料庫查詢優化器（Query Planner）決定
EXPLAIN SELECT * FROM orders o JOIN users u ON o.user_id = u.id;
```

#### 算法一：Nested Loop Join（巢狀迴圈）

```
for each row in orders:              ← 外層：1,000,000 次
    for each row in users:           ← 內層：100,000 次
        if orders.user_id == users.id:
            output

時間複雜度：O(N × M) = O(10^11) → 極慢，不適合大表

適用場景：外層表很小（< 幾百行），或內層表有索引（退化成 Index Nested Loop）
```

#### 算法二：Hash Join <!-- 🔴 資深 -->

```
步驟：
  1. Build Phase：把「較小的表」（users）載入記憶體，建 HashMap
     HashMap<user_id, UserRow>  → O(M) 空間和時間

  2. Probe Phase：掃描「較大的表」（orders），
     對每行用 HashMap 查 user_id → O(N) 時間

總時間：O(N + M)  ← 遠快於 O(N×M)！

問題：若較小的表放不進記憶體（> work_mem）？
  → Grace Hash Join：把兩個表都分成多個 Bucket，分批處理

Java 模擬：
HashMap<Long, User> userMap = users.stream()
    .collect(Collectors.toMap(User::getId, u -> u));

List<Result> results = orders.stream()
    .filter(o -> userMap.containsKey(o.getUserId()))
    .map(o -> new Result(o, userMap.get(o.getUserId())))
    .collect(Collectors.toList());
```

#### 算法三：Sort-Merge Join

```
步驟：
  1. 把兩個表都按 join key 排序（若有索引則省略）
  2. 用雙指標合併（類似 Merge Sort 的 Merge 步驟）

排好序後的合併：
  orders: [1001, 1001, 1002, 1003, ...]
  users:  [1001,       1002, 1003, ...]
  雙指標同步前進 → O(N + M)

適用場景：
  ✅ Join key 上已有索引（省略排序步驟）
  ✅ 需要排序的 ORDER BY 和 GROUP BY（排序可以複用）
  ✅ 記憶體不夠做 Hash Join 時（Sort-Merge 可以 spillover 到磁碟）
```

## 三、Git — DAG（有向無環圖） <!-- 💡 進階 -->

### Git 的物件模型

```
Git 中的一切都是「物件」，物件之間形成有向無環圖（DAG）：

Commit 物件：
  ┌──────────────────────────────┐
  │ commit abc123                │
  │ tree: def456                 │  ← 指向這次提交的目錄快照
  │ parent: 789xyz               │  ← 指向上一次提交（父節點）
  │ author: Alex                 │
  │ message: "feat: add order"   │
  └──────────────────────────────┘

Tree 物件：
  ┌──────────────────────────────┐
  │ tree def456                  │
  │ blob aaa111  src/Main.java   │  ← 指向文件快照
  │ blob bbb222  src/Order.java  │
  │ tree ccc333  test/           │  ← 指向子目錄
  └──────────────────────────────┘

Blob 物件：
  文件的二進制內容（不存文件名，文件名在 Tree 中）

SHA-1/SHA-256 Hash：
  每個物件的 ID = 其內容的 hash
  → 只要內容相同，hash 一定相同（不同 Commit 中相同的文件只存一次！）
  → hash 不同，內容一定不同（可靠的去重機制）
```

### Git Merge 的本質：DAG 合併

```
merge 前：
  main:    A → B → C
                    ↑ HEAD
  feature:     B → D → E

merge 後（merge commit）：
  main:    A → B → C → F（merge commit）
                    ↑       ↑
  feature:     B → D → E ──┘

DAG 特性：
  F 有兩個父節點（C 和 E）
  git log --graph 能顯示這個分叉合流結構
  git log 用 DFS 遍歷這個 DAG

rebase 的本質：
  把 feature 分支的 commit（D, E）重新「嫁接」到 C 之後
  D' = 相同修改，但父節點從 B 改為 C，所以 hash 不同
```

```java
// Git 物件儲存的 Java 概念模擬：
public class GitObjectStore {

    // SHA → 物件內容（Content-Addressable Storage）
    private final Map<String, byte[]> store = new HashMap<>();

    /** 儲存物件，返回 SHA hash */
    public String store(byte[] content) {
        String sha = sha256(content);       // 內容的 hash = 物件的 ID
        store.put(sha, content);             // 相同內容只存一次（天然去重）
        return sha;
    }

    /** 用 SHA 取物件 */
    public byte[] load(String sha) {
        return store.get(sha);
    }
}

// 這就是為什麼 Git 非常高效：
// 同一個大文件在 100 個 Commit 中都沒有修改，只存了 1 份！
```

## 四、瀏覽器 DOM — N 叉樹 <!-- 入門 -->

### DOM 的樹狀結構

```html
<html>
  <head><title>My App</title></head>
  <body>
    <div id="app">
      <h1>Hello</h1>
      <ul>
        <li>Item 1</li>
        <li>Item 2</li>
      </ul>
    </div>
  </body>
</html>
```

```
對應的 N 叉樹：
  html
  ├── head
  │   └── title: "My App"
  └── body
      └── div#app
          ├── h1: "Hello"
          └── ul
              ├── li: "Item 1"
              └── li: "Item 2"
```

### React Virtual DOM — Diff 算法

```
問題：當資料更新，如何高效地只更新變化的 DOM 節點？

暴力做法：刪除整個 DOM，重新建立 → O(n)，而且瀏覽器重排（Reflow）很貴
React 的做法：Virtual DOM Diff

步驟：
  1. 資料變化 → 產生新的 Virtual DOM 樹（輕量 JS 物件，不是真的 DOM）
  2. 比較新舊 Virtual DOM 樹（Tree Diff）
  3. 只把「差異」應用到真實 DOM

Diff 算法的鍵：
  同層比較（同一層級的節點才比較，不跨層級）
  key 屬性：告訴 React 哪些節點是「相同的節點移動了位置」

如果沒有 key（<li>）：
  舊: [Apple, Banana, Cherry]
  新: [Avocado, Apple, Banana, Cherry]
  React 以為三個節點都變了 → 更新 3 次（錯誤！）

有 key（<li key={item.id}>）：
  React 知道 Apple/Banana/Cherry 沒有變 → 只插入 Avocado（正確！）
```

## 五、JVM 垃圾回收 — 各種圖結構 <!-- 🔴 資深 -->

### 物件引用圖

```
JVM 堆中的物件形成一個有向圖：
  A → B → D
  A → C
  E → F（孤島，不可達）

GC Root（根節點）：
  - 棧幀中的局部變數
  - 靜態欄位
  - JNI 全局引用

可達性分析（Mark 階段）：
  從 GC Root 出發，DFS/BFS 遍歷整個物件圖
  可以到達的物件 = 存活
  不可到達的物件（E, F）= 垃圾，可以回收
```

### G1 GC 的 Remember Set

```
問題：G1 GC 把 Heap 分成多個 Region，每次只回收部分 Region
      如果舊 Region 中的物件引用了待回收 Region 中的物件，怎麼辦？

Remember Set（記憶集）：
  每個 Region 維護一個「其他 Region 中有誰引用了我」的紀錄
  類似：倒排索引！

  傳統正向索引：「物件A 引用了哪些物件」→ 遍歷物件的欄位
  Remember Set（倒排）：「誰引用了這個 Region 中的物件」→ O(1) 查詢

回收時：
  只需掃描待回收 Region + 它的 Remember Set 中記錄的外部引用
  不需要掃描整個 Heap！

硬體視角：
  Remember Set 存在 Native Memory（不受 GC 管理，不會被自己回收）
  G1 的「Card Table」：把 Heap 分成 512 Byte 的 Card，
  dirty card = 最近有寫入操作，可能有跨 Region 引用
  寫入屏障（Write Barrier）：每次物件欄位賦值時，標記 dirty card（~5ns 開銷）
```

## 六、網路封包路由 — 最長前綴匹配（Trie） <!-- 🔴 資深 -->

### IP 路由表

```
路由器的路由表（簡化）：
  10.0.0.0/8      → 介面 eth0
  10.20.0.0/16    → 介面 eth1
  10.20.30.0/24   → 介面 eth2
  0.0.0.0/0       → 預設閘道（default gateway）

封包目標：10.20.30.5
  匹配 10.0.0.0/8    ✓（/8 匹配前 8 bit）
  匹配 10.20.0.0/16  ✓（/16 匹配前 16 bit）
  匹配 10.20.30.0/24 ✓（/24 匹配前 24 bit）← 最長前綴，選這個！
  匹配 0.0.0.0/0     ✓（/0 什麼都匹配）

原則：最長前綴優先（Longest Prefix Match）
  前綴越長，路由越精確
```

### 用 Trie 實作路由查找

```java
/**
 * IP 路由表（二進制 Trie）
 * 每個 bit 是 0 或 1，所以是二叉 Trie（Binary Trie）
 */
public class IpRoutingTable {

    static class TrieNode {
        TrieNode[] children = new TrieNode[2];  // 0 或 1
        String nextHop = null;  // 這個前綴對應的下一跳（null = 中間節點）
    }

    private final TrieNode root = new TrieNode();

    /** 加入路由：prefix = "10.20.30.0", prefixLen = 24 */
    public void addRoute(String prefix, int prefixLen, String nextHop) {
        int ip = ipToInt(prefix);
        TrieNode node = root;

        for (int i = 31; i >= 32 - prefixLen; i--) {
            int bit = (ip >> i) & 1;
            if (node.children[bit] == null) {
                node.children[bit] = new TrieNode();
            }
            node = node.children[bit];
        }
        node.nextHop = nextHop;  // 標記這個前綴的下一跳
    }

    /** 查找：最長前綴匹配 */
    public String lookup(String destIp) {
        int ip = ipToInt(destIp);
        TrieNode node = root;
        String bestMatch = null;  // 記錄最長匹配的下一跳

        for (int i = 31; i >= 0; i--) {
            int bit = (ip >> i) & 1;
            if (node.children[bit] == null) break;
            node = node.children[bit];
            if (node.nextHop != null) {
                bestMatch = node.nextHop;  // 每次找到更長的前綴就更新
            }
        }
        return bestMatch;  // 最長匹配的下一跳
    }

    private int ipToInt(String ip) {
        String[] parts = ip.split("\\.");
        int result = 0;
        for (String part : parts) {
            result = (result << 8) | Integer.parseInt(part);
        }
        return result;
    }
}

// 使用：
IpRoutingTable table = new IpRoutingTable();
table.addRoute("0.0.0.0",   0,  "gateway0");  // 預設路由
table.addRoute("10.0.0.0",  8,  "eth0");
table.addRoute("10.20.0.0", 16, "eth1");
table.addRoute("10.20.30.0",24, "eth2");

System.out.println(table.lookup("10.20.30.5"));  // eth2（/24 最長）
System.out.println(table.lookup("10.20.99.1"));  // eth1（/16 最長）
System.out.println(table.lookup("172.16.0.1"));  // gateway0（只有預設路由匹配）
```

## 八、架構師的思維跳躍 <!-- 🔴 資深 -->

```
初級工程師：「這個功能怎麼實作？」
中級工程師：「用哪個庫？哪個 API？」
高級工程師：「這個選擇的性能邊界在哪？什麼時候會出問題？」
架構師：「這個底層資料結構和我在其他系統看到的 X 有相同的模式——
         我能不能把 X 的解法搬過來？」

模式識別練習：
  Linux CFS 用紅黑樹按 vruntime 排序
  → Nginx 定時器也用紅黑樹按「觸發時間」排序
  → Java DelayQueue 也用 Heap 按「到期時間」排序
  → 本質都是：「在一個動態變化的集合中，快速找到『最小值』並支援增刪」

  Git 用 DAG 追蹤 Commit 歷史
  → 微服務依賴圖也是 DAG（用拓撲排序決定啟動順序）
  → Maven 依賴也是 DAG（用拓撲排序決定編譯順序）
  → 電子試算表的公式依賴也是 DAG
  → 本質都是：「有依賴關係的事物，用 DAG 表示，用拓撲排序確定執行順序」

  Redis ZSet 用跳表
  → 撮合引擎訂單簿用 TreeMap（紅黑樹）
  → 都在解決：「有序集合，支援範圍查詢和任意增刪」
  → 差別：跳表並發友好，紅黑樹單線程更快

當你能跨系統識別這些模式，你就在往架構師進化。
```
