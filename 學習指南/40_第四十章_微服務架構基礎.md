# 第四十章：微服務架構基礎

> **前置章節**：第二十一章（Spring Boot）、第二十八章（Kafka）  
> **核心命題**：單體應用長大之後怎麼辦？微服務解決了什麼問題，又帶來了什麼新問題？

---

## 一、單體 vs 微服務

### 單體應用（Monolith）

```
所有功能打包成一個 JAR/WAR：

┌──────────────────────────────────────┐
│  撮合引擎 Monolith                    │
│  ┌──────────┐  ┌──────────┐          │
│  │ 訂單模組  │  │ 行情模組  │          │
│  └──────────┘  └──────────┘          │
│  ┌──────────┐  ┌──────────┐          │
│  │ 用戶模組  │  │ 結算模組  │          │
│  └──────────┘  └──────────┘          │
│         一個 DB，一個部署              │
└──────────────────────────────────────┘

優點：
  ✅ 開發簡單（一個 IDE，一個 project）
  ✅ 部署簡單（一個 JAR）
  ✅ Debug 容易（本地直接跑）

缺點：
  ❌ 一個模組有 bug，整個系統要重新部署
  ❌ 團隊擴大後，改一行程式碼要 merge 幾十個人的衝突
  ❌ 訂單模組流量爆了，但結算模組也要跟著擴容（浪費資源）
  ❌ 「訂單模組要用 Java 21，結算模組習慣 Java 11」→ 做不到
```

### 微服務（Microservices）

```
每個業務能力獨立部署：

  ┌─────────────┐   ┌─────────────┐
  │  訂單服務    │   │  行情服務    │
  │  Port: 8081 │   │  Port: 8082 │
  └──────┬──────┘   └──────┬──────┘
         │                  │
  ┌──────┴──────┐   ┌──────┴──────┐
  │  用戶服務    │   │  結算服務    │
  │  Port: 8083 │   │  Port: 8084 │
  └─────────────┘   └─────────────┘
         │                  │
  各自獨立 DB         各自獨立部署

優點：
  ✅ 獨立部署：訂單服務更新，其他服務不受影響
  ✅ 獨立擴容：行情服務流量高 → 只擴行情服務的 Pod
  ✅ 技術異構：結算服務可以用 Python，行情服務用 Java
  ✅ 團隊自治：訂單團隊和結算團隊可以完全獨立

缺點：
  ❌ 服務間通信複雜（本地呼叫 → 網路呼叫，有延遲和失敗）
  ❌ 分散式事務難（一個操作跨多個服務的 DB）
  ❌ 除錯困難（一個請求可能跨 5 個服務）
  ❌ 維運複雜（50 個服務各自的日誌、監控、部署）
```

### 什麼時候該拆微服務？

```
不成熟的理由（先別拆）：
  ❌ 「聽說微服務很厲害」
  ❌ 「我想學 Spring Cloud」
  ❌ 團隊只有 3 個人

成熟的理由（可以拆）：
  ✅ 不同模組的部署頻率差異很大
      （行情服務每天更新，結算服務每月更新）
  ✅ 不同模組的擴容需求差異很大
      （撮合引擎要 32 核，用戶服務只要 2 核）
  ✅ 團隊已經超過 20 人，合併衝突成本超過拆分成本
  ✅ 監管要求不同模組不同隔離等級
      （用戶資料不能和行情資料在同一個服務）
```

## 二、服務拆分原則 — DDD 界限上下文 <!-- 💡 進階 -->

### 錯誤的拆分方式

```
按技術層拆（❌ 反模式）：
  ├── controller-service（所有 Controller）
  ├── business-service（所有 Service）
  └── data-service（所有 Repository）

問題：一個訂單功能，要改 3 個服務 + 協調 3 個團隊 → 比單體更慢！
```

### 正確的拆分方式：領域邊界（Domain Boundary）

```
按業務能力拆（✅ 正確方向）：

交易所系統的領域分析：

┌──────────────────────────────────────────────────┐
│  界限上下文 1：行情（Market Data）                  │
│  - 接收交易所行情推送                               │
│  - 訂單簿快照                                       │
│  - 用戶訂閱行情                                     │
├──────────────────────────────────────────────────┤
│  界限上下文 2：訂單（Order）                         │
│  - 用戶下單/撤單                                    │
│  - 訂單狀態追蹤                                     │
│  - 訂單歷史查詢                                     │
├──────────────────────────────────────────────────┤
│  界限上下文 3：撮合（Matching）                      │
│  - 撮合引擎核心                                     │
│  - 成交記錄                                         │
│  - 延遲要求最高（獨立部署在高性能機器）               │
├──────────────────────────────────────────────────┤
│  界限上下文 4：結算（Settlement）                    │
│  - 成交後的資金結算                                  │
│  - 帳戶餘額更新                                     │
│  - 對帳（Reconciliation）                           │
└──────────────────────────────────────────────────┘

辨別界限上下文的信號：
  「同一個詞在不同部門有不同意義」
  例如：「訂單（Order）」
    行情團隊：訂單 = 掛在訂單簿上的報價
    結算團隊：訂單 = 需要結算的成交記錄
    客服團隊：訂單 = 用戶看到的購買記錄
  → 三個不同的 Order，應該是三個不同的服務！
```

## 三、服務間通信 <!-- 💡 進階 -->

### 同步通信（REST / gRPC）

```
適合：需要立即知道結果的操作
  下單 → 立刻得到訂單 ID
  查詢餘額 → 立刻得到數字

REST（第22章）：
  優點：簡單，人人會，工具多
  缺點：文字協定（JSON），序列化開銷大；無嚴格契約

gRPC（Google Remote Procedure Call）：
  優點：二進制協定（Protobuf），比 JSON 快 5-10x；有強型別契約
  缺點：需要生成程式碼；瀏覽器支援較差
  適合：服務間高頻呼叫（例如撮合服務 → 行情服務）

HTTP 延遲（同機房）：
  REST（JSON）：RTT ~0.1ms + 序列化 ~1ms = ~1.1ms
  gRPC（Protobuf）：RTT ~0.1ms + 序列化 ~0.1ms = ~0.2ms
```

```java
// gRPC 服務定義（.proto 檔案）
syntax = "proto3";

service OrderService {
  rpc PlaceOrder (OrderRequest) returns (OrderResponse);
  rpc GetOrderBook (OrderBookRequest) returns (stream OrderBookUpdate);  // 伺服器串流
}

message OrderRequest {
  string user_id = 1;
  string symbol  = 2;
  double price   = 3;
  double quantity = 4;
  bool   is_buy  = 5;
}
```

```java
// Spring Boot 整合 gRPC（使用 grpc-spring-boot-starter）
@GrpcService
public class OrderGrpcService extends OrderServiceGrpc.OrderServiceImplBase {

    @Override
    public void placeOrder(OrderRequest req, StreamObserver<OrderResponse> observer) {
        try {
            Order order = orderService.place(req.getUserId(), req.getSymbol(),
                                             req.getPrice(), req.getQuantity(), req.getIsBuy());
            observer.onNext(OrderResponse.newBuilder()
                .setOrderId(order.getId())
                .setStatus("ACCEPTED")
                .build());
            observer.onCompleted();
        } catch (Exception e) {
            observer.onError(Status.INTERNAL.withDescription(e.getMessage()).asRuntimeException());
        }
    }
}
```

### 非同步通信（Kafka / 事件驅動）

```
適合：不需要立即回應，或需要解耦的操作
  訂單成交 → 觸發結算（可以稍後處理）
  用戶行為 → 觸發風控分析（不需要同步等待）

選擇原則：
  ┌─────────────────────────────────────────────────┐
  │ 需要立即回應？                                    │
  │    是 → 同步（REST / gRPC）                       │
  │    否 → 非同步（Kafka）                           │
  │                                                   │
  │ 呼叫方掛掉後，操作還要繼續嗎？                     │
  │    是 → Kafka（訊息持久化）                        │
  │    否 → REST/gRPC 都可                            │
  │                                                   │
  │ 一個事件要通知多個訂閱者嗎？                        │
  │    是 → Kafka（Pub/Sub）                           │
  │    否 → REST/gRPC 點對點都可                       │
  └─────────────────────────────────────────────────┘
```

## 四、Spring Cloud 核心組件 <!-- 💡 進階 -->

### 整體架構

```
用戶請求
    │
    ▼
┌─────────────────────────────┐
│  API Gateway（Spring Cloud  │  ← 統一入口：路由、限流、認證
│  Gateway）Port 8080          │
└──────────────┬──────────────┘
               │
       ┌───────┴────────┐
       ▼                ▼
┌──────────────┐  ┌──────────────┐
│ 訂單服務      │  │ 行情服務      │  ← 各自向 Nacos 註冊
│ Port 8081    │  │ Port 8082    │
└──────┬───────┘  └──────┬───────┘
       │                  │
       ▼                  ▼
┌─────────────────────────────────┐
│  Nacos（服務發現 + 配置中心）     │  ← 「電話簿」：誰在哪個 IP
└─────────────────────────────────┘
```

### Spring Cloud Gateway — 統一入口

```java
// application.yml：路由配置
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service    # lb = load balance，從 Nacos 取服務位址
          predicates:
            - Path=/api/orders/**
          filters:
            - StripPrefix=1          # 移除 /api 前綴再轉發
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100     # 每秒 100 個請求
                redis-rate-limiter.burstCapacity: 200     # 峰值 200

        - id: market-service
          uri: lb://market-service
          predicates:
            - Path=/api/market/**
```

```java
// 全局 JWT 認證過濾器
@Component
public class AuthFilter implements GlobalFilter {

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String token = exchange.getRequest()
            .getHeaders().getFirst("Authorization");

        if (token == null || !jwtService.validate(token)) {
            exchange.getResponse().setStatusCode(HttpStatus.UNAUTHORIZED);
            return exchange.getResponse().setComplete();
        }

        // 把用戶 ID 往下傳（避免每個下游服務都要解析 JWT）
        String userId = jwtService.getUserId(token);
        ServerWebExchange mutated = exchange.mutate()
            .request(r -> r.header("X-User-Id", userId))
            .build();

        return chain.filter(mutated);
    }
}
```

### Nacos — 服務發現

```
服務發現解決的問題：
  不用微服務時：直接連 localhost:8081
  微服務多個 Pod 時：不知道應該連哪個 IP！
    訂單服務有 3 個 Pod：10.0.0.1:8081, 10.0.0.2:8081, 10.0.0.3:8081
    Pod 可能隨時崩潰、擴縮容

Nacos 的角色：
  ├── 服務 A 啟動 → 向 Nacos 報到「我是訂單服務，IP 是 10.0.0.1:8081」
  ├── 服務 B 要呼叫訂單服務 → 問 Nacos「訂單服務現在有哪些健康的 IP？」
  ├── Nacos 回傳 [10.0.0.1:8081, 10.0.0.2:8081, 10.0.0.3:8081]
  ├── 服務 B 用負載均衡（Round Robin / Random）選一個呼叫
  └── 某個 Pod 崩潰 → Nacos 心跳超時 → 自動從清單移除
```

```yaml
# application.yml：向 Nacos 註冊
spring:
  application:
    name: order-service   # 服務名稱（「電話簿」的名字）
  cloud:
    nacos:
      discovery:
        server-addr: nacos:8848   # Nacos 伺服器地址
      config:
        server-addr: nacos:8848   # 同時用作配置中心
        file-extension: yaml
```

```java
// 用 OpenFeign 呼叫其他服務（像呼叫本地方法一樣）
@FeignClient(name = "settlement-service")  // 服務名稱，自動從 Nacos 取 IP
public interface SettlementClient {

    @PostMapping("/api/settlements")
    SettlementResult settle(@RequestBody SettlementRequest request);
}

// 使用：
@Service
public class OrderService {

    @Autowired
    private SettlementClient settlementClient;  // Feign 自動處理服務發現 + 負載均衡

    public void completeOrder(Trade trade) {
        // 呼叫結算服務（看起來像本地呼叫，實際上是 HTTP 呼叫）
        SettlementResult result = settlementClient.settle(
            new SettlementRequest(trade.getBuyOrderId(), trade.getSellOrderId(),
                                  trade.getPrice(), trade.getQuantity()));
    }
}
```

### Spring Cloud Config — 集中配置管理

```
問題：50 個微服務，每個都有自己的 application.yml
      要改 DB 密碼？要改 Redis 地址？要更新 50 個文件！

解法：所有服務的配置集中存在 Git Repo 或 Nacos
      服務啟動時從配置中心取得設定
      配置更新後，服務不需要重啟（動態刷新）

@RefreshScope：標記 Bean，配置更新時自動重新載入
```

## 五、微服務的挑戰 <!-- 🔴 資深 -->

### 分散式事務問題

```
問題場景：
  用戶下單（訂單服務）→ 扣款（帳戶服務）→ 扣庫存（資產服務）

  這三個步驟分別在三個服務、三個 DB 中！

  中間失敗怎麼辦？
  Step 1 成功、Step 2 失敗 → 訂單建了但錢沒扣？
  Step 1 成功、Step 2 成功、Step 3 失敗 → 訂單建了、錢扣了但庫存沒扣？

解法：Saga 模式（詳見第四十一章）
```

### 服務雪崩

```
問題：A 呼叫 B，B 呼叫 C，C 超時了…

A → B（等 C 的結果）
    B → C（C 超時 30 秒）

B 的所有線程都在等 C → B 也超時 → A 也被拖死
最後三個服務全部崩潰！

這叫「雪崩效應（Cascading Failure）」

解法：熔斷器（Circuit Breaker）（詳見第四十二章）
```

### 微服務除錯難題

```
一個用戶投訴「我的訂單沒有結算」

請求鏈：
  Gateway → 訂單服務 → 撮合服務 → 結算服務 → 通知服務

問題出在哪一個服務？
  沒有工具的話：去看 5 個服務的日誌，手動關聯…

解法：分散式追蹤（Distributed Tracing）（詳見第四十三章）
  每個請求帶一個 Trace ID，全鏈路日誌一起查
```

## 六、微服務 vs 單體的選型建議

```
團隊 < 10 人，剛起步：
  → 用單體，別急著拆
  → 但把程式碼寫成「模組化單體」（Modular Monolith）
    好的包結構：
    com.example
      ├── order/（訂單模組，自己的 Service/Repository）
      ├── market/（行情模組）
      └── settlement/（結算模組）
    未來要拆成微服務，直接把包搬出去

團隊 10~50 人，業務穩定：
  → 按業務能力拆 2-5 個服務
  → 不要過度拆分（不是服務越多越好！）
  → 每個服務至少要有 1-2 個人負責

團隊 50+ 人，業務複雜：
  → 可以考慮完整的微服務 + Spring Cloud
  → 但要同時有 Observability（監控、追蹤、告警）
  → 沒有好的 DevOps，微服務是災難
```

## 七、本章總結

| 概念 | 說明 |
|------|------|
| 微服務 vs 單體 | 微服務解決「擴展性和部署隔離」，但引入「分散式複雜性」 |
| 拆分原則 | 按業務能力（DDD 界限上下文），不按技術層 |
| 同步通信 | REST（簡單）、gRPC（高效，適合服務間高頻呼叫） |
| 非同步通信 | Kafka（解耦，持久化，Pub/Sub） |
| API Gateway | 統一入口，路由 + 限流 + 認證 |
| 服務發現 | Nacos 作「電話簿」，Pod 動態增減不需要硬寫 IP |
| 核心挑戰 | 分散式事務、服務雪崩、追蹤困難（後三章各自深入） |
