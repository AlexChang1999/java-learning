## 一、為什麼分散式系統很難？

```
單機系統的假設（都是真的）：
  ✅ 函數呼叫永遠成功（或拋例外）
  ✅ 記憶體讀寫是即時的
  ✅ 時鐘是準確的
  ✅ 操作要麼完全成功，要麼完全失敗

分散式系統打破了所有假設：
  ❌ 網路呼叫可能：成功 / 失敗 / 超時（不知道對方收到了沒！）
  ❌ 不同機器的時鐘可能差幾毫秒（NTP 同步也有誤差）
  ❌ 操作可能部分成功（錢扣了但訂單沒建）
  ❌ 同一個問題，不同節點看到的答案可能不同

「任何分散式系統都面臨這八個謬誤（Fallacies of Distributed Computing）：
  1. 網路是可靠的
  2. 延遲為零
  3. 頻寬是無限的
  4. 網路是安全的
  5. 拓撲不會改變
  6. 只有一個管理員
  7. 傳輸成本為零
  8. 網路是同質的
  — Peter Deutsch, L Peter Deutsch, 1994」
```

## 二、CAP 定理 <!-- 🔴 資深 -->

### 三個屬性

```
C（Consistency，一致性）：
  所有節點在同一時刻看到的資料是相同的
  「你在台北讀到餘額 1000 元，我在高雄也讀到 1000 元」

A（Availability，可用性）：
  每個請求都能收到回應（不管是成功還是失敗），不會一直等待
  「就算部分節點掛掉，系統仍然可以服務請求」

P（Partition Tolerance，分區容忍性）：
  當節點間的網路通信失敗（網路分區），系統仍然能運作
  「台北機房和高雄機房的網路斷了，兩邊還是能繼續服務」
```

### CAP 定理：三選二（P 幾乎不可避免）

```
現實中，網路故障（Partition）是無法避免的
  台北機房 → 高雄機房的網路線被挖斷了
  這是你無法控制的，必須容忍 Partition（P 是必選）

所以問題變成：C vs A，當網路分區時，你選哪個？

選 CP（犧牲 A）：
  網路分區時，拒絕服務（回傳 503）直到資料同步完成
  「寧可讓你等，也不給你錯誤的資料」
  例子：銀行轉帳（不能因為網路問題就允許超額提款）

選 AP（犧牲 C）：
  網路分區時，繼續服務，但可能回傳舊資料（最終一致）
  「先給你一個答案，可能不是最新的，但系統不會停」
  例子：Facebook 按讚（稍微延遲看到也沒關係）

可視化：
┌──────────────────────────────┐
│  C ──────────────── P        │  CA：理論上存在，但現實中
│  │         ╲                  │      網路不可能不分區
│  │          ╲                 │
│  │     CP    ╲  AP            │  CP：ZooKeeper, HBase
│  │            ╲               │  AP：Cassandra, DynamoDB
│  A ────────────────           │      Redis（預設）
└──────────────────────────────┘
```

### 各資料庫的 CAP 定位

| 系統 | CAP 定位 | 說明 |
|------|---------|------|
| MySQL（主從）| CP | 主節點故障時，等選出新主才服務 |
| Redis Sentinel | AP | 主節點故障時，可能短暫服務舊資料 |
| Kafka | AP | 允許短暫不一致，但最終追上 |
| ZooKeeper | CP | 過半節點不可用則拒絕服務 |
| Cassandra | AP | 可調一致性等級（QUORUM vs ONE） |
| Etcd | CP | Raft 共識，保證強一致 |

## 三、BASE 一致性 — 最終一致性的實踐 <!-- 💡 進階 -->

```
ACID（傳統資料庫）：
  Atomicity     原子性：要麼全成功，要麼全失敗
  Consistency   一致性：操作前後資料合法
  Isolation     隔離性：並發事務互不干擾
  Durability    持久性：已提交的資料不丟失

BASE（分散式系統）：
  Basically Available    基本可用：允許部分失敗，但系統繼續服務
  Soft state             軟狀態：資料可能暫時不一致（還在同步中）
  Eventually consistent  最終一致：一段時間後，所有節點資料會收斂一致

比喻：
  ACID = 銀行轉帳「鎖定資源，確認完才放手」（強一致）
  BASE = 快遞追蹤「掃描了但系統還沒更新，等一下再查就好」（最終一致）
```

### 最終一致性的實作模式

```
事件驅動最終一致（Event-Driven Eventually Consistent）：

Step 1: 訂單服務：在本地 DB 同時寫訂單 + 一筆「待發送事件」（Outbox 模式）
  INSERT INTO orders (id, ...) VALUES (...);
  INSERT INTO outbox (event_type, payload) VALUES ('ORDER_CREATED', '...');

Step 2: 後台進程掃描 outbox，把事件發到 Kafka
  SELECT * FROM outbox WHERE sent = false;
  kafkaTemplate.send("order-events", event);
  UPDATE outbox SET sent = true WHERE id = ?;

Step 3: 結算服務訂閱 Kafka，收到事件後執行結算
  @KafkaListener(topics = "order-events")
  void onOrderCreated(OrderEvent event) {
      settlementService.processOrder(event);
  }

關鍵：Outbox 和訂單在同一個 DB 事務中，Kafka 可能延遲但最終到達
      → 保證「訂單建立」和「結算觸發」最終都會發生
```

## 四、Raft 共識算法 — 讓多個節點達成一致 <!-- 🔴 資深 -->

### 為什麼需要共識算法？

```
問題：3 個節點組成的 Redis Cluster，如何選出新的 Leader？
      如何保證所有節點的資料順序一致？

無腦方案：讓最新的節點當 Leader
  問題：「最新」怎麼判斷？節點間的時鐘不同步！

Raft 的答案：透過「投票」達成共識（就像民主選舉）
  多數決（Quorum）：超過半數節點同意才能提交
  3 個節點 → 至少 2 票才算通過
  5 個節點 → 至少 3 票才算通過
```

### Raft 的三個核心機制

```
機制一：Leader 選舉

初始狀態：所有節點都是 Follower
  ↓ Follower 超時沒收到 Leader 心跳
Candidate：「我要參選 Leader！投我！」
  ↓ 向其他節點發送投票請求
  ↓ 收到超過半數的「同意票」
Leader：當選！開始接收客戶端請求 + 發心跳給 Follower

選舉規則：
  - 每個節點每任期（Term）只能投一票
  - 只能投給「日誌比自己新（或相同）」的 Candidate
  - 先到先得（收到投票請求就立刻投）

機制二：日誌複製

Client → Leader：「寫入 x=5」
Leader → 所有 Follower：「複製這條日誌」
Follower 確認收到 → Leader 收到超過半數的確認
Leader → 提交這條日誌（寫入狀態機）
Leader → Client：「成功」
Leader → 告知 Follower 也可以提交

關鍵：超過半數才提交 → 就算 1 個節點掛了，資料也不丟

機制三：安全性保證

已提交的日誌永遠不會被覆蓋
只有包含最新已提交日誌的節點才能當選 Leader
```

### Raft 在 Java 生態的實際應用

```
你不需要自己實作 Raft，但你要知道它在哪裡：

Etcd：Kubernetes 的元資料儲存，使用 Raft
ZooKeeper：Kafka 的元資料管理，使用 ZAB（類似 Raft）
Kafka（KRaft 模式）：第28章提到，不再依賴 ZooKeeper，改用 Raft

你作為工程師需要知道的：
  ✅ 設定 Leader 選舉超時（election timeout）不能太短，否則頻繁選舉
  ✅ 節點數量建議奇數（3/5/7），偶數數量不能提升容錯能力
      3 節點：最多容忍 1 個掛掉（需要 2 票）
      4 節點：最多容忍 1 個掛掉（需要 3 票）← 和 3 節點一樣，多白花 1 台機器！
      5 節點：最多容忍 2 個掛掉（需要 3 票）← 值得！
```

## 五、分散式事務 <!-- 🔴 資深 -->

### 場景：跨服務的原子操作

```
交易所下單流程（橫跨 3 個服務的操作）：

  訂單服務：建立訂單記錄
  帳戶服務：凍結用戶資金（防止超額下單）
  資產服務：預扣庫存（防止超賣）

要求：這三個操作「要麼全成功，要麼全失敗」
但：它們在三個不同的 DB！
```

### 方案一：2PC（Two-Phase Commit，兩階段提交）

```
有一個協調者（Coordinator）：

Phase 1（準備階段）：
  Coordinator → 訂單服務：「你能提交嗎？」
  Coordinator → 帳戶服務：「你能提交嗎？」
  Coordinator → 資產服務：「你能提交嗎？」
  → 全部回答「能」

Phase 2（提交階段）：
  Coordinator → 所有服務：「提交！」
  所有服務提交並釋放鎖

問題：
  ❌ 協調者是單點故障（Coordinator 掛了？所有服務鎖死！）
  ❌ Phase 1 期間，所有服務都持有鎖 → 吞吐量低
  ❌ 超時難處理（服務在 Phase 2 沒收到提交指令怎麼辦？）

適用：同一個公司內、低延遲、小事務（一些資料庫集群內部使用）
不適用：跨網際網路的微服務
```

### 方案二：Saga 模式（推薦）

```
思路：把大事務拆成多個有「補償操作」的小事務
      如果中間失敗，執行反向的補償操作（撤銷）

編舞（Choreography）模式 — 用事件：
  訂單服務：建立訂單 → 發 OrderCreated 事件
  帳戶服務：收到事件 → 凍結資金 → 發 FundsReserved 事件
  資產服務：收到事件 → 預扣庫存 → 發 InventoryReserved 事件
  ↓ 失敗
  資產服務：扣庫存失敗 → 發 InventoryFailed 事件
  帳戶服務：收到失敗 → 解凍資金（補償） → 發 FundsReleased 事件
  訂單服務：收到失敗 → 取消訂單（補償）

指揮（Orchestration）模式 — 用 Saga 協調器：
  有一個 SagaOrchestrator 統一協調所有步驟
  失敗時按相反順序執行補償操作
```

```java
// Saga Orchestration 實作（概念版）
@Service
public class PlaceOrderSaga {

    public void execute(PlaceOrderCommand cmd) {
        SagaContext ctx = new SagaContext(cmd.getOrderId());

        try {
            // Step 1：建立訂單
            orderService.create(cmd);
            ctx.addCompensation(() -> orderService.cancel(cmd.getOrderId()));

            // Step 2：凍結資金
            accountService.reserve(cmd.getUserId(), cmd.getAmount());
            ctx.addCompensation(() -> accountService.release(cmd.getUserId(), cmd.getAmount()));

            // Step 3：預扣庫存
            inventoryService.reserve(cmd.getSymbol(), cmd.getQuantity());
            // 所有步驟成功，提交！

        } catch (Exception e) {
            // 失敗：按相反順序執行補償
            ctx.rollback();
            throw new SagaFailedException(e);
        }
    }
}

class SagaContext {
    private final Deque<Runnable> compensations = new ArrayDeque<>();

    public void addCompensation(Runnable comp) {
        compensations.push(comp);  // 用 Stack，後加的先補償
    }

    public void rollback() {
        while (!compensations.isEmpty()) {
            try {
                compensations.pop().run();  // 逐一執行補償操作
            } catch (Exception e) {
                // 補償失敗需要人工介入！記錄告警
                log.error("補償失敗，需要人工處理", e);
            }
        }
    }
}
```

### 方案三：TCC（Try-Confirm-Cancel）

```
比 Saga 更精細的控制：

Try 階段：預留資源（不真正操作）
  帳戶服務：凍結 100 元（餘額 900，凍結 100）
  庫存服務：預扣 10 個（可用 90，預扣 10）

Confirm 階段：確認執行
  帳戶服務：真正扣款（餘額 900 → 800，凍結 0）
  庫存服務：確認扣減（可用 90，預扣 0）

Cancel 階段：取消預留
  帳戶服務：解凍（餘額 900，凍結 0）
  庫存服務：退回預扣（可用 100，預扣 0）

vs Saga：
  TCC 更快（2 次網路往返 vs Saga 的多次事件）
  但每個服務都需要實作 try/confirm/cancel 三個方法（開發成本高）
  適合：對延遲要求高的場景（例如撮合後的資金結算）
```

## 六、一致性 Hash — 分散式快取的擴縮容 <!-- 💡 進階 -->

```
問題：3 個 Redis 節點，如何分配 key？

簡單取模：hash(key) % 3
  問題：新增一個節點（3→4），幾乎所有 key 都要重新映射！
       大量快取失效 → 所有請求打到 DB → DB 炸了

一致性 Hash：
  把所有可能的 hash 值排成一個「圓環」（0 ~ 2^32）
  節點放在圓環的某些點上
  key 順時針找到的第一個節點就是它的歸屬

   0
   ↑
   │   Node-A（hash=100）
  Node-C  ──────────────── Node-A
（hash=600）               ↓
   │                   Node-B（hash=300）
   └──────────────────────┘
               ↑ 2^32 = 0（循環）

key-X（hash=150）→ 順時針 → Node-B（最近的節點）
key-Y（hash=450）→ 順時針 → Node-C

新增 Node-D（hash=400）：
  只有 hash 在 (300, 400] 的 key 從 Node-C 遷移到 Node-D
  其他 key 完全不受影響！（遷移量 = 1/N）

Java 實作（概念）：
TreeMap<Long, String> ring = new TreeMap<>();  // hash → 節點名稱
ring.put(100L, "Node-A");
ring.put(300L, "Node-B");
ring.put(600L, "Node-C");

// 查找 key 應該在哪個節點：
Long hash = hashFn(key);
// 找 >= hash 的第一個節點（順時針）
Map.Entry<Long, String> entry = ring.ceilingEntry(hash);
if (entry == null) entry = ring.firstEntry();  // 繞圓環
String node = entry.getValue();
```

## 七、Outbox Pattern：保證事件「一定被發送」<!-- 💡 進階 -->

分散式系統最常見的一致性問題：

```
問題場景：
  @Transactional
  public void createOrder(OrderRequest req) {
      orderRepository.save(order);         // ① DB 寫入成功
      kafkaTemplate.send("order-events",   // ② 發 Kafka 事件
          OrderCreatedEvent.from(order));
  }

  ❌ 風險 1：DB 成功，Kafka 發失敗（網路抖動）
     → 訂單建立了，但庫存服務永遠不知道，庫存沒扣
  
  ❌ 風險 2：Kafka 成功，但 DB 事務回滾
     → 庫存被扣了，但訂單不存在
```

**Outbox Pattern** 的解法：把「發事件」這個操作也寫入同一個 DB 事務裡。

### 實作：Outbox Table + 事件發布器

```sql
-- Outbox 表：和業務資料在同一個 DB
CREATE TABLE outbox_events (
    id              BIGINT AUTO_INCREMENT PRIMARY KEY,
    aggregate_id    VARCHAR(50)  NOT NULL,   -- 哪個 Aggregate（如 orderId）
    aggregate_type  VARCHAR(50)  NOT NULL,   -- 哪種類型（Order, Payment...）
    event_type      VARCHAR(100) NOT NULL,   -- 事件類型
    payload         JSON         NOT NULL,   -- 事件內容
    status          VARCHAR(20)  NOT NULL DEFAULT 'PENDING',
                                             -- PENDING / PUBLISHED / FAILED
    created_at      DATETIME(3)  NOT NULL DEFAULT CURRENT_TIMESTAMP(3),
    published_at    DATETIME(3),
    retry_count     INT          NOT NULL DEFAULT 0,
    INDEX idx_status_created (status, created_at)
);
```

```java
// Step 1：在同一個 DB 事務裡，同時儲存業務資料 + outbox 記錄
@Service
@Transactional                   // 同一個事務！
public class OrderService {

    private final OrderRepository orderRepo;
    private final OutboxEventRepository outboxRepo;

    public Order createOrder(CreateOrderRequest req) {
        // 1. 儲存訂單
        Order order = Order.create(req);
        orderRepo.save(order);

        // 2. 把「要發送的事件」寫進 outbox（也在同一個事務裡）
        OutboxEvent outboxEvent = OutboxEvent.builder()
            .aggregateId(order.getId())
            .aggregateType("Order")
            .eventType("OrderCreated")
            .payload(toJson(OrderCreatedEvent.from(order)))
            .status(EventStatus.PENDING)
            .build();
        outboxRepo.save(outboxEvent);

        // 事務提交：兩個寫入要麼都成功，要麼都回滾
        // Kafka 還沒發！
        return order;
    }
}

// Step 2：獨立的 Publisher（定時掃描 outbox，發布到 Kafka）
@Component
public class OutboxEventPublisher {

    private final OutboxEventRepository outboxRepo;
    private final KafkaTemplate<String, String> kafkaTemplate;

    // 每 500ms 掃一次未發送的事件
    @Scheduled(fixedDelay = 500)
    @Transactional
    public void publishPendingEvents() {
        List<OutboxEvent> pendingEvents = outboxRepo
            .findTop100ByStatusOrderByCreatedAt(EventStatus.PENDING);

        for (OutboxEvent event : pendingEvents) {
            try {
                kafkaTemplate.send(
                    toTopicName(event.getEventType()),
                    event.getAggregateId(),
                    event.getPayload()
                ).get(3, TimeUnit.SECONDS);  // 同步等待確認

                event.setStatus(EventStatus.PUBLISHED);
                event.setPublishedAt(LocalDateTime.now());
            } catch (Exception e) {
                event.setRetryCount(event.getRetryCount() + 1);
                if (event.getRetryCount() >= 5) {
                    event.setStatus(EventStatus.FAILED);  // 超過重試次數
                }
                log.error("事件發布失敗: {}", event.getId(), e);
            }
            outboxRepo.save(event);
        }
    }
}
```

不需要輪詢，延遲低，也不會在 DB 上產生額外的查詢負載

### Outbox Pattern 的保證

```
保證：
  ✅ 業務資料和事件記錄原子性（同一個 DB 事務）
  ✅ 事件一定會被發送（Publisher 會重試）
  ✅ 即使 Publisher 宕機重啟，未發送的事件不會丟失

注意：
  ⚠️ At-Least-Once（至少一次）語義：可能重複發送（Publisher 發完就宕機）
  → Kafka 消費者必須做冪等處理（見第五十八章）

適用場景：
  任何需要「DB 寫入 + 發送事件/消息」且必須保持一致的場景
  例如：下單後發庫存扣減事件、支付後發通知事件、用戶注冊後發歡迎郵件
```

---

## 八、本章總結

```
問面試官：「你們系統的一致性要求是什麼？」
  CP 型回答：「我們是金融系統，一致性優先，分區時寧可停服」
  AP 型回答：「我們是社群平台，可用性優先，短暫不一致可以接受」

三大模式速記：
  一致性問題 → CAP 分析：我選 CP 還是 AP？
  跨服務事務 → Saga（通用）或 TCC（高性能）
  分散式協調 → Raft（選主 + 日誌複製）
  快取分片 → 一致性 Hash（擴縮容遷移量最小）
```

| 概念 | 一句話 | Java 生態 |
|------|--------|---------|
| CAP | 分散式系統只能保證 C/A/P 的其中兩個 | MySQL=CP, Redis=AP |
| BASE | 不強求 ACID，接受最終一致 | Kafka + Outbox 模式 |
| Raft | 多數票決定一切，保證強一致 | Etcd, ZooKeeper, KRaft |
| 2PC | 全部準備好才提交，協調者是弱點 | 部分 DB cluster |
| Saga | 補償事務，最終一致 | Spring + Kafka/事件 |
| TCC | 預留→確認→取消，更快但更複雜 | 金融結算場景 |
| 一致性 Hash | 擴縮容只遷移 1/N 的 key | Redis Cluster, Cassandra |
