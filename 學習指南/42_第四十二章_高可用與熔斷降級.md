# 第四十二章：高可用與熔斷降級

> **前置章節**：第四十章（微服務）、第四十一章（分散式系統）  
> **核心命題**：系統不可能不出問題，但出問題時怎麼讓影響最小？

---

## 一、可用性（Availability）怎麼算？

```
可用性 = 系統正常運作時間 / 總時間

幾個九的意思：
  99%     = 每年可停機 3.65 天   （一般系統）
  99.9%   = 每年可停機 8.76 小時 （普通網站）
  99.99%  = 每年可停機 52 分鐘   （電商平台）
  99.999% = 每年可停機 5.26 分鐘 （金融交易系統要求）

如何提高可用性：
  1. 消除單點故障（SPOF）：每個組件至少 2 個副本
  2. 快速故障偵測：心跳 + 健康檢查
  3. 快速故障轉移：自動切換，不需要人工介入
  4. 熔斷降級：局部失敗不擴散成全局崩潰
  5. 限流：保護系統不被過量請求壓垮

串聯系統的可用性 = 各組件可用性相乘：
  A(99.9%) → B(99.9%) → C(99.9%)
  總體可用性 = 0.999 × 0.999 × 0.999 ≈ 99.7%
  → 微服務越多，整體可用性越低！所以每個服務的健壯性很重要
```

## 二、熔斷器（Circuit Breaker） <!-- 💡 進階 -->

### 電路熔斷的比喻

```
家裡電路過載 → 保險絲熔斷 → 電流中斷 → 防止火災
恢復後 → 換保險絲 → 電路恢復

軟體熔斷器：
  下游服務持續失敗 → 熔斷器跳閘 → 不再呼叫下游（快速失敗）→ 防止雪崩
  等一段時間 → 嘗試少量請求 → 如果成功 → 熔斷器關閉（恢復正常）
```

### 三個狀態

```
CLOSED（關閉）→ 正常狀態，所有請求都轉發
    │
    │ 失敗率超過閾值（例如 50%）
    │ 或慢響應率超過閾值
    ↓
OPEN（打開）→ 熔斷！所有請求立刻返回 fallback，不呼叫下游
    │
    │ 等待 waitDuration（例如 60 秒）
    ↓
HALF_OPEN（半開）→ 允許少量試探性請求通過
    │         │
    │ 成功    │ 失敗
    ↓         ↓
CLOSED     OPEN（重新熔斷，等待更長時間）
```

### Resilience4j 實作

```java
// 引入依賴
// implementation 'io.github.resilience4j:resilience4j-spring-boot3:2.1.0'

// application.yml 配置
resilience4j:
  circuitbreaker:
    instances:
      settlement-service:
        # 基於次數的滑動視窗（最近 10 次請求）
        slidingWindowType: COUNT_BASED
        slidingWindowSize: 10
        # 失敗率超過 50% 就熔斷
        failureRateThreshold: 50
        # 慢響應（> 2 秒）也算失敗
        slowCallDurationThreshold: 2s
        slowCallRateThreshold: 50
        # 熔斷後等 60 秒再嘗試
        waitDurationInOpenState: 60s
        # HALF_OPEN 時允許 5 個試探請求
        permittedNumberOfCallsInHalfOpenState: 5
        # 最少 5 次請求才開始計算失敗率（防止樣本太少誤判）
        minimumNumberOfCalls: 5
```

```java
@Service
public class OrderService {

    @Autowired
    private SettlementClient settlementClient;

    /**
     * @CircuitBreaker：包裹這個方法，監控成功/失敗/慢響應
     * fallbackMethod：熔斷或失敗時呼叫的降級方法
     */
    @CircuitBreaker(name = "settlement-service", fallbackMethod = "settleFallback")
    @Retry(name = "settlement-service")  // 搭配重試
    public SettlementResult settle(Trade trade) {
        return settlementClient.settle(trade);
    }

    /**
     * 降級方法：熔斷時的備用邏輯
     * 參數簽名必須和原方法相同，最後加上 Throwable
     */
    public SettlementResult settleFallback(Trade trade, Throwable ex) {
        // 降級策略：把結算任務放入重試佇列，稍後再處理
        retryQueue.push(trade);
        log.warn("結算服務熔斷，交易 {} 已加入重試佇列", trade.getId());

        // 回傳一個「待結算」的狀態，不回傳錯誤
        return SettlementResult.pending(trade.getId());
    }
}
```

### 監控熔斷器狀態

```java
// Actuator Endpoint 可以查看熔斷器狀態
// GET /actuator/circuitbreakers
// GET /actuator/circuitbreakerevents

// 程式碼監聽狀態變化（用於告警）
@EventListener
public void onCircuitBreakerStateChange(CircuitBreakerOnStateTransitionEvent event) {
    String name = event.getCircuitBreakerName();
    CircuitBreaker.State from = event.getStateTransition().getFromState();
    CircuitBreaker.State to   = event.getStateTransition().getToState();

    log.warn("熔斷器 {} 狀態變化：{} → {}", name, from, to);

    if (to == CircuitBreaker.State.OPEN) {
        // 傳送告警！
        alertService.send("熔斷器跳閘", name + " 服務不可用");
    }
}
```

## 三、重試策略與指數退避 <!-- 💡 進階 -->

### 為什麼不能無腦重試？

```
場景：結算服務超時了，你重試 10 次
  → 結算服務更忙了（它已經在處理第一次的請求）
  → 重試讓服務更忙 → 更多超時 → 更多重試
  → 雪崩！

原因：你不知道第一次請求是否真的失敗了
      超時 ≠ 失敗（可能只是慢）
```

### 指數退避（Exponential Backoff）+ Jitter（抖動）

```
不好的重試：立刻重試，固定間隔
  T+0ms   第 1 次
  T+100ms 第 2 次（固定 100ms 後重試）
  T+200ms 第 3 次
  → 多個客戶端同時這樣做 → 請求峰值剛好同步 → 更糟糕！

指數退避：等待時間按指數增長，加入隨機抖動
  T+0ms              第 1 次
  T+(100 + random)ms 第 2 次（base * 2^1 + jitter）
  T+(400 + random)ms 第 3 次（base * 2^2 + jitter）
  T+(800 + random)ms 第 4 次（base * 2^3 + jitter）
  → 不同客戶端的重試時間分散開來 → 避免同步峰值
```

```java
// Resilience4j Retry 配置
resilience4j:
  retry:
    instances:
      settlement-service:
        maxAttempts: 3        # 最多重試 3 次（包含第一次）
        waitDuration: 500ms   # 初始等待 500ms
        enableExponentialBackoff: true
        exponentialBackoffMultiplier: 2.0   # 每次翻倍
        exponentialMaxWaitDuration: 10s     # 最長等 10 秒
        retryExceptions:                    # 只重試這些例外
          - java.net.SocketTimeoutException
          - org.springframework.web.client.ResourceAccessException
        ignoreExceptions:                   # 不重試業務錯誤
          - com.example.exception.BusinessException

// 注意：重試只適合「冪等操作」（重複執行結果相同）
// 例如：查詢、新增（帶唯一ID）是冪等的
// 例如：「扣款 100 元」不是冪等的！（不能重試，會重複扣款）
```

## 四、限流算法 <!-- 🔴 資深 -->

### 四種限流算法對比

```
計數器（Fixed Window Counter）：
  每秒最多 100 個請求，用計數器計數，超過就拒絕
  問題：
    0.9 秒：100 個請求通過
    1.0 秒：計數器重置
    1.1 秒：又 100 個請求通過
    → 在 0.9~1.1 秒的 0.2 秒內，通過了 200 個請求！窗口邊界突刺

滑動視窗（Sliding Window）：
  用最近 1 秒的請求數量（精確但記憶體消耗大）

令牌桶（Token Bucket）：
  桶裡有令牌，請求需要消耗令牌
  定速補充令牌（例如每秒補 100 個）
  桶滿時不再補充（允許突發流量到桶容量上限）
  適合：允許適度突發的限流

漏桶（Leaky Bucket）：
  請求進入桶，以固定速率從桶底流出處理
  多餘的請求排隊（或拒絕）
  適合：嚴格控制處理速率（例如給外部 API 的請求）
```

```
令牌桶直覺：
  🪣 桶容量 = 200（允許瞬間最多 200 個請求通過）
  ⏰ 補充速率 = 100/s（穩定流量上限 100 rps）

  平時（流量低）：令牌累積到 200
  突發（秒殺）：瞬間消耗 200 個令牌 → 允許短暫突發
  持續高流量：穩定處理 100 rps，多餘的拒絕
```

### Spring Cloud Gateway 整合令牌桶限流

```java
// application.yml
spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          filters:
            - name: RequestRateLimiter
              args:
                # 使用 Redis 存儲（支援多節點共享限流狀態）
                rate-limiter: "#{@redisRateLimiter}"
                key-resolver: "#{@userKeyResolver}"  # 按用戶 ID 限流

# 按用戶 ID 限流（每個用戶獨立計算）
@Bean
KeyResolver userKeyResolver() {
    return exchange -> Mono.justOrEmpty(
        exchange.getRequest().getHeaders().getFirst("X-User-Id")
    ).defaultIfEmpty("anonymous");
}

# 令牌桶配置
@Bean
RedisRateLimiter redisRateLimiter() {
    return new RedisRateLimiter(
        100,   // replenishRate：每秒補充 100 個令牌
        200,   // burstCapacity：桶容量 200（允許瞬間 200 rps）
        1      // requestedTokens：每個請求消耗 1 個令牌
    );
}
```

## 五、降級策略 <!-- 💡 進階 -->

```
降級 = 當某個功能不可用時，提供「簡化版」替代

降級層次（從優到劣）：

Level 1：快取降級
  正常：查 DB 取最新訂單狀態
  降級：查 Redis 取快取的訂單狀態（可能稍舊，但有資料）

Level 2：靜態降級
  正常：根據用戶歷史行為推薦商品
  降級：返回預先準備的「熱門商品」清單（不個人化但至少有內容）

Level 3：空降級
  正常：顯示即時行情
  降級：顯示「行情暫時不可用，請稍後再試」

Level 4：完全拒絕
  正常：接受所有請求
  降級：只接受 VIP 用戶的請求，普通用戶排隊等待

撮合引擎的降級策略設計：
  ┌────────────────────────────────────────────────────────┐
  │ 行情推送掛了 → 降級：停止推送，保留撮合功能（核心不降）  │
  │ 結算服務掛了 → 降級：繼續撮合，結算任務進入重試佇列      │
  │ 風控服務掛了 → 降級：採用保守規則（不允許大額下單）       │
  │ 撮合引擎掛了 → 不能降級：這是核心，直接告警人工介入       │
  └────────────────────────────────────────────────────────┘
```

```java
// 使用 @Fallback 搭配 Resilience4j
@Service
public class MarketDataService {

    @CircuitBreaker(name = "external-market", fallbackMethod = "getCachedMarketData")
    public MarketData getLatestMarketData(String symbol) {
        return externalMarketClient.getLatest(symbol);  // 外部行情源
    }

    // 降級：返回快取的行情資料
    public MarketData getCachedMarketData(String symbol, Throwable ex) {
        log.warn("外部行情源不可用，使用快取資料 symbol={}", symbol);
        return redisTemplate.opsForValue().get("market:" + symbol);
    }
}
```

## 六、健康檢查設計 <!-- 💡 進階 -->

```
Kubernetes 有兩種探針（第32章提到）：
  Liveness Probe：「你還活著嗎？」死了就重啟
  Readiness Probe：「你準備好接收流量了嗎？」沒準備好就不送流量

Spring Boot Actuator 內建健康檢查：
```

```yaml
# application.yml
management:
  endpoint:
    health:
      show-details: always
      probes:
        enabled: true  # 啟用 /health/liveness 和 /health/readiness
  health:
    livenessstate:
      enabled: true
    readinessstate:
      enabled: true
```

```java
// 自訂健康指標
@Component
public class OrderBookHealthIndicator implements HealthIndicator {

    @Autowired
    private OrderBook orderBook;

    @Override
    public Health health() {
        // 撮合引擎核心檢查：訂單簿是否正常工作
        try {
            int orderCount = orderBook.getActiveOrderCount();
            if (orderCount < 0) {
                return Health.down()
                    .withDetail("reason", "訂單簿異常：負數訂單數量")
                    .build();
            }
            return Health.up()
                .withDetail("activeOrders", orderCount)
                .withDetail("bestBid", orderBook.getBestBid())
                .withDetail("bestAsk", orderBook.getBestAsk())
                .build();
        } catch (Exception e) {
            return Health.down(e).build();
        }
    }
}

// GET /actuator/health 回傳：
// {
//   "status": "UP",
//   "components": {
//     "orderBook": { "status": "UP", "details": { "activeOrders": 1523 } },
//     "db": { "status": "UP" },
//     "redis": { "status": "UP" }
//   }
// }
```

## 七、超時設定的重要性

```
每個外部呼叫都必須設超時！沒有超時 = 最大的 BUG

超時設定原則：
  讀取操作：100ms ~ 500ms（快速返回或失敗）
  寫入操作：1s ~ 3s（允許稍長，但不能無限等）
  關鍵路徑：越短越好，超時就熔斷

超時層級（從外到內都要設）：
  Nginx/Gateway：最外層超時（例如 5s）
  Feign/RestTemplate：HTTP 連線超時 + 讀取超時
  資料庫連線池：query timeout
  Redis：socket timeout

feign:
  client:
    config:
      default:
        connectTimeout: 500    # 連線超時 500ms
        readTimeout: 2000      # 讀取超時 2000ms

spring:
  datasource:
    hikari:
      connectionTimeout: 3000      # 取得連線超時 3s
      validationTimeout: 1000      # 驗證連線超時 1s

// Redis timeout
spring:
  data:
    redis:
      timeout: 500ms   # 命令執行超時 500ms
```

## 八、本章總結

```
高可用的設計哲學：
「假設任何事情都會出錯，然後設計一個出錯時影響最小的系統」

高可用四件套：
  1. 熔斷器（Circuit Breaker）：局部失敗不擴散
  2. 重試 + 指數退避：暫時性失敗自動恢復
  3. 限流（Rate Limiter）：保護下游不被壓垮
  4. 降級（Fallback）：核心功能保住，非核心暫停

每個微服務的高可用清單：
  ✅ 設定了熔斷器嗎？
  ✅ 有降級策略嗎？（熔斷時回傳什麼？）
  ✅ 所有外部呼叫都有超時嗎？
  ✅ 重試只用在冪等操作上嗎？
  ✅ 健康檢查端點正確回報了所有依賴的狀態嗎？
  ✅ 熔斷器跳閘時有告警嗎？
```
