## 一、索引原理（回顧第三十三章 B+Tree）

```
沒有索引的查詢：
  SELECT * FROM orders WHERE user_id = 'user-123';
  → 全表掃描（Full Table Scan）：從第一行讀到最後一行
  → 100 萬行 × 100μs/IO ≈ 100 秒（超慢！）

有索引的查詢：
  CREATE INDEX idx_user_id ON orders(user_id);
  → B+Tree 索引：3 次 IO 找到用戶的所有訂單
  → ~300μs（快 30 萬倍！）

B+Tree 索引的物理結構（MySQL InnoDB）：
  主鍵索引（Clustered Index）：葉節點存整行資料
  二級索引（Secondary Index）：葉節點只存主鍵值
  → 二級索引查完後，還要「回表」到主鍵索引取整行資料（多 1 次 IO）
```

### 複合索引的最左前綴規則

```sql
-- 建立複合索引
CREATE INDEX idx_user_symbol_time ON orders(user_id, symbol, created_at);

-- 可以用到索引：
SELECT * FROM orders WHERE user_id = 'u1';                               -- ✅ 用到 user_id
SELECT * FROM orders WHERE user_id = 'u1' AND symbol = 'BTC';           -- ✅ 用到 user_id, symbol
SELECT * FROM orders WHERE user_id = 'u1' AND symbol = 'BTC'
                       AND created_at > '2024-01-01';                    -- ✅ 全用到

-- 不能用到索引（跳過了 user_id）：
SELECT * FROM orders WHERE symbol = 'BTC';                               -- ❌ 沒用到
SELECT * FROM orders WHERE symbol = 'BTC' AND created_at > '2024-01-01'; -- ❌ 沒用到

-- 部分用到：
SELECT * FROM orders WHERE user_id = 'u1' AND created_at > '2024-01-01'; -- ⚠️ 只用到 user_id
                                                                          -- symbol 跳過了，
                                                                          -- created_at 無法用索引

最左前綴規則：
  複合索引 (A, B, C) 等同於有三個索引：(A)、(A,B)、(A,B,C)
  查詢條件必須從最左邊開始連續使用
```

### 覆蓋索引（避免回表）

```sql
-- 問題：每次都要回表取整行資料
SELECT order_id, status FROM orders WHERE user_id = 'u1';
-- → 先查 idx_user_id 索引找到 order_id
-- → 再回主鍵索引取整行（包含 status）← 這次 IO 可以省掉！

-- 解法：建立覆蓋索引（把查詢需要的欄位都放進索引）
CREATE INDEX idx_user_order_status ON orders(user_id, order_id, status);
-- → 索引葉節點直接包含 order_id 和 status
-- → 不需要回表！直接從索引返回結果
-- EXPLAIN 看到 Extra: Using index → 表示走了覆蓋索引

-- 撮合引擎常用查詢的覆蓋索引設計：
-- 查「用戶的所有活躍訂單」
CREATE INDEX idx_active_orders ON orders(user_id, status, created_at, order_id, symbol, price, quantity)
-- WHERE user_id=? AND status='ACTIVE'
-- SELECT order_id, symbol, price, quantity → 全部在索引裡，不回表
```

## 二、EXPLAIN 執行計畫 <!-- 💡 進階 -->

### 讀懂 EXPLAIN

```sql
EXPLAIN SELECT o.*, u.name
FROM orders o
JOIN users u ON o.user_id = u.id
WHERE o.status = 'ACTIVE'
  AND o.created_at > '2024-01-01'
ORDER BY o.created_at DESC
LIMIT 10;
```

```
執行計畫輸出：
┌────┬───────────┬──────────┬──────┬───────────────────────┬──────────────┬────────┬──────┐
│ id │ table     │ type     │ key  │ Extra                 │ rows         │ key_len│ ref  │
├────┼───────────┼──────────┼──────┼───────────────────────┼──────────────┼────────┼──────┤
│  1 │ o(orders) │ range    │ idx_active│ Using index cond │ 50,000      │ 8      │NULL  │
│  1 │ u(users)  │ eq_ref   │ PRIMARY│ NULL              │ 1            │ 4      │o.user_id│
└────┴───────────┴──────────┴──────┴───────────────────────┴──────────────┴────────┴──────┘

關鍵欄位解讀：

type（掃描類型，效能從好到差）：
  system  → 只有一行，最快
  const   → 主鍵或唯一索引等值查詢，找到一行
  eq_ref  → JOIN 時用主鍵，每次找到一行（很好！）
  ref     → 普通索引等值查詢
  range   → 索引範圍查詢（> < BETWEEN）← 還不錯
  index   → 全索引掃描（比 ALL 好一點但不多）
  ALL     → 全表掃描（最慢！看到這個就要加索引）

key：實際使用的索引（NULL = 沒用到任何索引）
rows：預計掃描的行數（越小越好）
Extra 重要項目：
  Using index      → 覆蓋索引，不需要回表（很好！）
  Using filesort   → 需要額外排序（可能慢，看能否用索引排序）
  Using temporary  → 用臨時表（GROUP BY 或 ORDER BY 複雜查詢，很慢）
  Using index cond → 索引條件下推（ICP），減少回表次數（好）
```

## 三、事務隔離等級與鎖 <!-- 🔴 資深 -->

### 四個隔離等級

```
問題：多個事務並發執行，可能出現什麼問題？

  臟讀（Dirty Read）：讀到另一個事務未提交的資料
  不可重複讀（Non-Repeatable Read）：同一事務內，兩次讀到不同值（別人修改了）
  幻讀（Phantom Read）：同一事務內，兩次查詢的行數不同（別人新增了行）

隔離等級：

READ UNCOMMITTED：最低，允許臟讀（幾乎不用）
READ COMMITTED：不允許臟讀，允許不可重複讀
  → 最常用！Oracle, PostgreSQL 預設
REPEATABLE READ：不允許臟讀和不可重複讀，允許幻讀
  → MySQL InnoDB 預設（但 MVCC 解決了大部分幻讀）
SERIALIZABLE：完全隔離，所有問題都不存在，但效能最差（加表鎖）

隔離等級越高 → 問題越少，但並發效能越差
```

### MVCC（多版本並發控制）— InnoDB 的秘密武器

```
MVCC 讓讀操作不需要加鎖！
  普通 SELECT 不加任何鎖，讀的是「快照版本」
  修改操作加寫鎖，但不阻塞讀操作

工作原理（簡化版）：
  每行資料有兩個隱藏欄位：
    db_trx_id（最後修改這行的事務 ID）
    db_roll_ptr（指向 Undo Log，可以還原舊版本）

  事務開始時，記錄當前的「最大事務 ID」= 快照
  讀取時：若某行的 db_trx_id > 快照ID → 這行是之後修改的，不可見
         若某行的 db_trx_id <= 快照ID → 這行是之前修改的，可見

結果：
  Reader 不阻塞 Writer
  Writer 不阻塞 Reader
  只有 Writer 互相阻塞（寫寫衝突才需要鎖）
```

```java
// Spring 中的事務傳播（第27章 @Transactional 深化）
@Service
public class OrderService {

    // 預設 PROPAGATION_REQUIRED：如果外層有事務就加入，沒有就建立
    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        // 呼叫撮合服務：如果這裡拋例外，整個事務回滾（包括 save）
        matchingService.match(order);
    }

    // ISOLATION_SERIALIZABLE：完全隔離，用於帳戶結餘計算（防止幻讀）
    @Transactional(isolation = Isolation.SERIALIZABLE)
    public BigDecimal calculateFinalBalance(String userId) {
        // 這個查詢期間，其他事務不能插入新的交易記錄
        return transactionRepository.sumByUserId(userId);
    }

    // PROPAGATION_REQUIRES_NEW：強制建立新事務（獨立提交）
    // 用途：記錄操作日誌，就算主事務回滾，日誌也要保留
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void saveAuditLog(String action) {
        auditRepository.save(new AuditLog(action));
    }
}
```

### MySQL 鎖機制詳解：行鎖 / 間隙鎖 / Next-Key Lock <!-- 🔴 資深 -->

InnoDB 的鎖不只有「鎖住這一行」這麼簡單，面試和生產問題的來源往往在這裡。

```
三種鎖：
  Record Lock（行鎖）：鎖住索引上的一條記錄
  Gap Lock（間隙鎖）：鎖住兩條索引記錄之間的「空隙」（防止插入）
  Next-Key Lock = Record Lock + Gap Lock（前面的間隙）
```

**為什麼需要間隙鎖？— 防止幻讀**

```sql
-- Session A（REPEATABLE READ 隔離等級）
BEGIN;
SELECT * FROM orders WHERE amount BETWEEN 100 AND 200 FOR UPDATE;
-- 結果：找到 amount=150 這一行，並鎖住它

-- Session B（同時執行）
INSERT INTO orders (amount) VALUES (180);  -- ❌ 被阻塞！
-- 原因：Session A 的 Next-Key Lock 鎖住了 (100, 200] 這個範圍
-- 如果允許插入，Session A 再次查詢就會出現 amount=180（幻讀）
```

**Next-Key Lock 的範圍：**

```
假設 orders.amount 有索引，且有值：50, 100, 150, 200

查詢 WHERE amount = 150 FOR UPDATE：
  → Next-Key Lock 鎖住 (100, 150] 這個區間（左開右閉）

查詢 WHERE amount BETWEEN 100 AND 200 FOR UPDATE：
  → Next-Key Lock 鎖住 (50, 100] + (100, 150] + (150, 200]
  → 即 (50, 200] 整個範圍
  → 沒有索引的欄位：直接鎖全表！
```

**鎖等待和死鎖場景：**

```sql
-- 容易造成死鎖的模式：
-- Session A: 先鎖 id=1，再鎖 id=2
-- Session B: 先鎖 id=2，再鎖 id=1  → 互相等待 → 死鎖

-- InnoDB 偵測到死鎖後，自動回滾其中一個 Transaction（犧牲代價較小的那個）
-- 應用層需要捕捉 DeadlockLoserDataAccessException 並重試
```

**如何觀察鎖：**

```sql
-- 查看當前鎖等待
SELECT * FROM performance_schema.data_lock_waits;

-- 查看持有的鎖
SELECT * FROM performance_schema.data_locks;

-- 查看 InnoDB 狀態（包含最近的死鎖資訊）
SHOW ENGINE INNODB STATUS\G
```

**降低鎖競爭的實務技巧：**

| 技巧 | 說明 |
|------|------|
| 縮短 Transaction 時間 | Transaction 越長，持鎖越久，阻塞越多 |
| 避免大範圍 `FOR UPDATE` | 盡量用精確條件（`WHERE id = ?`）而非範圍查詢 |
| 加索引 | 沒有索引的 WHERE 條件會鎖全表 |
| 使用樂觀鎖 | `@Version` 欄位，避免 `SELECT FOR UPDATE` |
| 一致的鎖定順序 | 多個 Transaction 按同樣順序獲取鎖，避免死鎖 |

## 四、主從複製與讀寫分離 <!-- 🔴 資深 -->

### 主從複製原理

```
Master（主庫）← 接收所有寫操作

                     Binlog（Binary Log）
                     記錄所有 DML/DDL 操作的日誌
                     ↓
      ┌──────────────────────────────────┐
      ↓              ↓                  ↓
   Replica-1      Replica-2           Replica-3
  （從庫1）       （從庫2）          （從庫3）
   接收讀操作     接收讀操作          接收讀操作

複製流程：
  1. Master 執行 INSERT/UPDATE/DELETE → 寫入 Binlog
  2. Replica 的 IO Thread 連接 Master，讀取 Binlog
  3. Replica 把 Binlog 寫到本地的 Relay Log
  4. Replica 的 SQL Thread 重放 Relay Log（延遲複製）

複製延遲（Replication Lag）：
  通常 < 100ms（同機房）
  網路差時可能達到 1~10s
  → 這是讀寫分離的代價：讀到的可能是稍舊的資料！
```

### Spring Boot 讀寫分離

```java
// 動態數據源路由（AbstractRoutingDataSource）
public class ReadWriteRoutingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        // 根據當前上下文決定使用哪個數據源
        return DataSourceContextHolder.get();  // "master" or "replica"
    }
}

// 使用 AOP 自動切換數據源
@Aspect
@Component
public class DataSourceAspect {

    @Before("@annotation(org.springframework.transaction.annotation.Transactional)")
    public void setDataSource(JoinPoint jp) {
        Transactional tx = getTransactional(jp);
        if (tx.readOnly()) {
            DataSourceContextHolder.set("replica");  // 唯讀事務用從庫
        } else {
            DataSourceContextHolder.set("master");   // 可寫事務用主庫
        }
    }
}

// 使用
@Transactional(readOnly = true)  // → 自動路由到 Replica
public List<Order> getOrdersByUser(String userId) {
    return orderRepository.findByUserId(userId);
}

@Transactional  // readOnly = false → 路由到 Master
public Order placeOrder(OrderRequest req) {
    return orderRepository.save(new Order(req));
}
```

### 主從複製的坑：寫後讀不一致

```
場景：
  用戶下單 → 寫入 Master → 馬上跳轉到訂單列表頁
  訂單列表查詢 → 從 Replica 讀 → Replica 還沒複製到最新資料 → 訂單不見了！

解法方案：
  1. 剛剛寫入後的讀取，強制走 Master（Write-After-Read Routing）
  2. 接受短暫不一致，提示用戶「訂單正在處理中」
  3. 用 Sharding Key 保證特定用戶的讀寫都在同一個節點

實作方案 1：
// 在 Session 裡記錄「最近寫入時間」
// 若時間在 Replication Lag 範圍內，強制走 Master
@Service
public class OrderReadService {

    public List<Order> getRecentOrders(String userId) {
        Long lastWriteTime = sessionService.getLastWriteTime(userId);
        boolean recentlyWrote = lastWriteTime != null
            && System.currentTimeMillis() - lastWriteTime < 500;  // 500ms 內的讀取走主庫

        if (recentlyWrote) {
            DataSourceContextHolder.set("master");
        }
        return orderRepository.findByUserId(userId);
    }
}
```

## 五、分庫分表 <!-- 🔴 資深 -->

```
場景：訂單表 10 億行，單表查詢很慢，索引也沒用

分表策略：

水平分片（Sharding）— 按行分（最常用）：
  orders_0：user_id % 4 = 0 的訂單
  orders_1：user_id % 4 = 1 的訂單
  orders_2：user_id % 4 = 2 的訂單
  orders_3：user_id % 4 = 3 的訂單

  每張表只有 1/4 的資料，查詢快 4 倍

垂直分片（Vertical Sharding）— 按列分：
  orders_main：常用欄位（id, user_id, status, price）
  orders_detail：不常用欄位（用戶備註、附件 URL）

分庫：
  把不同分片放到不同的 DB 實例
  user_id % 4 = 0,1 → DB-1（實體機器 A）
  user_id % 4 = 2,3 → DB-2（實體機器 B）

分片帶來的問題：
  ❌ 跨分片 JOIN：SELECT * FROM orders o JOIN users u 
      orders 在 DB-1, users 在 DB-2 → 無法直接 JOIN
  ❌ 全局排序：ORDER BY created_at → 需要從所有分片取資料再合併排序
  ❌ 分散式事務：更新跨分片資料需要 Saga/TCC
  ❌ 擴容困難：從 4 分片擴到 8 分片，需要遷移一半資料

解法：Apache ShardingSphere（Java 生態最成熟的分片中間件）
```

## 六、Change Data Capture（CDC） <!-- 🔴 資深 -->

```
CDC = 捕獲資料庫的每一次變更（INSERT/UPDATE/DELETE）

應用場景：
  1. 資料同步：MySQL → Elasticsearch（搜尋索引同步）
  2. 事件溯源：每次資料變更都產生一個事件
  3. 快取更新：DB 資料變了 → 自動清除/更新 Redis 快取
  4. 審計日誌：不修改業務程式碼，透過 CDC 記錄所有變更

Debezium（最流行的 CDC 工具）：
  原理：讀取 MySQL 的 Binlog（第四十四章主從複製部分）
       把 Binlog 的每條變更翻譯成 Kafka 事件
       下游消費者訂閱 Kafka，處理這些變更事件
```

```java
// Debezium 的 Kafka Connect 配置（JSON）
{
  "name": "orders-cdc-connector",
  "config": {
    "connector.class": "io.debezium.connector.mysql.MySqlConnector",
    "database.hostname": "mysql",
    "database.port": "3306",
    "database.user": "debezium",
    "database.password": "...",
    "database.include.list": "trading_db",
    "table.include.list": "trading_db.orders",
    "topic.prefix": "cdc"
  }
}

// Kafka 會收到如下格式的訊息：
// Topic: cdc.trading_db.orders
// {
//   "op": "u",           // u=update, c=create, d=delete
//   "before": { "id": "123", "status": "PENDING" },
//   "after":  { "id": "123", "status": "FILLED"  },
//   "ts_ms": 1704067200000
// }

// 消費 CDC 事件，同步到 Elasticsearch
@KafkaListener(topics = "cdc.trading_db.orders")
public void syncToElasticsearch(OrderCdcEvent event) {
    if ("d".equals(event.getOp())) {
        esClient.delete(event.getBefore().getId());
    } else {
        esClient.index(event.getAfter());
    }
}
```
