# 第四十五章：效能分析與調優方法論

> **前置章節**：第十一章（JIT）、第二十五章（GC）、第十四章（JMH）  
> **核心命題**：系統變慢了，怎麼找到真正的瓶頸？不靠猜，靠工具。

## 一、效能調優的正確順序

```
常見的錯誤順序：
  1. 系統慢了
  2. 加快取（Redis）
  3. 還是慢
  4. 換更快的伺服器
  5. 還是慢
  6. ？？？

正確順序：
  1. 量測：用工具找到「真正的瓶頸」在哪裡
  2. 分析：為什麼是瓶頸？
  3. 修復：針對瓶頸做最小改動
  4. 驗證：修復後重新量測，確認有改善且沒引入新問題
  5. 重複

原則：
  「不要過早優化（Premature Optimization is the root of all evil — Knuth）」
  大部分時間花在 20% 的熱點程式碼上
  先找到熱點，再優化熱點
```

## 二、JMH 基準測試（深化第14章） <!-- 💡 進階 -->

### 為什麼需要 JMH？

```
問題：用 System.currentTimeMillis() 量測效能的陷阱：
  1. JVM JIT 會優化熱點程式碼，「前幾次慢，後面快」
  2. GC 隨時可能觸發，導致量測時間不穩定
  3. JIT 可能看出你的程式碼「結果沒有使用」而把它優化掉（量測結果是 0！）

JMH（Java Microbenchmark Harness）解決這些問題：
  - 自動 Warmup（預熱）：讓 JIT 先編譯好
  - 多輪量測取平均值
  - Blackhole：防止 JIT 消除你要量測的程式碼
```

```java
@BenchmarkMode(Mode.AverageTime)        // 量測平均執行時間
@OutputTimeUnit(TimeUnit.NANOSECONDS)  // 結果單位：奈秒
@State(Scope.Benchmark)                 // 每個 Benchmark 方法共享這個 State
@Warmup(iterations = 3, time = 1)      // 預熱 3 輪，每輪 1 秒
@Measurement(iterations = 5, time = 1) // 量測 5 輪，每輪 1 秒
@Fork(1)                                // JVM Fork 數量
public class OrderBookBenchmark {

    private OrderBook treeMapBook;
    private ArrayOrderBook arrayBook;
    private Order buyOrder;

    @Setup  // 每次 Benchmark 前執行一次（初始化）
    public void setup() {
        treeMapBook = new OrderBook();
        arrayBook   = new ArrayOrderBook();
        buyOrder    = new Order("BUY", new BigDecimal("50000.00"), new BigDecimal("1.0"));
        // 預先插入 1000 筆訂單，模擬真實場景
        for (int i = 0; i < 1000; i++) {
            treeMapBook.addBid(new Order("BUY",
                new BigDecimal(50000 - i), new BigDecimal("0.1")));
        }
    }

    @Benchmark
    public void treeMapInsert(Blackhole bh) {
        // Blackhole 消費結果，防止 JIT 把這段程式碼優化掉
        bh.consume(treeMapBook.addBid(buyOrder));
    }

    @Benchmark
    public void arrayInsert(Blackhole bh) {
        bh.consume(arrayBook.addBid(50_000_00, 100L));  // 定點數：50000.00, 100
    }

    @Benchmark
    public void getBestBidTreeMap(Blackhole bh) {
        bh.consume(treeMapBook.getBestBid());
    }
}
```

```
JMH 結果範例：
Benchmark                          Mode  Cnt    Score   Error  Units
OrderBookBenchmark.treeMapInsert   avgt    5  312.456 ± 8.234  ns/op
OrderBookBenchmark.arrayInsert     avgt    5   12.123 ± 0.456  ns/op
OrderBookBenchmark.getBestBidTreeMap avgt  5   45.678 ± 1.234  ns/op

解讀：
  arrayInsert 比 treeMapInsert 快 25 倍
  在高頻交易場景（每秒 100 萬筆），這個差距非常顯著
```

## 三、JFR（Java Flight Recorder） <!-- 🔴 資深 -->

### 什麼是 JFR？

```
JFR = JVM 內建的「黑盒飛行記錄器」
  以極低開銷（< 2% CPU）持續記錄：
  - JVM 事件（GC、JIT 編譯、類加載）
  - OS 事件（CPU 使用率、I/O、線程切換）
  - 自訂事件（你的業務操作）

Java 11+ 免費！（之前是商業授權功能）

優點 vs Profiler：
  Profiler（例如 JProfiler）：需要連接到 JVM，有採樣誤差，影響效能
  JFR：在 JVM 內部記錄，開銷極小，可以在生產環境開著
```

### 開啟 JFR 的方式

```bash
# 方式一：JVM 啟動參數（持續記錄）
java -XX:+UnlockDiagnosticVMOptions \
     -XX:+DebugNonSafepoints \
     -XX:StartFlightRecording=duration=60s,filename=recording.jfr \
     -jar matching-engine.jar

# 方式二：對運行中的 JVM 觸發（不需要重啟！）
jcmd <PID> JFR.start duration=60s filename=recording.jfr
jcmd <PID> JFR.stop

# 用 JDK Mission Control 打開 .jfr 文件分析
# 可以看到：方法熱點、GC 事件、Lock 競爭、I/O 阻塞...
```

```java
// 自訂 JFR 事件（追蹤業務操作）
@Label("Order Matched")
@Description("撮合引擎完成一次撮合")
public class MatchingEvent extends Event {

    @Label("Order ID")
    String orderId;

    @Label("Symbol")
    String symbol;

    @Label("Match Price")
    double price;

    @Label("Match Quantity")
    double quantity;
}

// 使用
public Trade match(Order buy, Order sell) {
    MatchingEvent event = new MatchingEvent();
    event.begin();   // 開始記錄

    try {
        Trade trade = doMatch(buy, sell);
        event.orderId   = buy.getId();
        event.symbol    = buy.getSymbol();
        event.price     = trade.getPrice().doubleValue();
        event.quantity  = trade.getQuantity().doubleValue();
        return trade;
    } finally {
        event.commit();  // 提交事件到 JFR（如果 Duration 超過閾值才記錄）
    }
}
```

## 四、async-profiler 火焰圖 <!-- 🔴 資深 -->

### 什麼是火焰圖？

```
火焰圖（Flame Graph）：視覺化程式的 CPU 時間分佈

                    ████ doMatch() [30%]
            ████████████ matchOrder() [60%]
    ████████████████████ MatchingEngine.process() [90%]
████████████████████████ Thread-1 [100%]

X 軸：表示方法在呼叫棧中的寬度 = CPU 時間佔比
Y 軸：呼叫棧深度（底部是入口，頂部是葉函數）

最寬的「平頂」= 真正花時間的地方 → 這就是熱點！
```

```bash
# 下載 async-profiler（支援 Linux/macOS）
# https://github.com/async-profiler/async-profiler

# 對運行中的 Java 進程生成火焰圖（30 秒採樣）
./profiler.sh -d 30 -f flamegraph.html <PID>

# 採樣類型：
./profiler.sh -e cpu -d 30 -f cpu.html <PID>    # CPU 熱點
./profiler.sh -e alloc -d 30 -f alloc.html <PID> # 記憶體分配熱點（哪裡在瘋狂 new 物件？）
./profiler.sh -e lock -d 30 -f lock.html <PID>   # Lock 競爭（哪個鎖是瓶頸？）

# 在容器（Docker/Kubernetes）中使用：
# 需要添加 --privileged 或 SYS_PTRACE capability
```

### 閱讀火焰圖的技巧

```
1. 找最寬的「平台」（Plateau）：
   如果有一個方法很寬但頂部是平的（沒有子調用），
   說明這個方法本身就在消耗 CPU，不是因為它的子函數

2. 找意外出現的方法：
   GC 相關方法（java.lang.ref.*, gc）出現在火焰圖上 → 頻繁 GC！
   Lock.park(), Object.wait() → 大量線程等待鎖！
   sun.nio.ch.* → I/O 阻塞！

3. 看記憶體分配火焰圖（-e alloc）：
   最寬的方法 = 分配最多物件的地方
   → 可以用物件池或減少建立來優化 GC 壓力
```

## 五、GC 日誌分析 <!-- 💡 進階 -->

```bash
# 開啟 GC 日誌（Java 11+）
java -Xlog:gc*:file=gc.log:time,uptime:filecount=5,filesize=20m \
     -jar matching-engine.jar

# GC 日誌範例：
# [2024-01-01T12:00:01.123] GC(42) Pause Young (Normal) (G1 Evacuation Pause) 512M->256M(2048M) 12.345ms
#                                                                               ↑before↑after↑heap  ↑pause

# 關鍵指標：
# Pause time（停頓時間）：> 200ms 需要關注（撮合引擎要求 < 1ms！）
# GC frequency（GC 頻率）：太頻繁說明堆太小或分配過多
# Heap usage trend：是否一直在增加（記憶體洩漏）
```

```bash
# GC 日誌分析工具：GCEasy (gceasy.io) 或 GCViewer

# 常見 GC 問題的症狀和解法：
#
# 症狀：Pause time > 500ms
# 原因：堆太大，GC 一次要掃描太多
# 解法：啟用 G1/ZGC，設定更小的暫停目標
# java -XX:+UseG1GC -XX:MaxGCPauseMillis=200

# 症狀：GC 非常頻繁（每秒幾次）
# 原因：Eden 區太小，短命物件太多
# 解法：增大堆記憶體 -Xmx；用物件池減少建立
# java -Xms4g -Xmx4g  # 堆初始=最大，避免動態擴展的停頓

# 症狀：Full GC 頻繁且慢
# 原因：老年代（Old Gen）不夠大，或有記憶體洩漏
# 解法：增大老年代；用 MAT 分析 Heap Dump 找洩漏點
```

## 六、執行緒問題排查 <!-- 🔴 資深 -->

### Thread Dump 分析

```bash
# 取得 Thread Dump（不需要重啟）
jstack <PID> > threaddump.txt
# 或
kill -3 <PID>  # 傳送 SIGQUIT 信號，JVM 輸出 Thread Dump 到 stdout

# 要找的模式：

# 1. 死鎖（Deadlock）
# Thread Dump 最後會有：
# "Found one Java-level deadlock:"
# "Thread-A" waiting to lock 0x00000006c7e01b88, held by "Thread-B"
# "Thread-B" waiting to lock 0x00000006c7e01c98, held by "Thread-A"

# 2. 大量 BLOCKED 線程（Lock 競爭）
# "Thread-XX" #5 daemon prio=5
#    java.lang.Thread.State: BLOCKED (on object monitor)
#    waiting to lock <0x...> (a java.util.HashMap)
# → 找到鎖住的物件，是哪個類？考慮改用 ConcurrentHashMap

# 3. 線程洩漏（線程一直在增加）
# 用 jstack 定期取 dump，比較線程數量
# 或用 /actuator/metrics/jvm.threads.live 監控
```

```java
// 撮合引擎排查：某時刻 P99 突然升高
// 步驟：
// 1. 取 Thread Dump
// 2. 搜尋 BLOCKED 線程
// 3. 找到鎖競爭的物件

// 例如發現：
// "matching-thread-1" BLOCKED on java.util.TreeMap@0x...
// "matching-thread-2" BLOCKED on java.util.TreeMap@0x...
// "matching-thread-3" BLOCKED on java.util.TreeMap@0x...

// → TreeMap 不是線程安全的！多線程訪問導致競爭
// 解法：改用 ConcurrentSkipListMap，或加明確的讀寫鎖
```

## 七、效能優化決策樹

```
系統變慢了？

↓ 第一步：量測，找瓶頸
  CPU 高？→ async-profiler CPU 火焰圖，找熱點方法
  記憶體高？→ async-profiler alloc 火焰圖，找分配熱點；MAT 分析 Heap Dump
  I/O 高？→ async-profiler 看 I/O 相關方法；考慮快取
  DB 慢？→ 慢查詢日誌 + EXPLAIN 分析
  GC 停頓？→ GC 日誌分析；考慮 G1/ZGC
  Lock 競爭？→ async-profiler lock 火焰圖；Thread Dump

↓ 第二步：針對瓶頸優化

CPU 熱點：
  └── 算法複雜度高？→ 換更好的算法（O(n²) → O(n log n)）
  └── 序列化/反序列化太多？→ 換 Protobuf；減少轉換次數
  └── 正則表達式？→ 預先編譯（Pattern.compile 快取）

記憶體分配過多：
  └── 短命物件太多（字串拼接？）→ StringBuilder；對象池
  └── 大集合被頻繁複製？→ 原地修改；傳引用
  └── 快取沒有限制大小？→ LRU 快取（Guava Cache / Caffeine）

資料庫慢：
  └── 全表掃描？→ 加索引
  └── N+1 查詢？→ JOIN 或 @EntityGraph（第27章）
  └── 連線池不夠？→ 增大 HikariCP maxPoolSize

↓ 第三步：驗證
  用 JMH 或負載測試確認改善幅度
  確認 P50/P99 延遲都有改善（不只是平均值）
  確認沒有引入新問題（記憶體洩漏？更多 GC？）
```

## 八、本章總結

| 工具 | 用途 | 開銷 | 適合場景 |
|------|------|------|---------|
| JMH | 微基準測試 | 無（開發時用） | 比較兩個實作哪個快 |
| JFR | 全面 JVM 監控 | < 2% CPU | 生產環境持續開著 |
| async-profiler | CPU/記憶體/Lock 熱點 | 低（採樣式） | 發現效能問題時觸發 |
| jstack | Thread Dump | 瞬間開銷 | 排查線程死鎖/Block |
| GC 日誌 | GC 行為分析 | 低 | JVM 調優，減少停頓 |
| MAT | Heap Dump 分析 | 事後分析 | 排查記憶體洩漏 |

```
效能工程師的心法：
  「數字不說謊，感覺會騙人」
  永遠先量測，再優化
  優化後再量測，確認效果
  優化的目標是改善 P99，不是平均值
```
