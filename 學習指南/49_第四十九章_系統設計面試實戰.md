# 第四十九章：系統設計面試實戰

## 前言：系統設計考的是什麼？

系統設計面試不是考你背答案，而是考你：

1. **需求分析**：能不能把模糊的問題變成具體的技術規格
2. **數量估算**：能不能用 Back-of-envelope 計算出 QPS/儲存量
3. **架構選型**：知道什麼情況用什麼技術，知道取捨（tradeoff）
4. **深度討論**：面試官問「這樣做有什麼問題？」時，你能說出來

這章用 4 道經典題目示範整個思考流程。

---

## 一、答題框架（記這個）

```
1. 澄清需求（2-3 分鐘）
   - 功能需求：核心功能是什麼？哪些可以不做？
   - 非功能需求：規模（DAU）、延遲要求、一致性要求？

2. 數量估算（3-5 分鐘）
   - 用戶量 → 每日活躍用戶（DAU）
   - QPS（每秒請求數）= DAU × 每人每天操作次數 ÷ 86400
   - 儲存空間 = 每筆資料大小 × 每日新增量 × 保存天數

3. 高階架構（5-10 分鐘）
   - 畫出主要元件：Client / API Server / DB / Cache / MQ
   - 確定主要的讀寫流程

4. 深入設計（15-20 分鐘）
   - 最複雜的核心功能怎麼實作？
   - 資料庫 Schema / 索引設計
   - 如何處理高並發？如何保證一致性？

5. 瓶頸與優化（5 分鐘）
   - 如果 QPS 再大 10 倍，哪裡會是瓶頸？
   - 如何做到 99.9% 可用性？
```

## 二、設計 URL 短縮服務（如 bit.ly）

### 需求澄清

**功能：**
- 輸入長 URL，回傳短碼（如 `https://bit.ly/3xY9kP`）
- 點擊短碼，302 重定向到原始 URL
- 可選：短碼自訂、設過期時間、點擊統計

**非功能：**
- 100M DAU，每人每天生成 1 條 + 點擊 10 條
- 寫入 QPS：100M / 86400 ≈ **1,200 QPS**
- 讀取 QPS：1M / 86400 ≈ **12,000 QPS**（讀多寫少，10:1）
- 每條 URL 100 bytes，保存 5 年：1200 × 86400 × 365 × 5 × 100B ≈ **18 TB**

### 核心演算法：如何生成短碼？

**方案一：MD5 截取（不推薦）**
```java
// 問題：衝突時難以處理；相同 URL 得到相同短碼（可能是需求，也可能不是）
String shortCode = DigestUtils.md5Hex(longUrl).substring(0, 7);
```

**方案二：全局自增 ID + Base62 編碼（推薦）**
```java
// Base62 = 0-9 + a-z + A-Z = 62 個字符
// 7 位 Base62 = 62^7 = 3.5 兆，夠用幾十年
public class Base62Encoder {
    private static final String CHARS = "0123456789abcdefghijklmnopqrstuvwxyzABCDEFGHIJKLMNOPQRSTUVWXYZ";

    public static String encode(long id) {
        StringBuilder sb = new StringBuilder();
        while (id > 0) {
            sb.append(CHARS.charAt((int)(id % 62)));
            id /= 62;
        }
        // 反轉 + 補齊到 7 位
        return sb.reverse().toString();
    }

    public static long decode(String code) {
        long id = 0;
        for (char c : code.toCharArray()) {
            id = id * 62 + CHARS.indexOf(c);
        }
        return id;
    }
}
```

```
全局 ID 生成器（Snowflake 或資料庫 AUTO_INCREMENT）
   ID = 1000000000 → Base62 → "dNPlLx"（7位）
```

### 系統架構

```
用戶 → API Gateway → 短網址服務
                         │
                    ┌────┴────┐
              [Write Path]  [Read Path]
                    │            │
              生成短碼        查 Redis 快取
                    │        (hit: 直接重定向)
              存入 MySQL      │
              存入 Redis   (miss: 查 MySQL → 存 Redis)
```

### 資料庫設計

```sql
-- 核心表：URL 映射
CREATE TABLE url_mapping (
    id          BIGINT PRIMARY KEY AUTO_INCREMENT,
    short_code  VARCHAR(10) NOT NULL UNIQUE,   -- 短碼（索引）
    long_url    TEXT NOT NULL,
    user_id     BIGINT,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP,
    expires_at  DATETIME,                       -- NULL = 永不過期
    click_count BIGINT DEFAULT 0,
    INDEX idx_short_code (short_code),
    INDEX idx_expires_at (expires_at)           -- 用於清理過期資料
);
```

### 讀取快取（重點）

```java
// 讀取流程：Redis 快取 → MySQL
@Service
public class UrlRedirectService {

    private static final Duration CACHE_TTL = Duration.ofHours(24);

    public String getLongUrl(String shortCode) {
        // 1. 查 Redis
        String cacheKey = "url:" + shortCode;
        String longUrl = redisTemplate.opsForValue().get(cacheKey);

        if (longUrl != null) {
            // 快取命中：非同步更新點擊數（不影響重定向速度）
            CompletableFuture.runAsync(() -> incrementClickCount(shortCode));
            return longUrl;
        }

        // 2. 快取未命中：查資料庫
        UrlMapping mapping = urlRepository.findByShortCode(shortCode)
            .orElseThrow(() -> new NotFoundException("短碼不存在"));

        // 3. 結果存入 Redis
        redisTemplate.opsForValue().set(cacheKey, mapping.getLongUrl(), CACHE_TTL);

        return mapping.getLongUrl();
    }
}
```

### 面試加分討論

**Q：如何處理熱門短碼（如名人分享，瞬間百萬請求）？**
> Redis 多副本讀取 + 本地快取（Caffeine）兩級快取。應用伺服器本地緩存最熱的 1000 個短碼，連 Redis 都不用打。

**Q：資料庫寫入怎麼擴展？**
> 短碼生成用 Snowflake（不依賴 DB），然後非同步批次寫入 MySQL。

## 三、設計推特（Twitter / X）時間軸

### 需求澄清

**功能：**
- 發推（140字 + 圖片）
- 關注用戶
- 查看 Home Timeline（關注的人最新的推文）
- 查看 User Timeline（某人的所有推文）

**規模：**
- 300M DAU；發推 QPS：6,000；讀 Timeline QPS：300,000（讀寫比約 50:1）
- 每條推文 200B，每天 500M 條：**100 GB/天**

### 核心難題：時間軸如何生成？

**方案一：Pull（查詢時生成）**
```
查詢 Home Timeline:
  1. 找出我關注的所有人（假設 200 人）
  2. 查每個人最新的推文 → 200 次 DB 查詢
  3. 合併、排序，取前 20 條
問題：200 次查詢太慢，用戶一多完全撐不住
```

**方案二：Push（發推時預先寫入）- Twitter 早期方案**
```java
// 發推時：把推文「推送」到每個粉絲的 Timeline 快取
@Service
public class TweetService {

    public void createTweet(Long userId, String content) {
        // 1. 儲存推文
        Tweet tweet = tweetRepository.save(new Tweet(userId, content));

        // 2. 找出所有粉絲（非同步執行）
        CompletableFuture.runAsync(() -> {
            List<Long> followerIds = socialGraph.getFollowers(userId);

            // 3. 把推文 ID 推入每個粉絲的 Timeline Redis List
            for (Long followerId : followerIds) {
                String key = "timeline:" + followerId;
                redisTemplate.opsForList().leftPush(key, tweet.getId());
                redisTemplate.opsForList().trim(key, 0, 799);  // 只保留最新 800 條
            }
        });
    }
}
```

```
查詢 Home Timeline:
  1. 讀取 Redis List "timeline:{userId}" → 20 個推文 ID（O(1)，極快）
  2. 批次查推文詳情（Redis Hash 或 DB）
```

**缺陷：明星問題（Celebrity Problem）**

Lady Gaga 有 8000 萬粉絲，每發一條推文要寫入 8000 萬個 Redis key，會造成延遲尖峰。

**方案三：混合方案（Twitter 現在的做法）**

```
普通用戶（粉絲 < N 萬）→ Push（預計算時間軸）
明星用戶（粉絲 > N 萬）→ Pull（查詢時實時計算）

查詢時：
  Timeline = 我關注的普通人（已預計算） + 實時查詢我關注的明星的最新推文
           → 合併排序 → 最終結果
```

### 架構圖

```
              ┌─────────┐
用戶發推 ─────→│ Tweet   │───── Kafka ────→ Fan-out Service
              │ Service │                       │
              └─────────┘                  （普通用戶：Push）
                   │                            │
              推文存 MySQL                 Redis List
                                          timeline:{id}

              ┌──────────────┐
用戶讀取 ─────→│ Timeline     │
              │ Service      │
              └──────────────┘
                   │
              讀 Redis List（普通人推文）
              + 實時查明星最新推文
              → 合併排序
```

## 四、設計訂單系統（電商高並發）

### 秒殺場景的核心挑戰

1000 件庫存，10 萬人同時搶購。如何保證：
- 不超賣（賣出不超過 1000 件）
- 不少賣（不因為鎖競爭導致本可以賣的賣不出去）
- 用戶體驗快（不能讓每個請求都等 DB 鎖）

### 解法：Redis Lua 原子扣減 + 非同步下單

**第一階段：Redis 原子扣減庫存（在 DB 之前）**

```lua
-- stock_deduct.lua
-- KEYS[1] = "stock:{productId}"
-- ARGV[1] = 購買數量
local stock = tonumber(redis.call('GET', KEYS[1]))
if stock == nil then
    return -1  -- 商品不存在
end
if stock < tonumber(ARGV[1]) then
    return -2  -- 庫存不足
end
redis.call('DECRBY', KEYS[1], ARGV[1])
return 1  -- 成功
```

```java
@Service
public class SeckillService {

    public OrderResult seckill(Long userId, Long productId, int quantity) {
        // 1. Redis 原子扣減（微秒級，拒絕超賣）
        Long result = (Long) redisTemplate.execute(
            stockDeductScript,
            List.of("stock:" + productId),
            String.valueOf(quantity)
        );

        if (result == -2L) {
            return OrderResult.OUT_OF_STOCK;
        }

        // 2. 防重：同一個用戶對同一個商品只能搶一次
        boolean isNew = redisTemplate.opsForSet()
            .add("seckill:buyers:" + productId, userId.toString()) == 1L;
        if (!isNew) {
            // 退還庫存
            redisTemplate.opsForValue().increment("stock:" + productId, quantity);
            return OrderResult.DUPLICATE;
        }

        // 3. 發 Kafka 非同步建立真實訂單（DB 操作移到後台）
        SeckillMessage msg = new SeckillMessage(userId, productId, quantity);
        kafkaTemplate.send("seckill-orders", msg);

        return OrderResult.SUCCESS;
    }
}
```

**第二階段：Kafka Consumer 非同步落庫**

```java
@KafkaListener(topics = "seckill-orders", concurrency = "3")
public void processSeckillOrder(SeckillMessage msg) {
    // 用資料庫 WHERE stock >= quantity 作為最後防線（以防 Redis 和 DB 不一致）
    int updated = productRepository.deductStock(
        msg.getProductId(), msg.getQuantity()
    );

    if (updated > 0) {
        Order order = orderRepository.save(Order.of(msg));
        notificationService.notifySuccess(msg.getUserId(), order.getId());
    } else {
        // DB 庫存不足（極少發生），退款補償
        redisTemplate.opsForValue().increment("stock:" + msg.getProductId(), msg.getQuantity());
        notificationService.notifyFailed(msg.getUserId(), "庫存不足");
    }
}
```

```sql
-- 資料庫扣減（以 WHERE stock >= 購買數量 防止超賣）
UPDATE products
SET stock = stock - #{quantity}
WHERE id = #{productId} AND stock >= #{quantity}
```

### 整體流量漏斗

```
10 萬個請求
    │
    ▼ Nginx 限流（令牌桶，每秒最多 5000）
5,000 QPS
    │
    ▼ Redis 庫存檢查（微秒級，1000 通過，其餘直接拒絕）
1,000 通過
    │
    ▼ Kafka 非同步佇列
    │
    ▼ 3 個 Consumer 處理，每秒 ~300 筆 DB 寫入
300 QPS → MySQL 完全承受得了
```

## 五、設計抖音/YouTube 影片推薦系統 <!-- 🔴 資深 -->

### 推薦系統的基本架構

```
              離線部分（每天跑一次）
┌─────────────────────────────────────────────────┐
│  用戶行為數據（Kafka → Hadoop/Spark）             │
│  ↓                                              │
│  協同過濾 / 矩陣分解 / 深度學習模型               │
│  ↓                                              │
│  預計算候選集：推薦給每個用戶的 1000 個影片 ID    │
│  → 存入 Redis / 推薦資料庫                       │
└─────────────────────────────────────────────────┘

              線上部分（用戶刷新時）
用戶請求 → 召回（Retrieval）→ 排序（Ranking）→ 過濾（Filter）→ 返回 20 個影片
```

**召回（Retrieval）：** 從幾千萬個影片中，快速找出幾千個「可能感興趣」的候選影片
- 協同過濾：喜歡這個的人也喜歡那個
- 內容相似：和你最近看的影片類似
- 熱門推薦：全局熱門的影片

**排序（Ranking）：** 對候選集做精細排序（點擊率預測模型），找出最可能互動的 20 個

**過濾（Filter）：** 去掉已看過的、違規的、作者封禁的

### 關鍵設計：向量相似度搜索

現代推薦系統用「向量表示」（Embedding）來計算相似度：

```java
// 每個影片和用戶都用一個 128 維向量表示
// 向量點積越大 = 越相似
// 餘弦相似度 = A·B / (|A| × |B|)

// ANN（近似最近鄰）搜索：FAISS、Milvus
// 不用遍歷所有影片，毫秒級找出最相似的 Top-K
```

## 本章練習題

**Q1：設計 URL 短縮服務時，為什麼建議用 302 重定向而不是 301？**
<details>
<summary>答案</summary>
301 是永久重定向，瀏覽器會快取結果，下次直接走到目標 URL 不再請求短網址服務。這樣你就無法統計點擊次數了。302 是暫時重定向，瀏覽器不快取，每次都會請求短網址服務，可以記錄每次點擊。如果你不需要統計功能，301 可以降低服務器負載。
</details>

**Q2：推特的 Push 方案中，「明星問題」的閾值 N 大概設多少合理？**
<details>
<summary>答案</summary>
Twitter 實際上大約用 100 萬粉絲作為閾值，超過這個數量的用戶就改為 Pull。閾值需要根據服務器容量和延遲要求動態調整。太低：大量用戶變成 Pull，讀取時需要實時計算，增加讀延遲；太高：Push 佔用太多 Redis 空間和計算資源。
</details>

**Q3：秒殺系統中，Redis 扣減成功但 Kafka 發送失敗，怎麼辦？**
<details>
<summary>答案</summary>
這是分散式事務問題。解法是 Outbox Pattern：不直接發 Kafka，而是在 Redis 操作成功後，把「待發送消息」記錄到一個持久化存儲（DB 或 Redis List）。一個獨立的 Relay 服務輪詢這個列表，確保消息被發出去。或者用本地事務：Redis 操作成功後，先把消息記到 DB（和庫存更新同一個事務），再由 CDC（如 Debezium）捕捉 DB 變更發到 Kafka。
</details>
