# 第五十六章：系統設計方法論

## 前言：為什麼系統設計沒有標準答案？

系統設計面試不像演算法題有唯一正解。面試官要看的是**你的思考過程**：你懂不懂取捨（tradeoff）、能不能把模糊需求轉化成具體決策、知不知道什麼情況用什麼工具。

這章是工具箱。先把工具備齊，再去 ch.49 和 ch.57 練習題目。

## 一、數量估算：先算再設計

不做估算就開始設計，等於不知道要建幾線道就開始蓋高速公路。

### 必記的硬體基準數字

```
儲存：
  1 個字元    = 1 Byte
  1 張縮圖    ≈ 100 KB
  1 張高清圖  ≈ 1 MB
  1 分鐘影片  ≈ 50 MB（720p）

速度：
  L1 Cache 讀取   ≈ 0.5 ns
  記憶體讀取      ≈ 100 ns（比 L1 慢 200x）
  SSD 順序讀取    ≈ 1 μs（比記憶體慢 10x）
  HDD 尋軌時間    ≈ 10 ms（比 SSD 慢 10,000x）
  同城網路延遲    ≈ 0.5 ms
  跨洲網路延遲    ≈ 150 ms

時間換算：
  1 天 = 86,400 秒 ≈ 10^5 秒（記這個！）
```

### 估算流程：DAU → QPS → 儲存

```
題目：設計一個每月活躍用戶 3 億的推特

步驟一：換算 DAU
  MAU 3 億，DAU ≈ MAU × 50% = 1.5 億 DAU

步驟二：計算 QPS（每秒請求數）
  假設每個 DAU 每天：發推 1 次 + 讀 Timeline 10 次
  寫入 QPS = 1.5 億 × 1 / 86,400 ≈ 1,700 QPS
  讀取 QPS = 1.5 億 × 10 / 86,400 ≈ 17,000 QPS
  峰值 ≈ 平均 × 2-3 = 讀取峰值 51,000 QPS

步驟三：計算儲存
  每條推文 ≈ 200 Bytes
  每天新推文 = 1,700 × 86,400 = 1.47 億條
  每天儲存 = 1.47 億 × 200B ≈ 29 GB/天
  10 年 = 29 GB × 365 × 10 ≈ 106 TB

步驟四：頻寬
  讀取每秒傳輸 = 17,000 QPS × 200B ≈ 3.4 MB/s（純文字，很小）
  若帶圖片（每圖 1MB）= 17,000 × 1MB ≈ 17 GB/s（需要 CDN！）
```

### 快速估算口訣

| 場景 | 數量級估算 |
|------|-----------|
| 每天讀取 QPS | DAU × 每人每天請求數 ÷ 10^5 |
| 寫入通常是讀取的 | 1/10 ~ 1/100 |
| 峰值 ≈ 平均 | × 2~3 |
| 一台普通後端服務器 | 承受 ~1,000 QPS |
| 一台 Redis | 承受 ~100,000 QPS |
| 一個 MySQL 主庫 | 安全寫入 ~2,000 QPS |

## 二、儲存選型決策樹

這是系統設計中最常被問到的問題之一：「你會用什麼資料庫？」

```
需要儲存什麼？
│
├── 結構化資料（表格、關聯）
│   ├── 需要強一致性（ACID）？ → MySQL / PostgreSQL
│   └── 需要無限水平擴展？    → CockroachDB / TiDB（分散式 SQL）
│
├── 半結構化（欄位不固定 / 嵌套）
│   └── MongoDB（文件資料庫）
│
├── 鍵值對（快速讀取）
│   ├── 需要持久化？     → Redis（開啟 AOF）
│   └── 純快取？        → Redis / Memcached
│
├── 時序資料（感測器、監控 metrics）
│   └── InfluxDB / Prometheus / TimescaleDB
│
├── 全文搜尋
│   └── Elasticsearch
│
├── 圖資料（社交關係、推薦）
│   └── Neo4j / Amazon Neptune
│
└── 大型 Binary（圖片、影片、檔案）
    └── Object Storage（AWS S3 / GCP GCS）
        不要存到資料庫！DB 只存 URL
```

### 常見組合模式

```
電商標準搭配：
  MySQL（訂單/帳戶）+ Redis（快取/Session/庫存）
  + Elasticsearch（商品搜尋）+ MongoDB（商品目錄）
  + S3（商品圖片）+ Kafka（事件流）

社交平台：
  MySQL（用戶資料）+ Redis（Timeline 快取/計數器）
  + Cassandra / HBase（大規模訊息歷史）+ S3（媒體檔案）
  + Elasticsearch（搜尋用戶/內容）
```

## 三、從 1 個用戶到 1 億用戶的擴展路線 <!-- 💡 進階 -->

這是一道常見的開放性題：「你的系統如何從 0 擴展到 1 億用戶？」

### 階段一：1 ~ 1 萬用戶（單機）

```
用戶 → Nginx → Spring Boot App → MySQL
                                  └── 同一台機器

不用想太多，先讓它跑起來。
瓶頸：App 和 DB 在同一台機器互搶資源。
解法：把 DB 移到獨立機器。
```

### 階段二：1 萬 ~ 10 萬用戶（讀寫分離）

```
用戶 → Load Balancer → App Server × 2
                           │
                    MySQL Primary（寫）
                           │ 複製
                    MySQL Replica × 2（讀）

加入 Redis 快取熱門資料（如商品詳情、用戶 Profile）
80% 的讀取打到 Redis，DB 壓力大幅降低。
```

### 階段三：10 萬 ~ 100 萬用戶（引入 CDN + 非同步）

```
靜態資源（圖片、JS、CSS）→ CDN（CloudFront / Cloudflare）
                            不再打到你的 App Server

耗時操作（發信、生成報表）→ Kafka → Worker 非同步處理
                            API 立刻回應，用戶不用等

Session 狀態 → Redis（讓 App Server 可以無狀態水平擴展）
```

### 階段四：100 萬 ~ 1000 萬用戶（微服務化）

```
把單體按功能拆分：
  User Service    → 獨立資料庫、獨立部署
  Order Service   → 獨立資料庫、獨立部署
  Payment Service → 獨立資料庫、獨立部署

API Gateway 統一入口（認證、限流、路由）
服務間通信：gRPC（內部）/ Kafka（非同步事件）
```

### 階段五：1000 萬 ~ 1 億用戶（資料庫分片）

```
單一 MySQL 寫入已是瓶頸：
  按用戶 ID 水平分片（Sharding）
  User ID 0-999 萬 → Shard 1
  User ID 1000-1999 萬 → Shard 2
  ...

每個服務獨立 K8s Deployment，HPA 自動擴縮容
多地域部署（Multi-Region），就近服務用戶
```

## 四、常見架構積木

設計時不用從零開始，這些是標準「積木」，知道每個積木解決什麼問題：

| 積木 | 解決的問題 | 代表工具 |
|------|-----------|---------|
| Load Balancer | 流量分散到多台 Server | Nginx、AWS ALB |
| CDN | 靜態資源加速、就近服務 | Cloudflare、CloudFront |
| API Gateway | 統一入口、認證、限流、路由 | Spring Cloud Gateway、Kong |
| Cache | 降低 DB 壓力、加速讀取 | Redis、Memcached |
| Message Queue | 解耦、削峰、非同步 | Kafka、RabbitMQ |
| Search Engine | 全文搜尋、複雜查詢 | Elasticsearch |
| Object Storage | 圖片/影片/檔案儲存 | AWS S3、GCP GCS |
| Service Discovery | 服務地址動態管理 | Nacos、Consul |
| Circuit Breaker | 防止服務雪崩 | Resilience4j |
| Distributed Lock | 跨服務互斥 | Redis SETNX、Zookeeper |

## 五、一致性選型：強一致 vs 最終一致 <!-- 🔴 資深 -->

```
強一致性（Strong Consistency）：
  讀取一定能看到最新的寫入
  代價：延遲高、可用性低
  適合：銀行轉帳、庫存扣減、訂單狀態

最終一致性（Eventual Consistency）：
  寫入後，過一段時間所有節點才看到最新值
  代價：可能讀到舊資料
  適合：社交按讚數、瀏覽量、用戶 Profile

因果一致性（Causal Consistency）：中間地帶
  有因果關係的操作保持順序
  「我發了一篇文章，我一定能看到自己的文章」
  適合：社交動態、留言系統
```

**設計時的問題清單：**

```
1. 如果用戶看到一秒前的舊資料，業務上接受嗎？
   → 接受 → 最終一致，可用性更高
   → 不接受 → 強一致，準備付出延遲和複雜度的代價

2. 這個操作失敗了，重試安全嗎？（冪等性）
   → 要設計冪等 ID 防重複扣款/下單

3. 多個服務需要同時成功或失敗嗎？
   → 需要 → Saga / TCC（ch.41）
```

## 六、效能數量級心理模型 <!-- 🔴 資深 -->

回答「這個設計能承受多大流量？」需要直覺：

```
單台機器的極限（大概數字，供估算）：
  Nginx（反向代理）    → ~50,000 QPS
  Spring Boot（業務邏輯）→ ~2,000 QPS（I/O 密集）
                          ~10,000 QPS（虛擬執行緒 Java 21）
  Redis（讀）          → ~100,000 QPS
  MySQL（讀，有索引）   → ~10,000 QPS
  MySQL（寫）          → ~2,000 QPS
  Kafka（生產）        → ~500,000 QPS

加一層快取後的效果：
  Redis 命中率 90% → DB 流量降低 10 倍
  CDN 命中率 95%  → 源站流量降低 20 倍
```

## 七、系統設計的常見陷阱

### 陷阱一：過早優化（最常見）

```
面試官：「設計一個有 100 個用戶的系統」
❌ 錯誤回答：「我要用 Kafka + 微服務 + Kubernetes + 分庫分表...」
✅ 正確思路：先設計最簡單可以跑的，再說「當用戶到 10 萬時，我會在 X 地方做 Y 優化」
```

### 陷阱二：單點故障（SPOF）

```
任何地方只有「一台」就是潛在的 SPOF：
  一台 Nginx → 加備援
  一台 MySQL → 主從複製 + Failover
  一個機房   → 多地域部署
```

### 陷阱三：沒考慮資料一致性

```
常見問題：
  「Redis 快取更新了，MySQL 還沒更新」
  「A 服務扣款成功，B 服務發貨失敗」

設計時要說清楚：
  用 Cache-Aside（旁路快取）還是 Write-Through？
  跨服務操作用 Saga 補償還是 TCC 兩階段？
```

### 陷阱四：忽略監控與告警

面試官很欣賞你說：「我會在 X 地方埋 Prometheus 指標，當 Y 超過閾值時觸發 PagerDuty 告警。」這代表你有生產意識。

## 八、一頁式系統設計答題模板

```
【1. 需求確認（2 分鐘）】
功能需求：核心功能是 A, B, C；先不做 D, E
非功能需求：DAU __，P99 延遲 < __ms，可用性 99.9%

【2. 估算（3 分鐘）】
QPS：___ （讀）/ ___ （寫）
儲存：___ GB/天，5 年 ≈ ___ TB

【3. 高階架構（5 分鐘）】
[畫出：Client → API Gateway → 核心服務 → DB/Cache]
說明主要讀路徑和寫路徑

【4. 深入設計（15 分鐘）】
核心功能 1 的詳細實作
  - DB Schema
  - 快取策略
  - 一致性保證
核心功能 2 的詳細實作...

【5. 擴展性與瓶頸（5 分鐘）】
目前設計的瓶頸在哪？（主動說出來更加分）
如果 QPS × 10，哪裡要改？
如何達到 99.99% 可用性？

【6. 加分討論】
監控：關鍵 Metrics 是什麼？
安全：如何防止 DDoS / 資料洩漏？
成本：哪裡可以降低雲端費用？
```

## 本章練習題

**Q1：一個系統的 P99 延遲是 200ms，P999 是 2 秒，這代表什麼？有問題嗎？**
<details>
<summary>答案</summary>
P99 = 200ms 表示 99% 的請求在 200ms 內完成，P999 = 2 秒表示 99.9% 的請求在 2 秒內完成。P99 和 P999 差了 10 倍，這是一個警訊——代表有「長尾延遲」（tail latency）問題，少數請求非常慢。原因可能是 GC 停頓、資料庫鎖等待、網路抖動。對於金融系統，P999 2 秒絕對不可接受（用戶交易卡住）；對於一般 Web 服務，要評估那 0.1% 的用戶對業務的影響。
</details>

**Q2：你的系統讀取 QPS 是 50,000，Redis 快取命中率 90%，實際打到 MySQL 的 QPS 是多少？如果要降到 1,000 以下，命中率需要達到多少？**
<details>
<summary>答案</summary>
打到 MySQL 的 QPS = 50,000 × (1 - 0.9) = 5,000 QPS。
要降到 1,000：命中率需要 = 1 - (1,000 / 50,000) = 98%。
實務上從 90% 提升到 98% 很難，因為最後幾個百分點是「長尾數據」（冷門商品、很少查的用戶）。可以改為 MySQL 讀副本 + 分佈讀取壓力，或改進快取淘汰策略（從 LRU 換成 LFU）。
</details>

**Q3：為什麼說「不要把圖片存在資料庫裡」？**
<details>
<summary>答案</summary>
三個原因：1) 效能：DB 是為結構化資料優化的，大型 Binary 會佔滿 Buffer Pool，擠掉有用的熱點資料的快取，導致正常查詢變慢。2) 備份：DB 備份會包含所有圖片，備份/還原速度慢，費用高。3) CDN：圖片需要 CDN 加速和邊緣分發，Object Storage（S3）天然與 CDN 整合；DB 裡的圖片無法直接接 CDN。正確做法：圖片存 S3，DB 只存 S3 的 URL（幾十個字元）。
</details>
