## 一、設計即時聊天系統（WhatsApp / LINE）

### 需求確認

**功能需求：**
- 一對一私訊（文字 + 圖片）
- 群組聊天（最多 500 人）
- 訊息送達回執（✓ 已送達、✓✓ 已讀）
- 離線訊息推送（App 沒開時用 Push Notification）

**非功能需求：**
- 500M DAU
- 訊息延遲 < 100ms
- 訊息保證不丟失（At-least-once delivery）
- 可用性 99.99%

### 數量估算

```
DAU = 5 億
每人每天傳送 40 條訊息
寫入 QPS = 5 億 × 40 / 86,400 ≈ 230,000 QPS

每條訊息 ≈ 100 Bytes（文字）
每天儲存 = 230,000 × 86,400 × 100B ≈ 2 TB/天
10 年 ≈ 7.3 PB（需要冷熱分層儲存）
```

### 核心難題：訊息如何即時送達？

**WebSocket 長連線是基礎：**

```
用戶 A ─── WebSocket ──→ Chat Server 1
用戶 B ─── WebSocket ──→ Chat Server 2

A 傳訊息給 B：
  A → Chat Server 1（A 連線的 Server）
  但 B 連的是 Chat Server 2！
  → Chat Server 1 怎麼把訊息送到 Chat Server 2？
```

**解法：用 Pub/Sub 解耦 Chat Server**

```
A 傳訊息 → Chat Server 1
    ↓ 發布到 Redis Pub/Sub Channel "chat:userB"
Chat Server 2（訂閱了 "chat:userB"）
    ↓ 收到訊息
    → 透過 WebSocket 推送給 B
```

### 完整架構

```
用戶端                API 層                  服務層              儲存層
─────────             ──────                  ──────             ──────
[App A]──WebSocket──→[Chat Server 1]──────→[Message Service]──→[Cassandra]
                            │               （儲存+路由）         （訊息歷史）
                     [Redis Pub/Sub]
                            │
[App B]──WebSocket──→[Chat Server 2]
                                         [Presence Service]──→[Redis]
                                         （誰在線上？）         （線上狀態）

                                         [Notification Svc]──→[APNs/FCM]
                                         （離線推播）
```

### 為什麼用 Cassandra 而不是 MySQL？

```
訊息歷史的查詢模式：
  "給我 userA 和 userB 的最近 50 條訊息"（按時間排序）

Cassandra 的 Partition Key 設計：
  Partition Key = (sender_id, receiver_id, bucket)
  Clustering Key = message_id DESC（時間倒序）

bucket 是時間桶（如每個月一個 bucket），
防止單一 Partition 無限增長。

每個 bucket 的資料都在同一個節點，查詢極快。
```

### 訊息可靠性：送達回執

```
訊息狀態機：
  SENT（A 的客戶端已發出）
    ↓ Server 確認收到
  DELIVERED（送到 B 的設備）
    ↓ B 開啟聊天視窗
  READ（B 已讀）

實作：
  B 的客戶端收到訊息 → 發 ACK 給 Server → Server 更新狀態 → 通知 A
  用 WebSocket 雙向傳遞 ACK（不用額外 HTTP 請求）
```

## 二、設計雲端儲存（Google Drive / Dropbox）

### 需求確認

**功能需求：**
- 上傳 / 下載檔案
- 多裝置同步
- 檔案版本歷史（最多 30 個版本）
- 資料夾分享與協作

**非功能需求：**
- 50M DAU；每用戶平均儲存 10 GB
- 上傳可靠（斷點續傳）
- 可用性 99.99%（檔案不能丟）

### 估算

```
總儲存 = 50M × 10 GB = 500 PB
每天上傳量（假設 1% 用戶每天上傳 100MB）：
  = 500,000 × 100 MB = 50 TB/天
上傳 QPS = 50M × (1 檔案/週) / 86,400 ≈ 80 QPS（小，瓶頸不在 QPS）
下載 QPS = 上傳 × 10 = 800 QPS（CDN 分擔大部分）
```

### 核心難題一：大檔案如何可靠上傳？

**分塊上傳（Chunked Upload）+ 斷點續傳：**

```
客戶端把 1GB 檔案分成 1000 個 1MB 的 Chunk
  ↓
1. 客戶端請求上傳 Session（Server 回傳 upload_id）
2. 客戶端並行上傳多個 Chunk（帶 chunk_index）
3. Server 記錄哪些 Chunk 已收到
4. 網路斷了？繼續上傳未完成的 Chunk（不用重頭來）
5. 所有 Chunk 收齊後，Server 組合成完整檔案

去重（Deduplication）：
  計算每個 Chunk 的 SHA-256
  如果 Chunk 已存在（其他用戶上傳過），直接複用
  → 相同內容的 Chunk 只存一份，節省大量空間
```

### 核心難題二：多裝置同步

```
裝置 A 修改了 report.xlsx
    ↓
1. A 上傳差異 Chunk（只上傳修改的部分，不是整個檔案）
2. Server 更新 Metadata Service（記錄檔案最新版本）
3. 通知 Sync Service：report.xlsx 有新版本
4. Sync Service 透過 Long Polling / WebSocket 通知裝置 B, C
5. B, C 下載差異 Chunk 並更新本地檔案
```

### 衝突解決（兩台裝置同時修改）

```
裝置 A 和 B 都在離線時修改了同一個檔案，然後同時上傳：
  → 後上傳的會被標記為「衝突版本」
  → 保留兩個版本：report.xlsx 和 report (Conflict B's Copy).xlsx
  → 讓用戶手動合併

（和 Git 的概念相同：需要手動 resolve conflict）
```

### 儲存架構

```
Metadata（檔案名稱、路徑、版本、Chunk 列表）→ MySQL（強一致）
Chunk（實際二進位內容）→ Object Storage（S3）+ CDN（下載加速）
版本歷史 → 只記 Chunk ID 列表的差異（很省空間）
```

## 三、設計 Uber 派車系統（地理位置服務）

### 需求確認

**功能需求：**
- 乘客叫車，顯示附近司機
- 司機接單後，即時更新位置
- 撮合：把最近的空閒司機指派給乘客

**非功能需求：**
- 全球 100M DAU；50M 司機
- 位置更新：每台司機每 4 秒回傳一次位置
- 附近司機查詢延遲 < 500ms
- 精度：1 公尺以內

### 估算

```
司機位置更新 QPS = 50M / 4 ≈ 12,500,000 = 12.5M QPS（超高！）
但：同時在線司機 ≈ 5%：12.5M × 0.05 = 625,000 QPS

乘客查詢附近司機 QPS：
  叫車時觸發，假設 1M DAU 同時叫車 / 60s = 16,700 QPS
```

### 核心難題：如何高效查詢「附近 5km 內的司機」？

**Geohash：把二維座標轉成一維字串**

```
地球表面 → 遞迴二等分 → 用 Base32 編碼
  精度6位 Geohash ≈ 1.2km × 0.6km 的格子
  精度7位 Geohash ≈ 153m × 153m 的格子

查詢附近：
  用戶位置的 Geohash = "wecpkb"
  附近 8 個格子 = ["wecpkb", "wecpkc", "wecpk8", ...]
  查詢這 9 個格子內的所有司機

Redis 實作：
  key = "drivers:geohash:{geohash前綴}"
  value = Set of driver IDs

  司機更新位置：
    計算新 Geohash
    從舊 Geohash 的 Set 移除自己
    加入新 Geohash 的 Set
    設定 TTL = 30 秒（司機若 30 秒沒更新，自動視為離線）
```

### 派車撮合邏輯

```
乘客叫車：
  1. 查詢附近 9 個 Geohash 格子的司機列表
  2. 計算每個司機的實際直線距離（Haversine 公式）
  3. 取最近的 K 個司機，按距離排序
  4. 向最近的司機發送接單請求（Dispatch Service）
  5. 如果 10 秒內沒有回應 → 發給第二近的司機
  6. 司機接單 → 更新狀態為「忙碌」，從可用集合移除

防止多個乘客同時搶同一個司機：
  Redis SETNX "lock:driver:{driverId}" → 原子鎖
  接單成功 → 鎖保持直到行程結束
  接單超時 → 釋放鎖，司機重新可接單
```

## 四、設計 Rate Limiter（限流器）<!-- 💡 進階 -->

### 需求確認

**功能：**
- API 限流：每個 userId 每秒最多 100 次請求
- 全局限流：整個 API 每秒最多 10,000 次
- 超過限制回傳 429 Too Many Requests

### 演算法比較

已在 ch.42 介紹，這裡重點說**分散式場景的實作**：

```java
// Redis + Lua 實作滑動視窗限流（精確，適合分散式）
// 優點：多台 App Server 共用同一個 Redis，計數準確
// 缺點：每次請求都要查 Redis（~0.1ms 額外延遲）

String script = """
    local key = KEYS[1]
    local window = tonumber(ARGV[1])  -- 視窗大小（秒）
    local limit = tonumber(ARGV[2])   -- 限制次數
    local now = tonumber(ARGV[3])     -- 當前時間（毫秒）

    -- 移除視窗外的舊記錄
    redis.call('ZREMRANGEBYSCORE', key, 0, now - window * 1000)

    -- 計算視窗內的請求數
    local count = redis.call('ZCARD', key)

    if count >= limit then
        return 0  -- 超限
    end

    -- 記錄本次請求
    redis.call('ZADD', key, now, now)
    redis.call('EXPIRE', key, window)
    return 1  -- 允許
    """;
```

### 架構位置

```
限流器應該放在哪？

選項一：API Gateway 層（推薦）
  → 一個地方統一控制，不侵入業務程式碼
  → Spring Cloud Gateway 內建 RedisRateLimiter

選項二：應用層（@RateLimiter AOP）
  → 粒度更細（可以按 userId + 方法）
  → 多台 App Server 要共用 Redis，否則各算各的

選項三：服務端點前（Nginx）
  → 最快，但只能做 IP 限流，不能按用戶限
```

## 五、設計搜尋自動補全（Typeahead）<!-- 💡 進階 -->

### 需求確認

**功能：**
- 搜尋框輸入時，即時顯示 10 個建議
- 建議按熱門程度排序（越多人搜的越靠前）
- 延遲 < 100ms（用戶打字速度要跟得上）

### 核心資料結構：Trie（前綴樹）

```
用戶輸入 "jav"：
  Trie 搜尋所有以 "jav" 開頭的詞
    → "java", "javascript", "java 教學", "java 入門"...
  取熱門度最高的 10 個
```

### 實際系統的取捨

**純 Trie 在記憶體裡放不下**（全球所有搜尋詞太多），實際方案：

```
預計算快取（Cache Top-K）：

離線（每小時跑一次 Spark Job）：
  統計過去 24 小時每個前綴的 Top 10 搜尋詞
  存到 Redis Hash：
    key = "typeahead:jav"
    value = ["java", "javascript", "java 教學", ...]

線上（用戶打字時）：
  GET "typeahead:jav" → 直接回傳 Redis 快取
  P99 延遲 < 5ms（遠小於 100ms 要求）

優缺點：
  優點：極快，實作簡單
  缺點：每小時更新一次，熱門趨勢有延遲（突發新聞來不及反映）
  解法：熱門前綴用 Kafka 即時更新，冷門前綴用離線更新
```

## 本章練習題

**Q1：WhatsApp 的群組訊息（500 人）和一對一訊息的架構有什麼不同？**
<details>
<summary>答案</summary>
一對一：Server 只需要把訊息路由給一個接收者，用 Redis Pub/Sub "chat:userB" 即可。群組（500人）：如果也用 Pub/Sub，一條訊息要推送到 500 個 WebSocket 連線——500 人都在線時沒問題，但如果 Server 逐一 push 會產生 fan-out 風暴。

實際做法（WhatsApp 採用）：群組訊息只存一份在 Server，不做 fan-out。每個成員的客戶端主動「拉取」（pull）自己尚未讀取的訊息。這樣寫入是 O(1)，讀取是各自按需拉取。Inbox 模式（每個用戶有自己的訊息收件箱）則把訊息複製到每個成員的收件箱，適合小群組但 500 人群組寫放大 500 倍。
</details>

**Q2：Google Drive 的「多裝置同步」如何偵測到哪些檔案發生了變更？**
<details>
<summary>答案</summary>
客戶端方面：監聽作業系統的檔案系統事件（inotify / FSEvents / ReadDirectoryChangesW），當檔案被建立/修改/刪除時立即通知同步服務。伺服器方面：用 Long Polling 或 WebSocket 保持連線，Server 有變更時主動推播通知給其他裝置。這樣不需要客戶端一直輪詢，效率高。同時記錄每個裝置的「同步向量時鐘」（sync cursor），只拉取上次同步後的變更。
</details>

**Q3：Uber 司機位置更新 QPS 高達 60 萬，如何避免 Redis 成為瓶頸？**
<details>
<summary>答案</summary>
幾個層次的優化：1) Redis Cluster 分片：按 Geohash 前綴分片，每個節點只處理特定地理區域的數據；2) 批次寫入：App Server 收集 100ms 內的位置更新，批次寫入 Redis（一次 pipeline）而非每條單獨寫入；3) 區分冷熱：靜止的車輛（速度 = 0）停止更新，只有移動的車輛才持續上報；4) 客戶端採樣：如果位置變化 < 10 公尺，不上報（車等紅燈時不必每 4 秒上報）。實際上真正需要即時位置的司機約佔 5%（行駛中），有效降低 QPS。
</details>
