# 第五十八章：冪等性設計（Idempotency）

## 前言：為什麼重複請求是個大問題？

```
場景：用戶按下「立即付款」按鈕
      → 網路超時，用戶不確定有沒有成功
      → 再按一次
      → 系統扣款兩次！
```

這不是假想的邊界情況。網路抖動、客戶端重試、消息佇列 at-least-once 語義……  
**重複請求在分散式系統中是必然發生的**，不是可能發生的。

**冪等性（Idempotency）**：同一個操作執行一次和執行多次，最終結果相同。

```
冪等的例子：
  GET /orders/123      → 無論查幾次，都只是讀取，不改變狀態 ✅
  DELETE /orders/123   → 第一次刪除成功，之後再刪都回傳 404，狀態一致 ✅
  PUT /user/1 {name:"Alex"} → 無論更新幾次，名字都是 Alex ✅

不冪等的例子：
  POST /orders         → 每次 POST 都建立一筆新訂單 ❌（需要額外處理）
  POST /payment/charge → 每次都扣一次款 ❌（最危險！）
```

## 一、冪等 Token（防重 Token）模式

**最通用的方案**：客戶端在關鍵操作前，先向服務端申請一個唯一的 Token，  
提交時帶著這個 Token，服務端用 Token 確保同一操作只執行一次。

```
流程：
  1. 客戶端 → GET /idempotency-token  → 服務端回傳 token: "uuid-abc-123"
  2. 客戶端 → POST /payment { token: "uuid-abc-123", amount: 1000 }
  3. 服務端檢查 token → 未使用 → 執行扣款 → 標記 token 為已使用
  4. （網路中斷）
  5. 客戶端 → POST /payment { token: "uuid-abc-123", amount: 1000 } （重試）
  6. 服務端檢查 token → 已使用 → 直接返回第一次的結果（不重複執行）
```

### Redis + Lua 的原子實現

```java
@Service
public class IdempotencyService {

    private final StringRedisTemplate redis;

    // Lua 腳本：SET NX + GET 原子操作，避免 check-then-act 競態條件
    private static final String LOCK_SCRIPT = """
        local current = redis.call('GET', KEYS[1])
        if current then
            return current   -- 已存在：返回上次的結果
        end
        redis.call('SET', KEYS[1], ARGV[1], 'EX', ARGV[2], 'NX')
        return nil           -- 不存在：設置成功（nil 代表首次執行）
        """;

    /**
     * 嘗試執行冪等操作
     * @param idempotencyKey  唯一 key（如 "payment:uuid-abc-123"）
     * @param ttlSeconds      token 有效期（如 24 小時）
     * @param action          實際業務邏輯
     * @return 本次或上次的執行結果
     */
    public <T> T executeOnce(String idempotencyKey, int ttlSeconds, Supplier<T> action) {
        String lockKey = "idempotency:" + idempotencyKey;
        String placeholder = "PROCESSING";

        // Step 1：嘗試搶占（NX = 只在 key 不存在時設置）
        Boolean acquired = redis.opsForValue()
            .setIfAbsent(lockKey, placeholder, ttlSeconds, TimeUnit.SECONDS);

        if (Boolean.FALSE.equals(acquired)) {
            // 已被搶占：等待並返回上次結果
            String cached = redis.opsForValue().get(lockKey);
            if (cached != null && !cached.equals(placeholder)) {
                return deserialize(cached); // 返回上次的執行結果
            }
            throw new DuplicateRequestException("請求正在處理中，請稍後重試");
        }

        // Step 2：首次執行業務邏輯
        try {
            T result = action.get();
            // Step 3：儲存結果（覆蓋 placeholder）
            redis.opsForValue().set(lockKey, serialize(result), ttlSeconds, TimeUnit.SECONDS);
            return result;
        } catch (Exception e) {
            // 執行失敗：刪除 key，允許重試
            redis.delete(lockKey);
            throw e;
        }
    }
}

// Controller 使用
@PostMapping("/payment")
public PaymentResult pay(
        @RequestHeader("Idempotency-Key") String idempotencyKey,  // 客戶端自帶 UUID
        @RequestBody PaymentRequest request) {

    return idempotencyService.executeOnce(
        "payment:" + idempotencyKey,
        86400,  // 24 小時有效
        () -> paymentService.charge(request)  // 實際扣款邏輯
    );
}
```

## 二、資料庫唯一約束防重

**最簡單可靠的方案**：讓資料庫的唯一索引替你做去重。

```sql
-- 訂單表加唯一索引（客戶端生成的請求 ID）
ALTER TABLE orders
    ADD COLUMN request_id VARCHAR(64) NOT NULL COMMENT '客戶端請求ID，防重',
    ADD UNIQUE INDEX uk_request_id (request_id);
```

```java
@Service
public class OrderService {

    public Order createOrder(CreateOrderDTO dto) {
        Order order = Order.builder()
            .requestId(dto.getRequestId())  // 客戶端傳來的 UUID
            .customerId(dto.getCustomerId())
            .amount(dto.getAmount())
            .build();

        try {
            return orderRepository.save(order);
        } catch (DataIntegrityViolationException e) {
            // 唯一索引衝突 → 重複請求
            // 返回已存在的訂單，而不是報錯
            return orderRepository.findByRequestId(dto.getRequestId())
                .orElseThrow(() -> new RuntimeException("Unexpected state"));
        }
    }
}
```

**優點**：簡單、強一致性（資料庫保證）  
**缺點**：依賴資料庫，高並發時衝突會產生異常（需要 try-catch 處理）

## 三、狀態機防重 <!-- 💡 進階 -->

對於有狀態流轉的業務（訂單、支付），用**狀態機**本身來保證冪等性：

```java
@Service
public class PaymentService {

    @Transactional
    public void confirmPayment(String orderId) {
        Order order = orderRepository.findById(orderId)
            .orElseThrow(() -> new OrderNotFoundException(orderId));

        // 關鍵：用 CAS 式更新（只有 PENDING 狀態才能轉成 PAID）
        int updated = orderRepository.updateStatusIfPending(orderId, OrderStatus.PAID);

        if (updated == 0) {
            // 更新失敗：訂單已不是 PENDING（可能已經 PAID 或 CANCELLED）
            OrderStatus currentStatus = order.getStatus();
            if (currentStatus == OrderStatus.PAID) {
                // 已經付款成功，屬於重複請求，直接返回成功（冪等）
                return;
            }
            throw new InvalidOrderStatusException(
                "訂單 " + orderId + " 狀態為 " + currentStatus + "，無法確認付款"
            );
        }

        // 觸發付款成功的後續流程
        eventPublisher.publish(new PaymentConfirmedEvent(orderId));
    }
}

// Repository
@Modifying
@Query("UPDATE Order o SET o.status = :newStatus " +
       "WHERE o.id = :orderId AND o.status = 'PENDING'")
int updateStatusIfPending(@Param("orderId") String orderId,
                          @Param("newStatus") OrderStatus newStatus);
```

```
狀態機冪等保證：
  PENDING → PAID    (第一次確認付款，成功)
  PAID    → PAID    (重複確認，直接返回，不報錯)
  PENDING → CANCELLED → PAID  (試圖對已取消訂單付款，報業務異常)
```

## 四、Kafka 精確一次語義（Exactly-Once）<!-- 💡 進階 -->

Kafka 消費者默認是 at-least-once（至少一次），重啟或 Rebalance 後可能重複消費消息。

### 方法一：消費者自己去重

```java
@KafkaListener(topics = "payment-events")
public void handlePaymentEvent(PaymentEvent event) {
    String messageId = event.getMessageId();  // 消息自帶唯一 ID

    // 用 Redis 記錄已處理的消息（24 小時過期）
    Boolean isNew = redis.opsForValue()
        .setIfAbsent("processed:" + messageId, "1", 24, TimeUnit.HOURS);

    if (Boolean.FALSE.equals(isNew)) {
        log.info("重複消息，跳過: {}", messageId);
        return;  // 冪等：直接跳過
    }

    // 執行業務邏輯
    processPayment(event);
}
```

### 方法二：Kafka Transactions（Producer 端精確一次）

```java
@Bean
public KafkaTemplate<String, Object> kafkaTemplate(ProducerFactory<String, Object> pf) {
    // 需要 enable.idempotence=true + transactional.id 設定
    return new KafkaTemplate<>(pf);
}

@Transactional("kafkaTransactionManager")
public void publishOrderEvent(Order order) {
    // 事務性發送：要麼成功，要麼回滾，不會重複
    kafkaTemplate.send("order-events", order.getId(), OrderCreatedEvent.from(order));
}
```

## 五、API 冪等性設計標準 <!-- 💡 進階 -->

### HTTP 方法的天然冪等性

```
GET    → 天然冪等（只讀）
PUT    → 天然冪等（覆蓋整個資源）
DELETE → 天然冪等（刪除後再刪回 404，狀態一致）
POST   → 天生不冪等（每次都可能建立新資源，需要額外設計）
PATCH  → 看具體語義（SET name="Alex" 冪等，但 increment count 不冪等）
```

### Idempotency-Key Header 標準

Stripe、Adyen 等支付公司採用的行業標準：

```
POST /charges
Idempotency-Key: 4d1a4c2b-3e5f-4a6b-8c7d-9e0f1a2b3c4d  ← 客戶端生成的 UUID v4

Response：
HTTP/1.1 200 OK
Idempotency-Key: 4d1a4c2b-3e5f-4a6b-8c7d-9e0f1a2b3c4d  ← 回傳相同的 key
```

重複攜帶相同 `Idempotency-Key` 的請求：
- 服務端返回第一次的結果（不重複執行）
- HTTP 狀態碼與第一次相同（成功還是失敗都回傳同樣的結果）

### 完整的 Spring Boot 冪等性攔截器

```java
@Component
public class IdempotencyFilter extends OncePerRequestFilter {

    private static final String IDEMPOTENCY_KEY_HEADER = "Idempotency-Key";
    private final IdempotencyService idempotencyService;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
                                    HttpServletResponse response,
                                    FilterChain chain) throws ServletException, IOException {

        // 只對 POST/PATCH 需要冪等保護
        String method = request.getMethod();
        if (!method.equals("POST") && !method.equals("PATCH")) {
            chain.doFilter(request, response);
            return;
        }

        String idempotencyKey = request.getHeader(IDEMPOTENCY_KEY_HEADER);
        if (idempotencyKey == null || idempotencyKey.isBlank()) {
            chain.doFilter(request, response);  // 無 key 則不做冪等處理
            return;
        }

        // 檢查快取
        Optional<CachedResponse> cached = idempotencyService.getCached(idempotencyKey);
        if (cached.isPresent()) {
            // 重複請求：直接回傳快取的 response
            CachedResponse cr = cached.get();
            response.setStatus(cr.getStatus());
            response.setContentType(cr.getContentType());
            response.getWriter().write(cr.getBody());
            return;
        }

        // 首次請求：執行並快取結果
        CachingResponseWrapper responseWrapper = new CachingResponseWrapper(response);
        chain.doFilter(request, responseWrapper);
        idempotencyService.cacheResponse(idempotencyKey, responseWrapper.getCachedResponse());
    }
}
```

## 六、各方案對比與選型

| 方案 | 適用場景 | 一致性 | 複雜度 |
|------|---------|--------|--------|
| **DB 唯一約束** | 資料寫入防重（訂單、用戶注冊）| 強一致 | 低 |
| **Redis SETNX** | 通用防重、跨服務冪等 | 最終一致（Redis 可能宕機）| 中 |
| **狀態機 CAS** | 有狀態流轉（付款、訂單狀態）| 強一致 | 中 |
| **Kafka 消費去重** | 消息消費防重 | 最終一致 | 低 |
| **Idempotency-Key Filter** | 完整 API 層防重 | 依賴儲存層 | 高 |

**選型建議：**
```
1. 資料寫入 → DB 唯一索引（最簡單最可靠）
2. 支付/核心業務 → DB 唯一索引 + 狀態機 CAS（雙重保障）
3. 對外 API → Idempotency-Key Header + Redis 快取
4. Kafka 消費 → Redis 消息 ID 去重
```

## 本章練習題

**Q1：Redis SETNX 做冪等時，如果 Redis 宕機會怎樣？**
<details>
<summary>答案</summary>
Redis 宕機後：（1）SETNX 失敗，原本已處理的 key 消失，下次重試時 SETNX 會成功，導致業務被重複執行。解決方案：對於絕對不能重複的操作（如支付扣款），MUST 同時使用 DB 唯一約束或狀態機 CAS 作為最後一道防線。Redis 是性能優化層，DB 才是正確性保障層。Redis 宕機最多讓防重「性能下降」，不能讓它成為「正確性的單點故障」。
</details>

**Q2：為什麼說 HTTP PUT 天然冪等，但 PATCH 不一定？**
<details>
<summary>答案</summary>
PUT 是「替換整個資源」：PUT /users/1 {name:"Alex", age:30} → 無論發幾次，用戶資料都是 {name:"Alex", age:30}，結果完全相同，所以冪等。PATCH 是「部分更新」：(a) PATCH /users/1 {name:"Alex"} 是「把名字設為 Alex」→ 冪等。(b) PATCH /orders/1 {action:"increment_quantity"} 是「把數量加一」→ 不冪等！每次執行結果都不同。所以 PATCH 的冪等性取決於 patch 的語義是「設置值」還是「增量操作」。
</details>

**Q3：訂單服務用 DB 唯一索引防重，同一個 requestId 第一次請求成功建立了訂單，第二次請求捕獲到 DataIntegrityViolationException 後，應該返回什麼 HTTP 狀態碼？**
<details>
<summary>答案</summary>
應該返回 200 OK（或 201 Created），並在 body 中返回第一次建立的訂單資料。這是冪等設計的核心：重複請求不是「錯誤」，而是「之前請求的重放」。如果返回 409 Conflict，客戶端可能誤以為請求失敗而做出錯誤的業務決策。正確流程：catch DataIntegrityViolationException → 用 requestId 查出已存在的訂單 → 返回該訂單 + 200 OK。業界標準（如 Stripe）也是這樣設計的。
</details>
