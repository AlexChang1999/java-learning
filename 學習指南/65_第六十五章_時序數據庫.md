# 第六十五章：時序數據庫（InfluxDB / TimescaleDB）

## 前言：時序數據的特殊性

```
傳統資料庫處理時序數據的問題：

場景：每台伺服器每秒上報 CPU、記憶體、網路 4 個指標
     100 台伺服器 × 4 指標 × 86400 秒/天 = 3,456 萬筆/天

MySQL 的問題：
  ❌ 大量 INSERT 使 B+Tree 索引不斷分裂，寫入性能快速下降
  ❌ 沒有內建的時間降採樣（downsampling）功能
  ❌ 沒有自動過期機制（要自己寫定時 DELETE）
  ❌ 「最近 1 小時的平均 CPU，按 5 分鐘聚合」這類查詢要寫複雜 SQL

時序數據庫解決的問題：
  ✅ 高吞吐寫入（10 萬~ 100 萬點/秒）
  ✅ 時間範圍查詢極快（資料按時間排序存儲）
  ✅ 內建降採樣：自動將細粒度數據壓縮成粗粒度（秒→分→時→天）
  ✅ TTL（資料保留策略）：超過 N 天自動刪除
  ✅ 專為監控/IoT/金融 Tick 數據設計的查詢語言
```

## 一、InfluxDB：專門的時序數據庫

### 核心概念

```
Measurement（測量）= 類似 SQL 的 Table
  └── Tag（標籤）= 有索引的維度，低基數字串（如 host="server01", region="taipei"）
  └── Field（欄位）= 實際數值，無索引（如 cpu=0.72, memory=1024）
  └── Timestamp（時間戳）= 每個數據點的時間（納秒精度）

資料點範例：
  cpu_usage,host=server01,region=taipei value=72.5 1705275000000000000
  ↑measurement  ↑tags                  ↑field       ↑timestamp (Unix ns)
```

### InfluxDB Line Protocol（寫入格式）

```
格式：<measurement>[,<tag_key>=<tag_value>...] <field_key>=<field_value>[,<field_key>=<field_value>...] [unix-nanosecond-timestamp]

範例：
cpu_usage,host=server01,env=prod cpu=72.5,memory=1024 1705275000000000000
network_io,host=server01,interface=eth0 rx_bytes=1024000,tx_bytes=512000
```

### Flux 查詢語言（InfluxDB 2.x）

```python
# 查詢過去 1 小時 server01 的 CPU 使用率，按 5 分鐘聚合
from(bucket: "monitoring")
  |> range(start: -1h)
  |> filter(fn: (r) => r._measurement == "cpu_usage")
  |> filter(fn: (r) => r.host == "server01")
  |> filter(fn: (r) => r._field == "cpu")
  |> aggregateWindow(every: 5m, fn: mean, createEmpty: false)
  |> yield(name: "mean_cpu")

# 查詢 CPU 超過 90% 的異常時間點
from(bucket: "monitoring")
  |> range(start: -24h)
  |> filter(fn: (r) => r._measurement == "cpu_usage" and r._field == "cpu")
  |> filter(fn: (r) => r._value > 90.0)

# 降採樣任務（Task）：每小時把秒級數據聚合成分鐘級並保存到另一個 bucket
option task = {name: "downsample_cpu", every: 1h}

data = from(bucket: "raw_metrics")
  |> range(start: -task.every)
  |> filter(fn: (r) => r._measurement == "cpu_usage")

data
  |> aggregateWindow(every: 1m, fn: mean)
  |> to(bucket: "downsampled_metrics")
```

### Spring Boot 整合 InfluxDB

```xml
<dependency>
    <groupId>com.influxdb</groupId>
    <artifactId>influxdb-client-java</artifactId>
    <version>7.0.0</version>
</dependency>
```

```java
@Configuration
public class InfluxDBConfig {

    @Value("${influxdb.url}")
    private String url;
    @Value("${influxdb.token}")
    private String token;
    @Value("${influxdb.org}")
    private String org;

    @Bean
    public InfluxDBClient influxDBClient() {
        return InfluxDBClientFactory.create(url, token.toCharArray(), org, "monitoring");
    }
}

@Service
public class MetricsCollector {

    private final InfluxDBClient client;

    // 寫入單個數據點
    public void recordCpuUsage(String host, double cpuPercent) {
        Point point = Point.measurement("cpu_usage")
            .addTag("host", host)               // Tag（有索引，低基數）
            .addTag("env", "production")
            .addField("cpu", cpuPercent)        // Field（實際值）
            .addField("timestamp_ms", System.currentTimeMillis())
            .time(Instant.now(), WritePrecision.NS);

        client.getWriteApiBlocking().writePoint(point);
    }

    // 批次寫入（生產環境推薦）
    public void recordBatch(List<ServerMetrics> metrics) {
        try (WriteApi writeApi = client.makeWriteApi()) {
            metrics.forEach(m -> {
                Point point = Point.measurement("server_metrics")
                    .addTag("host", m.getHost())
                    .addField("cpu", m.getCpu())
                    .addField("memory_mb", m.getMemoryMb())
                    .addField("disk_read_kb", m.getDiskReadKb())
                    .time(m.getTimestamp(), WritePrecision.MS);
                writeApi.writePoint(point);
            });
        }
    }

    // 查詢：最近 10 分鐘各主機平均 CPU
    public Map<String, Double> getAvgCpuByHost() {
        String flux = """
            from(bucket: "monitoring")
              |> range(start: -10m)
              |> filter(fn: (r) => r._measurement == "cpu_usage" and r._field == "cpu")
              |> group(columns: ["host"])
              |> mean()
            """;

        List<FluxTable> tables = client.getQueryApi().query(flux);
        return tables.stream()
            .flatMap(t -> t.getRecords().stream())
            .collect(Collectors.toMap(
                r -> r.getValueByKey("host").toString(),
                r -> (Double) r.getValue()
            ));
    }
}
```

## 二、TimescaleDB：PostgreSQL 的時序擴展

TimescaleDB 的優勢：**繼承 PostgreSQL 的全部能力**（JOIN、複雜 SQL、事務）+ 時序特性。

```sql
-- 啟用 TimescaleDB 擴展
CREATE EXTENSION IF NOT EXISTS timescaledb;

-- 建立普通 PostgreSQL 表
CREATE TABLE sensor_data (
    time        TIMESTAMPTZ  NOT NULL,
    device_id   TEXT         NOT NULL,
    temperature DOUBLE PRECISION,
    humidity    INT,
    location    TEXT
);

-- 轉換為 Hypertable（時序表）
-- TimescaleDB 自動按時間分割資料（chunk = 7天一塊）
SELECT create_hypertable('sensor_data', 'time');

-- 普通 SQL 查詢（自動利用時間分割加速）
SELECT
    time_bucket('5 minutes', time) AS period,
    device_id,
    AVG(temperature) AS avg_temp,
    MAX(temperature) AS max_temp
FROM sensor_data
WHERE time >= NOW() - INTERVAL '1 hour'
  AND device_id = 'sensor_001'
GROUP BY period, device_id
ORDER BY period DESC;

-- 設定資料保留策略（超過 30 天自動刪除）
SELECT add_retention_policy('sensor_data', INTERVAL '30 days');

-- 設定自動降採樣（連續聚合視圖）
CREATE MATERIALIZED VIEW sensor_hourly
WITH (timescaledb.continuous) AS
SELECT
    time_bucket('1 hour', time) AS hour,
    device_id,
    AVG(temperature) AS avg_temp,
    MIN(temperature) AS min_temp,
    MAX(temperature) AS max_temp
FROM sensor_data
GROUP BY hour, device_id;

-- 設定自動刷新（每 1 小時更新一次）
SELECT add_continuous_aggregate_policy('sensor_hourly',
    start_offset => INTERVAL '3 hours',
    end_offset   => INTERVAL '1 hour',
    schedule_interval => INTERVAL '1 hour'
);
```

## 三、InfluxDB vs TimescaleDB 選型

| 維度 | InfluxDB | TimescaleDB |
|------|---------|-------------|
| **SQL 支援** | Flux 語言（需要學習）| 完整 PostgreSQL SQL |
| **複雜查詢** | 時序查詢強，JOIN 弱 | 完整 SQL（JOIN、子查詢）|
| **上手難度** | 中（Flux 語言）| 低（已懂 PostgreSQL）|
| **寫入性能** | 極高（專用引擎）| 高（比 InfluxDB 略低）|
| **儲存效率** | 極高（時序壓縮）| 高（繼承 PG 壓縮）|
| **生態整合** | Grafana 原生支援 | Grafana 支援 |
| **適合場景** | 純監控/IoT（不需要 JOIN）| 需要與業務數據 JOIN 分析 |

**選型建議：**
- 只做伺服器監控、IoT 數據收集 → **InfluxDB**（專用最好）
- 時序數據需要和業務表 JOIN 分析 → **TimescaleDB**（SQL 生態）
- 已有 PostgreSQL 基礎設施 → **TimescaleDB**（減少技術棧複雜度）

## 本章練習題

**Q1：時序數據庫為什麼寫入速度比 MySQL 快？**
<details>
<summary>答案</summary>
時序數據庫的寫入都是「追加」模式（Append-Only），新數據永遠寫在最後，不需要更新或隨機插入現有資料。資料按時間順序排列，不需要維護 B+Tree 索引（B+Tree 在隨機插入時需要分裂頁，產生大量隨機 IO）。InfluxDB 和 TimescaleDB 都使用類似 LSM-Tree 的結構，把寫入先緩衝在記憶體，再批次刷到磁碟（順序 IO）。此外，時序數據壓縮率極高（同一個指標的相鄰值變化小，差值壓縮效果好），進一步減少磁碟 IO。
</details>

**Q2：「降採樣（Downsampling）」在時序數據庫中解決什麼問題？**
<details>
<summary>答案</summary>
監控系統通常按秒收集數據，但一年的秒級數據量是 3150 萬筆/機器。三年後磁碟爆炸，且查「去年全年趨勢」時沒有人需要秒級精度，小時級已經夠用。降採樣的做法：原始秒級數據保留 7 天 → 自動聚合成分鐘級，保留 30 天 → 自動聚合成小時級，保留 1 年 → 自動聚合成天級，長期保存。這樣既保留了近期的高精度數據（故障排查需要），又控制了長期存儲成本。
</details>
