# 第五章：數字系統與浮點數陷阱
> 理解 Brad01/03/06/19 背後的硬體原理

---

## 一、電腦只懂 0 和 1

電腦的硬體（電晶體）只有兩種狀態：通電（1）和不通電（0）。所有的數字、文字、圖片，在硬體層面都是一串 0 和 1 的組合，稱為**二進位（Binary）**。

---

## 二、整數在記憶體中的表示

### 無號整數（只有正數）

8 個 bit = 1 個 byte，可以表示 256 種狀態（0 ~ 255）：

```
十進位  二進位
  0   → 0000 0000
  1   → 0000 0001
  2   → 0000 0010
  10  → 0000 1010
 127  → 0111 1111
 255  → 1111 1111
```

**換算方式（二進位 → 十進位）：**
```
0000 1010
│││└─── 0 × 2⁰ =  0
││└──── 1 × 2¹ =  2
│└───── 0 × 2² =  0
└────── 1 × 2³ =  8
                 ───
                 10  ✓
```

### 有號整數（有正有負）— 補數表示法

Java 的 `int` 是 32 位元，最高位是**符號位**（0=正，1=負）：

```
byte（8位元）的範圍：-128 ~ 127

 0111 1111 = +127（最大正數）
 0000 0001 = +1
 0000 0000 =  0
 1111 1111 = -1
 1000 0001 = -127
 1000 0000 = -128（最小負數）
```

**這就是為什麼 byte 的範圍是 -128 到 127，而不是 -127 到 127。**

### 整數溢位（Overflow）

```java
int max = Integer.MAX_VALUE;  // 2147483647（約 21 億）
System.out.println(max);      // 2147483647
System.out.println(max + 1);  // -2147483648（變成最小值！）
```

```
0111 1111 1111 1111 1111 1111 1111 1111  (+2147483647)
+                                      1
= 1000 0000 0000 0000 0000 0000 0000 0000  (-2147483648)

最高位從 0 變成 1，正數變成負數！
```

**實際案例：** 2038年問題——很多舊系統用 32 位元 int 存 Unix 時間戳（從 1970/1/1 算起的秒數），2038年1月19日會溢位，這是真實存在的工程問題。

## 三、十六進位（Brad06 的 0x10）

十六進位用 0~9 和 A~F 共 16 個符號，每個 16 進位數字對應剛好 4 個 bit：

```
十六進位  →  二進位  →  十進位
   0      →  0000   →  0
   9      →  1001   →  9
   A      →  1010   →  10
   F      →  1111   →  15
  10      →  0001 0000  →  16
  FF      →  1111 1111  →  255
```

**為什麼工程師愛用十六進位？**  
因為 2 個十六進位數字剛好代表 1 個 byte，看記憶體位址時很方便：

```java
int c3 = 0x10;  // = 16（十進位）
// 顏色代碼也是十六進位：
// #FF0000 = 紅色（R=255, G=0, B=0）
// #1A1A1A = 深灰色
```

## 四、為什麼 `0.1 + 0.2 != 0.3`（IEEE 754 浮點數）

這是每個工程師都必須知道的「地雷」：

```java
System.out.println(0.1 + 0.2);         // 0.30000000000000004
System.out.println(0.1 + 0.2 == 0.3);  // false !!!
```

### 根本原因

十進位的小數在二進位中**不一定能精確表示**，就像 1/3 在十進位中是 0.333... 無限循環一樣。

```
十進位 0.1 轉成二進位：

0.1 × 2 = 0.2  → 整數位 0
0.2 × 2 = 0.4  → 整數位 0
0.4 × 2 = 0.8  → 整數位 0
0.8 × 2 = 1.6  → 整數位 1
0.6 × 2 = 1.2  → 整數位 1
0.2 × 2 = 0.4  → 整數位 0  ← 開始循環了！
...

0.1 = 0.0001100110011001100... (無限循環)
```

電腦只能存有限位數，所以會截斷，造成微小誤差，兩個誤差加在一起就不等於 0.3。

### Java 的 `double` 格式（IEEE 754 雙精度）

```
64 位元 = 1 bit 符號 + 11 bit 指數 + 52 bit 尾數

符號 │ 指數（11位）│ 尾數（52位）
  0  │ 01111111011 │ 1001100110011001...
```

能精確表示的數字大約有 15~16 位有效數字。

### 正確的解法

```java
// 方法一：用 Math.round 或允許微小誤差
double a = 0.1 + 0.2;
System.out.println(Math.abs(a - 0.3) < 1e-10);  // true

// 方法二：用 BigDecimal（精確，但慢）
import java.math.BigDecimal;
BigDecimal b1 = new BigDecimal("0.1");  // 注意：必須傳字串！
BigDecimal b2 = new BigDecimal("0.2");
System.out.println(b1.add(b2));          // 0.3（精確）

// ❌ 錯誤用法（double 轉 BigDecimal，誤差已存在）
new BigDecimal(0.1)  // 不是 0.1，而是那個有誤差的二進位值
```

**金融系統一定要用 `BigDecimal`，絕對不能用 `double` 做加減乘除！**

## 五、char 與 Unicode（Brad03 的中文字）

```java
char c3 = '資';  // 中文字也能存在 char 裡
```

**為什麼 Java 的 char 能存中文？**

Java 使用 **Unicode** 編碼，每個字元對應一個唯一的數字（Code Point）：

```
'A'  →  65（= 0x0041）
'a'  →  97（= 0x0061）
'資' →  36164（= 0x8D44）
'😀' →  128512（= 0x1F600）
```

`char` 是 16 位元（2 bytes），可以表示 0 ~ 65535，涵蓋大多數常用字元。

```java
char c = 'A';
System.out.println((int)c);      // 65（Unicode 數值）
System.out.println((char)65);    // A（數值轉字元）
System.out.println((char)('A' + 1));  // B（字元運算）
```

**ASCII 常用對照：**
```
'0'~'9' → 48~57
'A'~'Z' → 65~90
'a'~'z' → 97~122
```

## 六、XOR 交換為什麼有效（Brad19 底層解析）

```java
a = a ^ b;
b = a ^ b;
a = a ^ b;
```

用實際數字追蹤：設 a=5（`101`），b=3（`011`）

```
步驟一：a = a ^ b
  101 (5)
^ 011 (3)
─────────
  110 (6)   → a 現在是 6

步驟二：b = a ^ b
  110 (6)
^ 011 (3)
─────────
  101 (5)   → b 現在是 5（原來 a 的值）

步驟三：a = a ^ b
  110 (6)
^ 101 (5)
─────────
  011 (3)   → a 現在是 3（原來 b 的值）

交換完成！a=3, b=5
```

**XOR 的數學性質：**
- `x ^ x = 0`（任何數和自己 XOR 等於 0）
- `x ^ 0 = x`（任何數和 0 XOR 等於自己）
- `(a ^ b) ^ b = a`（可以「還原」）

## 本章重點整理

| 概念 | 重點 |
|------|------|
| 二進位 | 電腦只懂 0 和 1，所有資料都是二進位 |
| 整數溢位 | int 超過 2147483647 會繞回負數 |
| 十六進位 | 0x 開頭，每2位 = 1 byte，工程師常用 |
| 浮點數誤差 | 0.1 不能精確表示，金融計算用 BigDecimal |
| char/Unicode | Java 字元是 Unicode，中文字也能存 |
| XOR 交換 | 位元運算的數學技巧，不需要暫存變數 |

## 底層延伸問題

**Q1：下面程式碼的輸出是什麼？**
```java
byte b = 127;
b++;
System.out.println(b);
```
<details>
<summary>答案</summary>
-128。127 是 byte 的最大值（0111 1111），加 1 後溢位變成（1000 0000）= -128。
</details>

**Q2：為什麼不能用 `double` 比較金額是否相等？請說明原因並給出正確做法。**
<details>
<summary>答案</summary>
因為 double 使用 IEEE 754 二進位浮點數，很多十進位小數（如 0.1）無法精確表示，會有微小誤差。正確做法是用 BigDecimal，且要用字串建構子（new BigDecimal("0.1")），或者用 long 以「分」為單位計算（如 100 代表 1.00 元）。
</details>

**Q3：`'A' + 1` 的結果型別是 char 還是 int？**
```java
char c = 'A';
System.out.println(c + 1);        // ?
System.out.println((char)(c + 1)); // ?
```
<details>
<summary>答案</summary>
char + int 的結果是 int。所以 c + 1 印出 66（數字），(char)(c + 1) 才印出 B（字元）。
</details>
